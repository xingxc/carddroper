## Context

0015 and 0015.7 both operated on the assumption that `/auth/verify-email` should invalidate the user's session on success — `user.token_version += 1` + `revoke_all_user_tokens(...)` server-side, cookies cleared in the response (the 0015.7 addition), and a "Please log in to continue." success panel on the frontend forcing re-auth.

This ticket reverses that policy decision. The implementation shift is small; the reasoning is:

- **Industry convention preserves the session on verify.** GitHub, Linear, Notion, Stripe, and Canva all treat the verify click as a capability unlock, not a security event. Our spec was an outlier.
- **The defense-in-depth argument doesn't hold for our threat model.** A pre-verify attacker with the session cookie also has email access (because the cookie was issued to the person using the email). They can verify themselves. Killing the session on verify doesn't raise the attacker's cost — it just raises the legitimate user's cost.
- **Cross-device verify becomes hostile.** User registers on laptop, opens email on phone mail app, clicks verify. Before this ticket: phone has no cookies to clear, but laptop's token_version is now stale, so the laptop session is silently 401'd on next API call. The user comes back to their laptop to find themselves logged out. Confusing and unnecessary.
- **Carddroper's verified/unverified distinction is a capability toggle, not a trust boundary.** Unverified users can reach `/app` (soft cap per auth.md). Verification just unlocks paid actions (future ticket). That's capability scope, not session scope.

**Scope of the policy change: verify-email only.** Other `token_version`-bumping endpoints correctly keep their bumps because they are genuine security events:

- `POST /auth/reset-password` (auth.py:462-463) — bumps tv + revokes refresh tokens. A user reset their password; this invalidates sessions that might be held by whoever made them need to reset. Correct.
- Change-email (lands in ticket 0017) — will bump tv + revoke refresh. Email change is a sensitive security action. Correct.

## Design decisions (pre-committed)

- **Drop `user.token_version += 1` and `await revoke_all_user_tokens(user.id, db)` from `verify_email` newly-verified branch.** Don't replace with anything.
- **Revert 0015.7's `JSONResponse` + `_clear_auth_cookies` wrap.** Newly-verified branch returns a bare dict. Idempotent branch was already a bare dict — no change.
- **Frontend:** `VerifyEmailBody.tsx` success state stops calling `markLoggedOut()`, keeps the `invalidateQueries` call (so `useAuth` picks up the new `verified_at`), and the success panel's CTA becomes "Continue" → `/app` instead of "Log in" → `/login`.
- **Preserve the 0015.7 ticket file** as a historical record — mark `status: superseded-by-0015.8` in frontmatter and prepend a one-paragraph note pointing to 0015.8 with the shift rationale. Don't delete.
- **Update `doc/systems/auth.md` line 48** to reflect the capability-toggle framing. Other bullets about reset-password tv bump and change-email tv bump stay unchanged.
- **Update `doc/issues/0018-chassis-hardening-audit.md`** token-version-bump audit bullet to remove the verify-email mention. Pattern continues to apply to reset-password and change-email.
- **Tests:** remove the two 0015.7 tests (`test_verify_email_newly_verified_clears_cookies`, `test_verify_email_idempotent_does_not_clear_cookies`). Add a positive test `test_verify_email_preserves_session` asserting that after a successful verify, the original register cookies still authenticate `GET /auth/me` and the response carries the updated `verified_at`. Also ensure any existing test that asserted `user.token_version == 1` post-verify is updated to assert `== 0` (no bump).

   > **Token-version-bump-and-cookie-clear pattern.** Endpoints that bump `user.token_version` on an authenticated session must clear the dead auth cookies (via `_clear_auth_cookies`) or re-issue fresh ones (via `_set_auth_cookies` with the new tv). Audit mechanic: `grep -n 'token_version += 1' backend/app/routes/` and classify each hit as "clears", "re-issues", "no session", or "intentionally none" with reasoning.

## Resolution

Closed 2026-04-22. Commit `890f141` landed the policy shift; staging redeploy confirmed the full golden-path walkthrough works.

**Staging verified (walkthrough step 1.6 was the gate, along with regressions):**
- Register fresh account with real inbox → `/verify-email-sent` renders.
- Click verify link from email → success panel shows "Your email is verified. You're all set." + Continue button.

**Regression checks green:**
- 1.7 Logout still works.
- 3.1 Authed user visiting `/login` or `/register` still redirects to `/app`.
- 2.6 Re-clicking a stale verify link still renders the idempotent success panel with the Continue button (routes to `/app` under the live session).

**Chassis knowledge preserved:**
- `auth.md` §Email verification line 48 now reflects the capability-toggle framing.
- `0018` token-version-bump-and-cookie-clear audit pattern remains correct for `reset-password`, `change-password`, and the future `change-email` (0017). Verify-email is explicitly called out as "intentionally none" per this ticket.
- `0015.7` retained with `superseded-by-0015.8` status for historical record.

**Design principle documented:** `verified_at` is a capability toggle (unlocks paid actions post-soft-cap), not a security event. Session-invalidation is reserved for genuine security actions (password reset, email change, admin revoke). This matches industry convention and is what the chassis ships going forward for any new project.
