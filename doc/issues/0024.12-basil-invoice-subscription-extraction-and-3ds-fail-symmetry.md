## Audit (per `doc/operations/audit-template.md`)

### 1. User-visible action

**Bug 1:** any user successfully subscribes (4242 or 3DS-success) with `BILLING_SUBSCRIPTION_GRANTS_TO_LEDGER=true`. They expect their balance to reflect the grant. The grant silently doesn't happen.

**Bug 2:** user submits 3DS card, passes SetupIntent 3DS modal, fails the second 3DS modal (subscription's first-invoice 3DS challenge). Asymmetric with the decline (`4000 0000 0000 0341`) flow which correctly shows "Your card was declined" and a working Subscribe button that resets the form.

### 2. Full request flow trace

**Bug 1 (basil API drift):**

                              upserts row with grant_micros=19990000 (per 0024.7+8 strict flag-gate)

6. customer.subscription.updated webhook fires (status incomplete → active)
7. Result: subscriptions row has status=active, grant_micros=19990000;

**Bug 2 (3DS-fail UX):**

2. Frontend handle3DSChallenge calls stripe.confirmPayment(clientSecret)

### 5. Adversarial scenario

**Bug 1:** Stripe API basil migration moved `invoice.subscription` to a nested location. Our test fixtures use OLD-API shape. Tests pass; production silently fails. Real users lose value. **The general adversarial pattern: any Stripe field migration silently breaks a chassis extractor whose tests use hand-constructed mocks reflecting the OLD shape.** This will recur for any future API migration unless we centralize extractions and structurally test against the new shape.

**Bug 2:** Two structurally similar failure modes (decline at first invoice; 3DS-fail at first invoice) take different code paths to surface to the user. One has a clean recovery channel (0024.9's soft-reset); the other doesn't. **The general adversarial pattern: failure-mode UX is built per-bug, not per-pattern, so each new failure mode needs its own wiring.**

## Context — why this is the second basil-API-drift bug in the arc

This is the same audit-gap pattern as **0024.4** (which fixed period extraction after basil moved `current_period_*` from sub top-level to `items.data[0].current_period_*`). The pattern:

1. Stripe basil migration moved a field
2. Chassis used `getattr(obj, "field", None)` — assumed top-level
3. Test mocks built objects with the field at top-level (matching old shape)
4. Tests passed; production silently failed
5. User-visible symptom (no grant; or NULL period) surfaces during manual smoke
6. Fix: defensive multi-path lookup with the basil canonical path tried first

We fixed the period instance (0024.4) but did not generalize the lesson. This ticket addresses the next instance AND introduces structural prevention.

## Stripe basil paths (confirmed via Stripe docs api/invoices/object)

The subscription ID can appear in **four** locations on a basil Invoice object:

| Path | Type | When present |
|---|---|---|
| `parent.subscription_details.subscription` | expandable string | Primary — invoice generated from a subscription |
| `subscription` (top-level) | string | Legacy / older API versions; absent in pure basil |
| `lines.data[0].parent.subscription_item_details.subscription` | optional string | Per-line-item; subscription this line belongs to |
| `lines.data[0].parent.invoice_item_details.subscription` | optional string | Per-line-item; for invoice items |

The helper's lookup order:
1. `parent.subscription_details.subscription` (basil canonical)
2. `subscription` (top-level legacy)
3. `lines.data[0].parent.subscription_item_details.subscription` (line-item fallback)
4. `lines.data[0].parent.invoice_item_details.subscription` (less common line-item fallback)

If all four miss, log the invoice object's top-level keys + a warning, return None.

## Design decisions (pre-committed)

### Backend (Phase 0a)

1. **Create `app/billing/stripe_extractors.py` module.** Single function this ticket: `extract_invoice_subscription_id(invoice) -> str | None`. The module is the future home for all "extract field X from Stripe object" helpers. Existing inline extractions (period, grant_micros) are NOT migrated in this ticket — defer to a future cleanup ticket. The module starts with one helper but exists as the discipline anchor.

2. **`handle_invoice_paid` calls the new helper.** Replaces the current `getattr(invoice, "subscription", None)` line. The handler MUST NOT retain any inline subscription-id extraction logic — must call only the helper. (Post-dispatch grep verifies.)

3. **The helper's signature is `(invoice) -> str | None` — pure function, no I/O, no logging side effects in the happy path.** Logging on the all-paths-missed case is OK (single warning at the lookup site, not buried in nested helpers).

4. **Helper internals: defensive dual-access on every nested object.** Same pattern as `subscription.py`'s existing extraction code: `getattr(...) or (obj.get(...) if hasattr(obj, "get") else None)` for each nesting layer. Handles both `MagicMock`-style attribute access (in tests using attribute mocks) and dict-style access (real Stripe responses, which are dict-backed StripeObject instances).

5. **Tests must use FIXTURES THAT REFLECT THE BASIL OBJECT TREE EXPLICITLY**, not flat MagicMock attributes. Specifically:

   ```python
   def _basil_invoice_fixture(sub_id="sub_test"):
       """Build an invoice payload mimicking the basil API shape exactly:
       invoice.parent.subscription_details.subscription = sub_id.
       Top-level invoice.subscription is NOT present (this is the post-basil shape).
       """
       parent = MagicMock()
       parent.subscription_details = MagicMock()
       parent.subscription_details.subscription = sub_id
       invoice = MagicMock(spec=["parent", "lines", "billing_reason", "id"])  # spec= prevents attribute auto-vivification
       invoice.parent = parent
       invoice.id = "in_test"
       invoice.billing_reason = "subscription_create"
       # invoice.subscription is intentionally NOT set — that's the bug condition
       return invoice
   ```

   The `spec=` parameter prevents MagicMock from auto-vivifying any attribute access (which would make `invoice.subscription` return a truthy MagicMock and falsely pass the test). This is critical — without `spec=`, the bug would survive into production again.

6. **Backfill SQL.** Two confirmed-affected users (54, 55). Manual SQL after fix lands. Use the actual Stripe event IDs to preserve audit trail and idempotency:

   ```sql
   INSERT INTO balance_ledger (user_id, amount_micros, reason, stripe_event_id)
   VALUES
     (54, 19990000, 'subscription_grant', 'evt_1TRMwQGPSfG3zV2fTVORsvSA'),
     (55, 19990000, 'subscription_grant', 'evt_1TRMyrGPSfG3zV2fSP41j7V8')
   ON CONFLICT (stripe_event_id) DO NOTHING;
   ```

   The `ON CONFLICT DO NOTHING` makes it idempotent (safe to run twice; safe if Stripe somehow re-delivers).

### Frontend (Phase 0b)

7. **`handle3DSChallenge` on confirmPayment error → `onInvoiceDeclined(error.message)`** instead of `onError(error.message)`. The success path of `handle3DSChallenge` (no error) is UNCHANGED.

8. **Add inline code comment explaining the broadened semantic of `onInvoiceDeclined`.** Note that despite the name, it's the "subscribe failed; reset form" channel covering both decline and 3DS-fail. Defer renaming to `onSubscribeFailure` to a future cleanup ticket — semantic-rename is purely cosmetic and decoupled from the bug fix.

9. **No backend cleanup endpoint.** 0024.9's lazy cleanup branch in `/billing/subscribe` already handles incomplete-row cleanup on the next subscribe attempt. After 3DS-fail soft-reset, the user retries → backend sees existing incomplete row → cancels + deletes → creates new sub. No new endpoint needed.

### Out of scope

- **Refactoring existing inline period/metadata extractions into the new module.** Defer to a future cleanup ticket. This ticket creates the module with one function; future tickets migrate other extractions as they're touched.
- **Renaming `onInvoiceDeclined` to `onSubscribeFailure`.** Defer.
- **Past_due renewal recovery.** 0025 / Customer Portal.
- **Dedicated cleanup endpoint for incomplete subs.** Lazy cleanup is sufficient.
- **Frontend test infrastructure.** No Vitest installed; same posture as 0024.9–0024.11. Document the manual coverage instead.
