## Context

Golden-path failure on staging after 0015 Phase 1 + 0015.5 landed. Observed in the manual browser walkthrough:

- **1.6 (blocker):** Login credentials submit → backend returns 200 with `Set-Cookie: access_token=...` → frontend calls `router.push('/app')` → browser navigates to `/app` on the frontend domain → `frontend/proxy.ts` redirects back to `/login`.
- **3.1 (blocker):** Authed user visits `/login` or `/register` → proxy should redirect to `/app` → doesn't.
- **2.7.1 (polish):** Click invalid verify link → invalid panel → click "Request a new email" → `/verify-email-sent` → page shows a Resend button → click → `POST /auth/resend-verification` → 401 (no session).

### Verified root causes (read against code, not speculated)

**1.6 and 3.1 — cookie domain scoping.**

Frontend at `https://staging.carddroper.com`, backend at `https://api.staging.carddroper.com`. Backend sets `access_token` / `refresh_token` via `_set_auth_cookies` in `backend/app/routes/auth.py:85-97`, which already supports a `domain=settings.COOKIE_DOMAIN` kwarg (lines 81 and 95). `COOKIE_DOMAIN` is declared in `backend/app/config.py:31` with default `None`. When unset, `response.set_cookie(..., domain=None)` scopes cookies to the exact request host — `api.staging.carddroper.com`.

The frontend proxy at `frontend/proxy.ts:6` reads `request.cookies.has("access_token")`. Its request arrives at `staging.carddroper.com`; the browser only forwards cookies scoped to that host or a parent domain. Cookies on `api.staging.carddroper.com` are invisible here. Result: proxy always sees "no cookie," both rules fail.

`cloudbuild.yaml` line 66 (post-0015.5) sets several env vars on the backend Cloud Run deploy but **not** `COOKIE_DOMAIN`. Fix is one line in cloudbuild plus a chassis-level invariant so the next project can't make the same omission silently.

**2.7.1 — verify-email-sent Resend not guarded on auth.**

`frontend/app/(auth)/verify-email-sent/page.tsx:56-64` renders the Resend button whenever `resendState === "idle"`. The preceding `user ? ... : ...` conditional only controls the paragraph copy (lines 37-52), not the button. When the user arrives at this page unauth'd (common path: verify-email success runs `markLoggedOut()` → user clicks "Request a new email" from an invalid-link panel → lands here with no session), the button is still visible. Clicking fires `POST /auth/resend-verification` which requires `get_current_user` → 401. Expected UX per the 0015 Phase 1 spec is the unauth fallback ("Check your inbox; if you don't have an account, [Register]") with no Resend button.

### Why local tests missed this

- `scripts/smoke_chassis.sh` runs against `http://localhost:3000` + `http://localhost:8000` — same-host cookie scope, cross-subdomain problem doesn't exist.
- `backend/scripts/smoke_auth.py` uses `httpx` — it sends cookies back along with requests regardless of Domain attribute on the Set-Cookie header, so cookie-domain misconfiguration is invisible to it.
- The 2.7.1 path was not in any automated test; the Phase 1 frontend-builder walkthrough went through the happy path only.

## Design decisions (pre-committed)

- **Use existing `settings.COOKIE_DOMAIN`.** The cookie setter already honors it. No new helper, no new API surface.
- **Treat the cookie-domain setting as a chassis invariant.** Every project with frontend-and-backend-on-different-subdomains needs this, and the cost of getting it wrong is what we just saw: silent proxy failure that looks like a login bug. Add a pydantic validator + `chassis-contract.md` entry per the Option C coupling rule established in 0015.5.
- **Validator semantics:** if `COOKIE_DOMAIN` is set, the host of `FRONTEND_BASE_URL` must either equal `COOKIE_DOMAIN` (with leading dot stripped) or end with `"." + COOKIE_DOMAIN` (stripped). If `COOKIE_DOMAIN` is unset (default), skip the check — correct for local dev on localhost.
- **Do not validate backend URL.** Settings don't store a `BACKEND_BASE_URL` today, and adding one is scope creep. If the backend is deployed at a host outside the cookie domain, the browser will silently reject the Set-Cookie (browser refuses Domain attributes the request host can't claim). Document this caveat in the contract entry.
- **Extend `smoke_auth.py` rather than adding a new smoke.** The natural place to assert "register/login response Set-Cookie carries `Domain=<expected>`" is alongside existing auth smoke assertions. One optional flag, `--expected-cookie-domain`, that when set, asserts the attribute is present.
- **Resend guard is a frontend-only fix.** Not a chassis invariant — just a missing conditional. Small PR, separate phase, same commit cycle for coherent review.
- **Do not touch cookie paths** (`access_token` path=`/`, `refresh_token` path=`/auth`). They are correct. Only the Domain attribute is the issue.

## Acceptance

### Phase 0 — chassis backend + infra (backend-builder)

1. **`backend/app/config.py`** — add `@model_validator(mode="after")` `validate_cookie_domain` (or equivalent name, adjacent to the existing `validate_cors_origins`):

   ```
   if COOKIE_DOMAIN is None → skip (local-dev shape).
   Else:
     host = urlparse(FRONTEND_BASE_URL).hostname  (e.g. "staging.carddroper.com")
     suffix = COOKIE_DOMAIN.lstrip(".")           (e.g. "staging.carddroper.com")
     valid = (host == suffix) or host.endswith("." + suffix)
     if not valid: raise ValueError(...)
   ```

   Error message (copy verbatim so adopters get consistent text):
   ```
   Cookie-domain misconfiguration: FRONTEND_BASE_URL host=<host> is not covered by COOKIE_DOMAIN=<cookie_domain>.
   Browsers will not forward cookies scoped to COOKIE_DOMAIN to the frontend host, so the frontend proxy cannot gate auth routes.
   Either leave COOKIE_DOMAIN unset (single-host deployments) or set it to a parent domain of FRONTEND_BASE_URL (e.g. ".example.com" for https://app.example.com).
   ```

   Imports: add `from urllib.parse import urlparse` if not already present.

2. **`backend/tests/test_settings_validator.py`** — extend (do not replace) the existing file from 0015.5 with cases for the new validator:
   - Happy: `COOKIE_DOMAIN=None` → no-op → constructs.
   - Happy: `COOKIE_DOMAIN=".staging.carddroper.com"`, `FRONTEND_BASE_URL="https://staging.carddroper.com"` → constructs.
   - Happy: `COOKIE_DOMAIN=".carddroper.com"`, `FRONTEND_BASE_URL="https://app.carddroper.com"` → constructs (subdomain under the cookie domain).
   - Happy (no leading dot): `COOKIE_DOMAIN="staging.carddroper.com"`, `FRONTEND_BASE_URL="https://staging.carddroper.com"` → constructs (leading-dot stripping works).
   - Failing: `COOKIE_DOMAIN=".other.com"`, `FRONTEND_BASE_URL="https://staging.carddroper.com"` → raises with the expected message substring.
   - Failing: `COOKIE_DOMAIN=".staging.carddroper.com"`, `FRONTEND_BASE_URL="http://localhost:3000"` → raises.

   The CORS validator cases from 0015.5 stay green — if you need to pass a valid `CORS_ORIGINS` to avoid the CORS validator tripping first, use the `FRONTEND_BASE_URL` as its own origin in each test's kwargs.

3. **`cloudbuild.yaml`** — in the backend `gcloud run deploy carddroper-backend` step's `--set-env-vars` CSV (line 66, post-0015.5), append `,COOKIE_DOMAIN=.staging.carddroper.com`. The leading dot is explicit; modern browsers treat it as equivalent to no-dot but explicit signals intent and matches historical RFC 2109 convention. Repo-root file; "backend only" default is **relaxed for this one file** per CLAUDE.md §Dispatch brief template.

4. **`doc/operations/chassis-contract.md`** — add a second invariant section below the CORS one. Structure mirrors the CORS entry (required / purpose / error message / enforcement location / how-to-satisfy). Specifics:
   - **Invariant:** `FRONTEND_BASE_URL host ⊆ COOKIE_DOMAIN` (when `COOKIE_DOMAIN` is set).
   - **Purpose:** when frontend and backend live on different subdomains, the auth cookies must be scoped to a common parent so both hosts see them.
   - **Error message on violation:** as spec'd above.
   - **Enforcement location:** `backend/app/config.py` — `Settings.validate_cookie_domain` (or final symbol name).
   - **How to satisfy:**
     - Single-host deployment (frontend and backend on the same origin): leave `COOKIE_DOMAIN` unset.
     - Multi-subdomain deployment (e.g. `app.X.com` + `api.X.com`): set `COOKIE_DOMAIN=.X.com`.
     - Cross-domain deployment (e.g. `app.A.com` + `api.B.net`): cookie-based auth across domains is not supported by this chassis; use a same-origin reverse-proxy or a bearer-token strategy.
     - Caveat: the backend's deployed host must itself be under `COOKIE_DOMAIN`. If it isn't, the browser silently rejects the `Set-Cookie`. The chassis does not validate the backend host (no `BACKEND_BASE_URL` setting today); adopters must verify this during deploy setup.

5. **`backend/scripts/smoke_auth.py`** — add an optional `--expected-cookie-domain` CLI arg. When provided, after the register/login calls, assert the response `Set-Cookie` header contains `Domain=<value>` for both `access_token` and `refresh_token`. When not provided, skip the assertion (keeps local dev runs green). Use the existing PASS/FAIL print style.

## Resolution

Closed 2026-04-22. Backend + frontend fixes landed atomically in commit `5e376a4`.

**Staging verified:**
- `smoke_auth.py --expected-cookie-domain=.staging.carddroper.com` green — both `access_token` and `refresh_token` Set-Cookie headers on register/login now carry `Domain=.staging.carddroper.com`.
- Browser walkthrough step 3.1 (authed user visiting `/login` or `/register` redirects to `/app`) works end-to-end.
- Step 2.7.1 (invalid verify → Request new email → `/verify-email-sent` as unauth'd) now shows the fallback copy only, no Resend button.

**Chassis contract grew:**
- Second invariant entry landed: `FRONTEND_BASE_URL host ⊆ COOKIE_DOMAIN` (when `COOKIE_DOMAIN` is set). Follows the Option C coupling rule from 0015.5 — validator + contract entry in the same commit.
- Caveat documented in the contract entry: backend deployed host must also be under `COOKIE_DOMAIN`; not validated by the chassis (no `BACKEND_BASE_URL` setting). Candidate for 0018 hardening audit.

**Follow-up discovered during the retry walkthrough:** step 1.6 showed login not redirecting to `/app` because verify-email was killing sessions with dead-cookie residue. That led to 0015.7 → 0015.8 (policy shift: verify becomes a capability toggle, not a session reset).
