## Context

0019 landed the authentication infrastructure (SPF, DKIM, DMARC, Cloudflare Email Routing for `dmarc@`). All three authentication checks pass on live mail. **But mail still goes to spam** because the SendGrid Dynamic Templates are still in their initial-stub state from 0010 ("real copy lands with the consumer tickets"):

- **Empty Subject lines** — the strongest single spam signal in Gmail.
- **Unprocessed ASM (Advanced Suppression Manager) variables** rendering as literal `<%asm_group_unsubscribe_raw_url%>` in the body — looks like template-leaking, a phishing artifact to mail filters.
- **Minimal body content** — `<p>Full name </p><p>Change URL ...</p>` looks template-leaky.

This ticket polishes the templates so authenticated mail reaches the Primary inbox. **No code changes, no DNS changes** — pure SendGrid Dashboard edits.

### Why transactional templates don't need the unsubscribe block

Per **CAN-SPAM Act**, transactional emails (verification, password reset, security canaries, receipts) are **exempt** from the unsubscribe-link requirement. Marketing emails need it; transactional doesn't. Removing the unsubscribe block from each template both fixes the spam-trigger issue AND is regulatory-correct for chassis-default templates. (If a future ticket adds marketing emails, those would need a separate ASM unsubscribe group.)

## Scope

For each of the 5 Dynamic Templates in SendGrid:

1. **Set the Subject line** on the active version (table below).
2. **Replace the HTML body** with the polished version (each template's section below).
3. **Replace the plain-text body** (each template's section below).
4. **Remove the unsubscribe block** entirely (don't include the `<%asm_group_unsubscribe_raw_url%>` block in the new body).
5. **Save and activate the version.**

Then re-run the verification test from 0019 §Scope item 5 Option B (change-email flow) and confirm Primary inbox placement.

## Design system (used in all 5 templates)

- **Layout:** centered card on neutral background; `<table>`-based for Outlook compatibility.
- **Card width:** 560px max-width.
- **Body bg:** `#f5f5f7`. **Card bg:** `#ffffff`. **Card radius:** 12px.
- **Primary text:** `#1d1d1f`. **Muted text:** `#86868b`. **Accent (button + links):** `#0071e3`.
- **Font stack:** `-apple-system, BlinkMacSystemFont, 'Segoe UI', Helvetica, Arial, sans-serif`.
- **Inline CSS only** (most email clients strip `<style>` blocks).
- **No images** (text-only is fine; avoids broken-image-on-load placement penalty + alt-text requirements).
- **Brand mark** is a simple uppercase wordmark in muted color above each heading (no logo image).

Adopters cloning the chassis substitute "Carddroper" with their own brand via search-replace in each template body.

### Side-finding: SendGrid tracking + Link Branding TLS gap

The polished templates wrap URLs in `<a href="{{var}}">` (versus the original 0010 placeholder stubs which had URLs in plain `<p>` tags). This caused SendGrid's **click tracking** to rewrite all email button URLs into the Link Branding subdomain (e.g., `url2160.carddroper.com/ls/click?upn=...`). Two layers:

1. The Link Branding subdomain CNAMEs were in DNS but the **Link Branding row in SendGrid Dashboard was never Verified**, so SendGrid hadn't provisioned a TLS cert for `url2160.carddroper.com`. Every clickable link returned Chrome's `doesn't support a secure connection with HTTPS` warning. The original placeholder templates had hidden this latent issue because their URLs weren't in `<a href>`s, so click-tracking couldn't rewrite them.

2. Choice between fixing-the-cert (Verify Link Branding) versus disabling-tracking. Picked the latter as the chassis-correct posture for transactional.

**Fix:** disabled all 4 SendGrid account-level tracking types — **Click Tracking, Open Tracking, Subscription Tracking, Google Analytics Tracking** (Settings → Tracking).

Rationale (chassis-default-off for transactional):
- **Click tracking** — privacy concern (SendGrid logs every click); reliability dependency on Link Branding TLS provisioning; UI exposes tracking-subdomain URLs that look spammy. Cleaner: links go directly to the chassis domain.
- **Open tracking** — 1×1 tracking pixel; no business value for one-off triggered emails; increasingly blocked by Apple Mail Privacy Protection + corporate scanners; privacy-icky on the security canary specifically (measuring whether the user noticed an alert).
- **Subscription tracking** — auto-inserts unsubscribe footers; doesn't belong on CAN-SPAM-exempt transactional. Would have undone 0019.1's unsubscribe-block removal.
- **Google Analytics tracking** — UTM tagging for marketing attribution; pure noise on auth/security emails; UTM-tagged URLs also look more spammy.

Marketing emails (if a future ticket adds them) override per-send via `mail.tracking_settings.X.enable=True` — keeps transactional clean by default and lets marketing opt in explicitly.
