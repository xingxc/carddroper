## Context

0016.2 added an auto-redirect `useEffect` in `(app)/layout.tsx` that fires `router.replace("/login")` when `useAuth` resolves to `!isLoading && !isAuthenticated`. Correct design тАФ except it assumed `/login` is reachable from the ghost-state device. It isn't, because:

- The victim device **still has stale cookies** (access_token + refresh_token) after a cross-device reset. The server cannot remote-clear them.
- `proxy.ts:18-25` gates `/login`: if `request.cookies.has("access_token")`, the proxy 307s to `/app`. This rule is correct for its intended case (authed user visits `/login` тЖТ skip the form, go to app) but wrong for the ghost case (stale cookie but session dead).
- Result: the layout effect's redirect is caught by the proxy on the RSC fetch (the user's observed `/login?_rsc=... тЖТ 307`) and bounced back. The effect re-fires, bounces again, loop.

### Empirical verification (from browser DevTools on the victim device)

| Observation | Status | Interpretation |
|---|---|---|
| `GET /auth/me` | 401 | Server rejects stale access_token (tv mismatch with bumped `user.token_version`). |
| `POST /auth/refresh` | absent | `HAS_SESSION_KEY` was already cleared by a previous reload's interceptor run (where refresh did fire and 401'd). Current reload's interceptor skips refresh because `hasSession()` is false. |
| `GET /login?_rsc=<id>` | **307** with `Location: /app` | Proxy treats stale `access_token` cookie as "authed"; redirects away from `/login`. The loop. |

### Why 0016.2 alone can't fix this

The server-side tokens are already revoked (`token_version` bumped and `revoke_all_user_tokens` called during the reset). The **client-side cookies** are the only thing keeping the proxy confused, and cookies are HttpOnly тАФ JavaScript can't clear them. The only way to clear HttpOnly cookies is a server response with a matching `Set-Cookie` clear header, which requires calling a server endpoint.

`POST /auth/logout` already does exactly this (it's unauthenticated, idempotent, and always returns 200 with `_clear_auth_cookies` applied). The fix is to call it from the 401 interceptor in the two branches where we know the session is dead.

## Design decisions (pre-committed)

- **Call `/auth/logout` from the interceptor in two branches, not just one.** Both the refresh-failed branch (`hasSession()` was true, refresh 401'd) AND the no-session branch (`hasSession()` was false from the start тАФ a prior cycle already cleared it but stale cookies linger) need to clear cookies. Empirical verification showed the victim-device reload we're trying to fix is the **no-session branch** (refresh not in logs) тАФ it's the more common path after any prior interceptor run.

- **Restructure the interceptor block** so cookie-cleanup is a shared step after both sub-branches. Current code has the `hasSession()` gate on the outer `if`; we move it inside so the block entry is now "401 on a non-exempt path" and the session signal affects only whether we try refresh first.

- **Awaited, not fire-and-forget.** The throw that follows must happen **after** the Set-Cookie clear headers apply, otherwise React Query's error state fires the `(app)/layout` effect before cookies are cleared тАФ same loop. Sequence: await cookie cleanup тЖТ throw 401 тЖТ React Query error тЖТ layout effect тЖТ `router.replace("/login")` тЖТ proxy no longer sees cookie тЖТ renders `/login`.

- **Dedupe via module-level promise**, same pattern as `getRefreshPromise`. Parallel 401s across multiple queries share one `/auth/logout` call. Idempotent anyway, but cleaner.

- **Direct `fetch()`, not `apiFetch`**, for the cleanup call. Bypasses the interceptor (which would recurse if `/auth/logout` ever returned anything unexpected тАФ can't happen today, but the bypass is defensive). Matches how `attemptRefresh` does it.

- **Best-effort error handling.** If `/auth/logout` itself fails (network blip, rare), we swallow the error and proceed with the 401 throw. Degraded recovery path: cookies stay, user stays in ghost state, next reload retries. Not worse than current.

- **No `markLoggedOut()` call in the no-session branch** тАФ `HAS_SESSION_KEY` is already clear in that branch, so the helper would be a no-op. Keep the helper calls explicit to where they matter.

- **No new chassis-contract entry.** Behavioral rule, not a startup invariant.

- **0016.2 stays in place.** Its effect is still the correct mechanism for detecting and redirecting from the ghost state. 0016.3 just enables the redirect to actually land.

## Acceptance

### Phase 0 тАФ frontend (frontend-builder, one dispatch)

Single file: `frontend/lib/api.ts`.

1. **Add `logoutCleanupPromise` module-level state and `attemptLogoutCleanup()` helper**, mirroring the `getRefreshPromise` pattern:

   ```ts
   let logoutCleanupPromise: Promise<void> | null = null;

   async function attemptLogoutCleanup(): Promise<void> {
     if (!logoutCleanupPromise) {
       logoutCleanupPromise = (async () => {
         try {
           await fetch(`${API_BASE_URL}/auth/logout`, {
             method: "POST",
             credentials: "include",
           });
         } catch {
           // Best-effort тАФ if logout fails, fall through to the 401 throw.
         }
       })().finally(() => {
         logoutCleanupPromise = null;
       });
     }
     return logoutCleanupPromise;
   }
   ```

   Place next to the existing `refreshPromise` / `getRefreshPromise` / `attemptRefresh` block. Same shape.

2. **Restructure the 401 interceptor block**. Current shape (around lines 146-194):

   ```ts
   if (
     response.status === 401 &&
     !isRefreshExempt(path) &&
     hasSession()
   ) {
     const refreshed = await getRefreshPromise();
     if (refreshed) { /* retry block */ }
     markLoggedOut();
     // Fall through to throw the original 401 below.
   }
   ```

   Target shape:

   ```ts
   if (response.status === 401 && !isRefreshExempt(path)) {
     if (hasSession()) {
       const refreshed = await getRefreshPromise();
       if (refreshed) { /* retry block тАФ unchanged */ }
       // Refresh failed тАФ clear the session signal.
       markLoggedOut();
     }
     // Refresh failed OR we had no session signal to refresh against. In
     // both cases server has already rejected these cookies; ask it to
     // clear them client-side so the proxy stops bouncing /login тЖТ /app
     // on ghost state. Awaited so cookies are gone before the throw below,
     // letting (app)/layout's auto-redirect land on /login cleanly.
     await attemptLogoutCleanup();
     // Fall through to throw the original 401 below.
   }
   ```

   The retry block's internals (retry fetch тЖТ parse тЖТ return or throw) stay exactly as-is.

3. **No other changes to `lib/api.ts`**. Don't touch `attemptRefresh`, `getRefreshPromise`, `markLoggedIn`, `markLoggedOut`, the typed wrappers, or `REFRESH_EXEMPT_PREFIXES`.

4. **No other files touched.** `(app)/layout.tsx` from 0016.2 stays as-is тАФ it's the downstream consumer that depends on this fix landing.

## Resolution

Landed on main (2026-04-22, commit `884a060`). 401 interceptor now fires `POST /auth/logout` (via module-level dedup promise `logoutCleanupPromise`) on both the refresh-failed and no-session branches, clearing stale cookies server-side via `Set-Cookie: Max-Age=0`. Proxy-loop on victim-device reload broken тАФ `(app)/layout`'s 0016.2 redirect now lands cleanly on `/login` without being bounced back by the proxy's authed-redirect rule.

**Partially superseded by 0016.8** (commit `a9057ac`). The no-session branch now gates cleanup on backend error code: fires only when `error.code === "INVALID_TOKEN"` (cookies actually present but stale тАФ Case 5 in the 0016.8 matrix). Skips cleanup when code is `AUTHENTICATION_REQUIRED` (no cookies, nothing to clean тАФ anonymous visitors + post-logout reloads). The refresh-failed branch still fires cleanup unconditionally (cookies were definitely present when hasSession was true).
