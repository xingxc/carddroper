## Context

Slowapi `Limiter` is initialized at `backend/app/routes/auth.py:63` with the default helper `get_remote_address`, which returns `request.client.host`. In FastAPI/Starlette, `request.client.host` is the peer address of the TCP/TLS connection terminating at the app process.

On Cloud Run, the connection terminating at the app is not the end-user's — it's Google Frontend (the load-balancing / TLS-terminating layer). Every request appears to come from a small rotating pool of GFE IPs. The real client IP is preserved in the `X-Forwarded-For` header Google sets on the upstream hop, but `get_remote_address` does not read it.

Consequence: every rate limit on staging is effectively **global across all users**, not per-IP. Affected endpoints (every `@limiter.limit(...)` decorator in `backend/app/routes/auth.py`):

- `/auth/register` (3/min)
- `/auth/login` (5/min — per-email lockout is the real defense here; the per-IP limit becomes trivially bypassable/unfair)
- `/auth/refresh` (10/min)
- `/auth/logout` (10/min)
- `/auth/forgot-password` (3/hour) ← user hit this
- `/auth/resend-verification` (3/hour)
- `/auth/verify-email` (10/min)
- `/auth/change-email` (3/hour) ← future 0017
- `/auth/confirm-email-change` (10/min) ← future 0017

One user exhausting a bucket blocks all users. Not catastrophic on low-traffic staging, but unacceptable in production where it becomes a trivially-weaponized DoS.

### Second consequence

`backend/app/routes/auth.py:274` (login endpoint) also calls `get_remote_address(request)` and stores the result as the `ip` column on `login_attempts` rows via `record_attempt` / `record_attempt_isolated`. Every login-attempt record will have a GFE IP instead of the real client IP — useless for forensics, anomaly detection, or any future admin view. Same fix scope.

### Why `extra="forbid"` (0018) doesn't catch this

`extra="forbid"` guards env-var typos. This bug is about the **semantics** of a correctly-named function call. Different class of issue; the audit in 0018 remains relevant for the env-var angle, but doesn't cover this.

## Verified (in-code, not speculated)

- `backend/app/routes/auth.py:26` — `from slowapi.util import get_remote_address`.
- `backend/app/routes/auth.py:63` — `limiter = Limiter(key_func=get_remote_address)`.
- `backend/app/routes/auth.py:274` — `ip = get_remote_address(request)` inside `login`.
- Only two usages of `get_remote_address` in the entire backend. No other call sites to migrate.
- Cross-device empirical test: user's home-WiFi IP + user's cellular-data IP both hit the `/auth/forgot-password` "too many attempts" response within the same hour. Confirms the limiter's bucket-key is NOT either of those IPs.

## Design decisions (pre-committed)

- **Custom helper, not `slowapi.util.get_ipaddr`.** Slowapi has its own XFF-aware helper, but its behavior has drifted across versions. A small custom helper pins the contract (reads `X-Forwarded-For`, takes first entry, falls back to `request.client.host`), is easy to unit-test, and makes the trust assumption explicit in a docstring.
- **Trust `X-Forwarded-For` because Cloud Run ingress is locked.** Google Frontend sets `X-Forwarded-For` on behalf of the client; a malicious client can't spoof it because GFE overwrites any client-supplied value before forwarding. For local development (no proxy), XFF is absent and the helper falls back to `request.client.host`. For tests (FastAPI `TestClient`), XFF is absent too and falls back to `"testclient"` — existing tests keep passing because they stay well under rate limits.
- **Fix both call sites.** Line 63 (limiter key) AND line 274 (login_attempt IP). Different purposes (rate limiting vs. audit logging) but same fix: use the real client IP.
- **Keep rate-limit values unchanged.** `3/hour`, `5/min`, etc., all stay. This ticket only corrects what "per-IP" actually resolves to.
- **No chassis-contract entry.** Behavioral rule, not a startup invariant. The helper itself is chassis (will serve every future project behind a trusted-proxy deployment), but there's no validator to couple to the contract doc.
- **Helper location: inline in `backend/app/routes/auth.py` for now.** If a third caller appears, refactor into `backend/app/utils/` in a separate commit. YAGNI.
- **Out of scope for this ticket:** multi-instance Cloud Run rate-limit weakening (slowapi's in-memory buckets are per-instance; if we ever run `--max-instances>1` under load, a user could get `quota × instances` effective throughput). Known slowapi limitation; Redis-backed storage is the standard fix. Carddroper staging currently runs `--max-instances=3` but typically scales to 1 under low traffic, so the practical impact today is nil. Re-raise if/when prod launches.

## Acceptance

### Phase 0 — backend fix (backend-builder, one dispatch)

1. **Add helper** to `backend/app/routes/auth.py`, placed above the `limiter = Limiter(...)` line so the `key_func` reference is in scope:

   ```python
   def get_client_ip(request: Request) -> str:
       """Return the real client IP.

       On Cloud Run, ``request.client.host`` is Google Frontend's IP, not the
       end-user's — Google sets ``X-Forwarded-For`` with the real client IP and
       we trust it because Cloud Run ingress is locked (a malicious client
       cannot spoof the header because GFE overwrites client-supplied values).

       Locally or in tests, XFF is absent and we fall back to
       ``request.client.host`` (or ``"unknown"`` if even that is missing —
       edge case with non-HTTP ASGI scopes).
       """
       forwarded = request.headers.get("x-forwarded-for")
       if forwarded:
           # First entry is the original client; subsequent entries are proxies.
           return forwarded.split(",")[0].strip()
       return request.client.host if request.client else "unknown"
   ```

2. **Swap both `get_remote_address` call sites** to use `get_client_ip`:
   - `backend/app/routes/auth.py:63` — `Limiter(key_func=get_client_ip)`.
   - `backend/app/routes/auth.py:274` — `ip = get_client_ip(request)`.

3. **Remove the now-unused import** `from slowapi.util import get_remote_address` (line 26). If `get_remote_address` still has other usages you find, keep the import.

4. **Unit tests** in `backend/tests/test_auth_flow.py` (or a new `test_client_ip.py` if that file feels more appropriate — agent's call):
   - `test_get_client_ip_uses_forwarded_for_when_present` — request with `X-Forwarded-For: 1.2.3.4` → `"1.2.3.4"`.
   - `test_get_client_ip_uses_first_entry_when_multiple` — request with `X-Forwarded-For: 1.2.3.4, 10.0.0.1, 10.0.0.2` → `"1.2.3.4"` (strips whitespace).
   - `test_get_client_ip_falls_back_to_client_host` — no XFF header, `request.client.host = "127.0.0.1"` → `"127.0.0.1"`.
   - `test_get_client_ip_returns_unknown_when_no_client` — no XFF and `request.client is None` → `"unknown"`.

   Constructing the `Request` object for these tests: use Starlette's `Request(scope)` directly with a crafted ASGI scope, OR use FastAPI `TestClient` with a tiny throwaway endpoint that calls `get_client_ip(request)` and returns the result. Pick whichever is idiomatic for the existing suite.

5. **Integration sanity check (optional — skip if the in-memory slowapi state makes it fragile):** a test that hits a rate-limited endpoint multiple times with distinct `X-Forwarded-For` headers and confirms the buckets are separate. If slowapi test isolation makes this brittle, drop it — the helper unit tests + manual staging verification cover the contract.

## Resolution

Landed on main (2026-04-22, commit `b12e1d0`). `get_client_ip(request)` helper reads `X-Forwarded-For` first (first entry = original client, subsequent = proxies), falls back to `request.client.host` when XFF is absent (local dev, tests). `Limiter(key_func=get_client_ip)` wired up. Post-fix cross-device test (home WiFi + iPhone cellular) confirmed distinct IP buckets. Cloud Run ingress is locked, so the chassis trusts XFF without header-stripping middleware. No infra-config changes; no new env vars.
