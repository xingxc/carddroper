## Context

Card `4000 0000 0000 0341` is a Stripe test card that:
1. **Passes** SetupIntent (PM gets attached to Customer) ✓
2. **Fails** the first subscription invoice (charge declined) ✗

This produces a Stripe Subscription with `status="incomplete"` AND `latest_invoice.payment_intent.status="requires_payment_method"`. The PI is in a terminal-fail state — there is no recoverable client_secret, no 3DS to authenticate.

But the chassis at `routes/billing.py:436` reads:

```python
requires_action = sub_status == "incomplete"
```

This is too broad. **Both** 3DS-required and decline-failed subscriptions have `sub.status="incomplete"`. The distinguishing field is `latest_invoice.payment_intent.status`:

| Stripe state | sub.status | invoice.PI.status | Meaning |
|---|---|---|---|
| Active | `active` | `succeeded` | Subscription created, first invoice paid |
| 3DS required | `incomplete` | `requires_action` | PM authenticated; first invoice needs 3DS challenge |
| Declined | `incomplete` | `requires_payment_method` | PM authenticated; first invoice charge declined |
| Auto-canceled | `canceled` (after 23h) | n/a | Stripe gave up |

The chassis treats decline as 3DS, returning `requires_action=true` + a client_secret. Frontend calls `stripe.confirmPayment(client_secret)` against a dead PI → confirm errors out → frontend shows generic "processing error". User clicks Subscribe again → frontend re-POSTs with same PM (no SetupIntent regeneration) → same idempotency key (per 0024.6) → Stripe replays the failed result → loop.

### Why this matters beyond the lifecycle test card

Any production user whose first invoice declines (insufficient funds, fraud rules, expired card slipping past SetupIntent — which can happen for many auth-but-not-charge banks) hits this loop. It's not test-card-specific.

### Why 0024.6 makes the loop sticky

0024.6's idempotency-key-includes-PM fix means same-PM retries replay rather than collide with `IdempotencyError`. The replay is intentional for legitimate network-retry double-submits but pathological when the cached response is a failure.

### Architectural decision: two scenarios, one ticket

Per discussion (and the design rationale recorded in chassis-design conversations 2026-04-28), failure recovery splits into two structurally distinct cases:

- **Scenario A — failed attempts** (subscription `incomplete`, never reached `active`): discardable. No continuity to preserve. Cancel + retry from scratch is correct.
- **Scenario B — failed renewals** (active → past_due): NOT discardable. Subscription has billing history, anchor, proration. Update PM + retry-on-existing is correct (Stripe Customer Portal, deferred to ticket 0025).

This ticket handles Scenario A only. Scenario B is 0025's job.

The two scenarios that produce Scenario A in our chassis:
1. **First invoice declined** (lifecycle test 5.1 — what we're seeing now)
2. **3DS canceled / failed** (test 7B variant — user cancels challenge modal)

Both produce the same DB state (`incomplete` row) and warrant the same recovery (cancel + retry). 0024.9 covers both.

### Pre-existing zombie rows

Phase 1 testing left ~1 incomplete row (user 47 currently). Plus any production user who hit the loop on staging/prod after deploy. 0024.9's design must handle "subscribe with pre-existing incomplete row" as a path, not just "create-and-cleanup synchronously".

## Design decisions (pre-committed)

1. **Distinguish substates of `incomplete` by inspecting `latest_invoice.payment_intent.status`.** The current proxy `requires_action = sub.status == "incomplete"` is wrong; replace with explicit branching on PI status.

2. **Three-way branch after `Subscription.create`:**

   ```python
   pi_status = getattr(sub.latest_invoice.payment_intent, "status", None)

   if sub.status == "active":
       # First invoice succeeded; no 3DS needed.
       requires_action = False
       client_secret = None
       # Proceed to upsert.

   elif sub.status == "incomplete" and pi_status == "requires_action":
       # 3DS challenge required; frontend confirms via stripe.confirmPayment.
       requires_action = True
       client_secret = sub.latest_invoice.payment_intent.client_secret
       # Proceed to upsert (subscription is in valid recoverable state).

   else:
       # Terminal failure: declined, canceled, requires_payment_method, etc.
       # Synchronously cancel the failed Stripe subscription and surface 402.
       # Do NOT upsert; the row should not exist for never-active subscriptions.
       cancel_failed_subscription(sub.id)
       raise payment_required(
           error_code="INVOICE_PAYMENT_FAILED",
           message=last_payment_error_message(sub) or "Your card was declined.",
           details={
               "decline_code": last_payment_error_field(sub, "decline_code"),
               "code": last_payment_error_field(sub, "code"),
           },
       )
   ```

3. **Reorder: upsert AFTER the success/3DS-required check, not before.** Currently the chassis upserts the row regardless. The new flow must NOT write a row for terminal-failure subscriptions — they're discarded synchronously.

4. **Pre-existing incomplete row cleanup at the start of the endpoint** — defensive against pre-fix data and any edge cases where sync cleanup was bypassed. Before the existing 409 ALREADY_SUBSCRIBED check:

   ```python
   if existing and existing.status in ("incomplete", "incomplete_expired"):
       # Pre-existing failed attempt; cancel it and clear the row before proceeding.
       cancel_failed_subscription(existing.stripe_subscription_id)
       await session.execute(delete(Subscription).where(Subscription.id == existing.id))
       await session.flush()
       existing = None  # so the 409 check below sees a clean state
   ```

5. **`cancel_failed_subscription()` helper** — wraps `stripe.Subscription.delete()` with idempotency safety:
   - Catches `stripe.error.InvalidRequestError` ("No such subscription" / "subscription_canceled") and treats as success
   - Catches generic `stripe.error.StripeError` (5xx, network), logs warning, does NOT re-raise — cleanup is best-effort, the request continues to surface the original failure
   - Stripe's auto-cancel at 23h is a backup if our cleanup ever fails

6. **`payment_required()` error helper added to `app/errors.py`** — mirrors the existing `conflict` / `validation_error` shape, returning HTTP 402 with structured body. Sets a precedent for any future payment-related structured errors.

7. **Frontend (Phase 0b): on 402 response with `error_code: INVOICE_PAYMENT_FAILED`, reset the form.** Specifically:
   - Show structured error message from response body (decline message inline)
   - Clear `clientSecret` state
   - Re-fetch SetupIntent → re-mount PaymentElement with new client_secret
   - Re-enable Subscribe button
   - User enters new card (fresh PM via fresh SetupIntent → different idempotency key per 0024.6 → fresh Stripe call)

8. **Other 4xx errors keep current handling.** 422 (validation) and 409 (already subscribed) don't reset the form — they show the inline error. Only 402 specifically triggers the reset.

9. **No backfill of historical incomplete rows.** Phase 0a's pre-existing-row cleanup branch handles them lazily on next subscribe. The user can also `DELETE FROM subscriptions WHERE status='incomplete'` manually if desired.

10. **No frontend changes for 3DS path** — `requires_action=true` returns work as before (post-0024.6).

11. **No chassis-contract entry.** Bug fix + UX recovery, not a new startup invariant.

## Out of scope

- **`past_due` recovery** (Scenario B) — different invariants (preserve continuity); 0025/Customer Portal handles it.
- **`unpaid` recovery** — same family as past_due; 0025.
- **Stripe webhook idempotency for the cancel call** — `customer.subscription.deleted` may fire from our cancel; existing handler treats it correctly (status='canceled'). No new wiring needed.
- **Granular error-code mapping** — return whatever Stripe gives us via `last_payment_error.message`. Don't translate decline codes to friendly strings (that's a UX polish ticket).
- **Backend rate-limiting on subscribe retries** — possible future hardening; not this ticket.
- **`Subscription.delete` retry queue** for cleanup failures — best-effort with logging is enough; Stripe auto-cancels at 23h.

**2. Add a private `_cancel_failed_subscription(sub_id)` helper to `routes/billing.py`** (or `app/billing/primitives.py` if that's the conventional location for Stripe-call wrappers):

```python
def _cancel_failed_subscription(sub_id: str) -> None:
    """Idempotent best-effort cancel of an incomplete subscription.

    Catches "already canceled" / "no such subscription" as success.
    Catches generic StripeError as warning (cleanup is best-effort; Stripe
    auto-cancels incomplete subs at 23h as a backup).
    """
    try:
        stripe.Subscription.delete(sub_id)
    except stripe.error.InvalidRequestError as e:
        if any(s in str(e).lower() for s in ("no such subscription", "already canceled")):
            return
        raise
    except stripe.error.StripeError as e:
        logger.warning(
            "subscribe: failed to cancel incomplete subscription during cleanup",
            extra={"sub_id": sub_id, "error": str(e)},
        )
        return
```

**1. `SubscribeForm.tsx` — error handling for 402 with `error_code: INVOICE_PAYMENT_FAILED`:**

Read the file in full first. The component manages a `phase` state machine and an `error` string. On a 402 with `error_code: INVOICE_PAYMENT_FAILED`:

a) Set `error` to the message from the response body (server-provided; do NOT translate decline codes locally).

b) Clear `clientSecret` state.

c) Re-fetch `/billing/setup-intent` to get a new client_secret. Set state to mount a fresh `<PaymentElement>` against the new client_secret.

d) Reset `phase` so the form is interactive again.

The cleanest pattern is a "soft reset": the user sees the inline error, the form re-mounts with a fresh card-input, and they retry with a different card. This contrasts with a "hard reset" (full unmount + remount) which would lose error message visibility.

## Chassis implications

After 0024.9, the subscribe response contract is:

| Response | Meaning | Frontend action |
|---|---|---|
| `200 {status: "active", requires_action: false}` | Subscription created, first invoice paid | Show success |
| `200 {status: "incomplete", requires_action: true, client_secret}` | 3DS challenge needed | Confirm via stripe.js |
| `402 {error_code: "INVOICE_PAYMENT_FAILED", message, details}` | Card declined; subscription cleaned up | Reset form, show error |
| `409 {error_code: "ALREADY_SUBSCRIBED"}` | User has active/trialing/past_due sub | Surface, redirect to manage |
| `422 {...}` | Validation error (missing tier, bad PM, etc.) | Inline form error |

The chassis no longer leaks Stripe-internal "incomplete" state for terminal failures — adopters can trust that `subscriptions` rows exist only for `active`, `trialing`, `past_due`, `canceled`, or `incomplete-pending-3DS` subscriptions. Failed-attempt subs are synchronously discarded.

Scenario B (`past_due` renewal recovery) remains explicitly out of scope — it requires Customer Portal (0025) and has different invariants (preserve subscription continuity).

No new chassis-contract entries (failure-recovery refinement, not new invariant).
