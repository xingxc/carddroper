## Context

The chassis subscribe endpoint (per 0024 design) calls `stripe.Subscription.create(...)` with `idempotency_key=f"subscribe:{user.id}:{body.price_lookup_key}"`. Stripe idempotency keys are scoped to **24 hours** with the rule: same key + identical params → replay; same key + different params → `IdempotencyError`.

The chassis key includes only `user.id` and `price_lookup_key` — NOT `payment_method_id`. So any subscribe retry with a different PM within 24h collides:

- 1st attempt: subscribe with `pm_1`. Stripe stores key+params, returns subscription.
- 2nd attempt: subscribe with `pm_2` (e.g., user retried after 3DS, after decline regenerated SetupIntent, after browser back-button + new form). Same key. Different params (different PM). Stripe rejects with `IdempotencyError`.
- Backend's `stripe.Subscription.create` raises; nothing in the endpoint catches `stripe.error.IdempotencyError`; FastAPI returns 500; frontend shows generic error.

### Why the original 0024 idempotency key was insufficient

The original design intent was: protect against network-retry double-submits (same params, retried due to timeout). For that case, a key based on `(user, tier)` would correctly replay the original response on retry.

But the design overlooked: legitimate user-driven retries with different PMs (decline → new card → retry; 3DS-fail → new SetupIntent → retry; abandon-and-resume mid-flow). For those, the old key correctly identifies the same conceptual "subscribe action" but the PM differs — and Stripe's rule is strict about params matching.

### Why the bug isn't 3DS-specific

3DS just happens to be the most reliable trigger because the 3DS test card forces a multi-step flow that often results in users regenerating SetupIntents. But the bug surfaces on:

- Decline + retry with new card (regenerates PM)
- Network error mid-confirm (frontend may regenerate SetupIntent)
- User abandons mid-flow, returns later within 24h (new SetupIntent on form re-mount)
- User opens subscribe in two browser tabs (different PMs)

Production user impact: any of these retry paths returns 500 + "Something went wrong" with no recovery path within the 24h window.

## Design decisions (pre-committed)

1. **Include `payment_method_id` in the idempotency key.** Each unique `(user, tier, PM)` combination becomes its own idempotency scope:

   ```python
   idempotency_key = f"subscribe:{user.id}:{body.price_lookup_key}:{body.payment_method_id}"
   ```

   Same-PM resubmits (network-retry double-submits) still replay the original response — Stripe's idempotency contract holds. Different-PM retries become fresh Stripe calls — no collision.

2. **Keep the same idempotency-key structure** rather than switching to a time-window pattern (like topup's `f"topup:{user.id}:{amount_micros}:{int(time.time() // 60)}"`). The PM-inclusion approach is more precise — the user never legitimately retries with the SAME PM on a NEW conceptual subscribe action; if PM is the same, it IS the same conceptual action.

3. **No defensive `try/except IdempotencyError`** in the endpoint. With the new key, the error class no longer fires for legitimate retry flows. Wrapping it would just paper over future similar key-design bugs.

4. **No change to the existing `409 ALREADY_SUBSCRIBED` check** at the top of the endpoint. That check (per 0024 design) returns 409 if the user has an `active|trialing|past_due` subscription before the chassis even calls Stripe. The IdempotencyError fires in a different scenario: user has an `incomplete` subscription (or none at all yet) and retries with a different PM. The two checks are orthogonal.

5. **Update tests to assert the new key shape.** Existing subscribe tests that mock `stripe.Subscription.create` should verify the `idempotency_key` kwarg includes the PM ID.

6. **No frontend changes.** Frontend already submits `payment_method_id` in the subscribe request body. The chassis just needs to use it in the key construction.

7. **Document in `payments.md` §Flows item 4.** Brief note on the idempotency key shape so adopters understand the chassis discipline.

8. **No `chassis-contract.md` entry.** Not a startup-time invariant.

## Out of scope

- **Defensive handling of generic `stripe.error.StripeError`** in the endpoint — could improve error-mapping for many cases, but expands the ticket. Future hardening pass.
- **Idempotency for `POST /billing/topup`** — that endpoint already uses a time-window key (`{int(time.time() // 60)}`); no parallel bug.
- **Idempotency for `POST /billing/setup-intent`** — currently uses minute-window per 0024; no parallel bug.
- **Frontend retry UX** (e.g., automatically clearing the form after a 500) — chassis stays out; project layer's call.
- **3DS-specific testing infrastructure** — chassis works correctly; 7B will pass once the idempotency key is fixed.

## Resolution

Resolved 2026-04-29. Subscribe idempotency key now `f"subscribe:{user.id}:{lookup_key}:{payment_method_id}"` (content-based pattern per the new `idempotency-policy.md`). Same-PM retries replay (correct for accidental double-submits); different-PM retries (3DS-cancel-then-retry, decline-then-new-card) become fresh Stripe calls with distinct keys. Verified via staging deploy 2026-04-29 — 3DS retry scenarios work cleanly; no IdempotencyError on legitimate user-driven retries within Stripe's 24h window. This bug's class motivated `idempotency-policy.md`'s three-pattern classification (content-based / per-request / time-window).
