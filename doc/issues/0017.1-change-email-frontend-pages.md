## API contracts (frontend-builder doesn't read backend; spec is here)

Both endpoints already exist; this ticket consumes them as-is.

### `POST /auth/change-email`

- **Body:** `{current_password: string, new_email: string}`.
- **Auth:** required (authed user; verified-status not required by design — lets users with a typo'd email fix it before verifying; documented in 0017 Resolution).
- **Success (200):** `{message: string}` — a generic success message; the backend has triggered a verification email to the new address.
- **Errors:**
  - `401` — wrong current password (`code: "INVALID_CREDENTIALS"` per 0016.8).
  - `409` — new email already registered (`code: "EMAIL_TAKEN"`).
  - `422` — invalid email format or other Pydantic validation (`detail` has the field error).
  - `429` — rate-limited (chassis convention).
- **Side effects (server):** signs a token, sends `SENDGRID_TEMPLATE_CHANGE_EMAIL` to the new address with a verification link `https://<frontend>/confirm-email-change?token=...`. Does NOT yet flip `user.email`.

### `POST /auth/confirm-email-change`

- **Body:** `{token: string}`.
- **Auth:** not required (reached from the verification email link; user may or may not be logged in).
- **Success (200):** `{message: string}` — `user.email` is now flipped; `token_version` is bumped (any active sessions on this account are invalidated).
- **Errors:**
  - `400`/`410` — invalid, expired, or already-used token.
  - `409` — race condition: new email taken by another user between request and confirm.
- **Side effects (server, in order):**
  1. Notification email sent to OLD address (`SENDGRID_TEMPLATE_EMAIL_CHANGED`) — the security canary.
  2. `user.email` updated to new address.
  3. `user.token_version` incremented.
  4. Auth cookies are NOT issued (the user logs in afresh with the new email).

## Approach (Phase 0b — implementation)

Three deliverables. Keep them in 1–3 commits per cohesion (e.g., one for the request page, one for the confirm landing, one for the ProfileMenu link — or bundle if the diffs are small).

### 1. Request page

Render a form that:

- Accepts current password + new email.
- Client-side validates email format (Pydantic-equivalent) and password non-empty.
- On submit: status → `"submitting"`; POSTs `{current_password, new_email}` to `/auth/change-email` via `lib/api`.
- On success: status → `"sent"`; render "Check your inbox at `<new_email>` — we sent a verification link. Click the link within 1 hour to confirm the change."
- On error: stay on the form (don't unmount); render inline error message keyed off `ApiError.code`:
  - `INVALID_CREDENTIALS` → "Current password is incorrect."
  - `EMAIL_TAKEN` → "That email is already registered."
  - `422` → field-level error from `detail`.
  - `429` → "Too many requests, please wait a moment."
  - Other → "Something went wrong, please try again."
- "Cancel" link returns to whatever sensible default (Settings landing, `/app`, or browser back).

### 2. Confirm landing page

Auto-submits on mount:

- Reads `?token=` from URL via `useSearchParams`.
- If no token: render "This link is invalid. Please request a new email change from your account settings."
- If token present: POST `{token}` to `/auth/confirm-email-change` via `lib/api`.
- On success: render "Your email has been updated. Please log in with your new email address." Auto-redirect to `/login` after a brief delay (3–5s) OR provide a "Go to login" button.
- On error:
  - `400`/`410` → "This link has expired or has already been used. Please request a new email change."
  - `409` → "That email is no longer available. Please choose a different email."
  - Other → "Something went wrong. Please try again."

The page works for both logged-in and logged-out users (the backend doesn't require auth on this endpoint). If the user is logged in on the same browser when they confirm, the `token_version` bump invalidates their session on next request — the chassis convention (per `chassis-contract.md` token-version-bump table classification "no session — reached via email link; session self-invalidates on next request via tv mismatch").

### 3. ProfileMenu link addition

In `frontend/components/app-shell/ProfileMenu.tsx`, add a "Change email" link in the Settings section, mirroring the Billing link pattern from 0023:

```tsx
<Link
  href="/app/<path-from-Phase-0a-decision>/change-email"
  role="menuitem"
  onClick={() => setOpen(false)}
  className="block px-4 py-2 text-sm text-gray-700 hover:bg-gray-100"
>
  Change email
</Link>
```

DOM order under Settings should be readable: Email (info) → Billing → Change email → Logout (or whatever existing order). Consult Phase 0a read of ProfileMenu to slot correctly.
