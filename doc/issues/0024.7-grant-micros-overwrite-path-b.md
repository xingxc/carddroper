## Context

0024.2 design: when `BILLING_SUBSCRIPTION_GRANTS_TO_LEDGER=false`, the subscribe endpoint stores `grant_micros=0` in the subscriptions row (grants are disabled; the value is unused). When the flag is true, the subscribe endpoint reads `metadata.grant_micros` from Stripe Price and stores the actual value.

But: the webhook handler `handle_subscription_created` ALWAYS extracts `metadata.grant_micros` from the event payload (it needs the value to know whether to call `grant()`). The handler's flag-gating is on the `grant()` call only, not on the row-field write. So the upsert overwrites the subscribe endpoint's `grant_micros=0` with the metadata value.

User observed:

| Subscription | grant_micros | When |
|---|---|---|
| sub_1TQzAe... (cancelled) | 0 | Earlier subscribe with flag=false; subscribe endpoint correctly stored 0 |
| sub_1TQzaQ... (active, after resubscribe) | 19990000 | Resubscribe with flag=false; subscribe endpoint stored 0; webhook overwrote |

This is the same class of bug as 0024.5 (period fields). 0024.5's Path B fix made `customer.subscription.created/updated` handlers stop being a source of truth for period fields — but only addressed period fields. `grant_micros` has the same architectural problem and was overlooked.

### Why this is a chassis-correctness bug

The subscribe endpoint and webhook handler disagree on what to write for `grant_micros` when flag=false:
- Subscribe endpoint: writes 0 (chassis-spec behavior per 0024.2)
- Webhook handler: writes metadata value (extracts unconditionally; no flag check on the row write)

The state in the database depends on whether the webhook delivered before the next read — race-y, inconsistent. Adopters reading `subscriptions.grant_micros` don't know whether to trust it.

### `grant()` call is correctly flag-gated; only the row-field write is not

To be clear: `balance_ledger` is correctly NOT receiving `subscription_grant` entries when flag=false. The bug is purely in the subscriptions table's `grant_micros` field.

## Design decisions (pre-committed)

Same Path B pattern as 0024.5 (period fields). Single source of truth per field.

1. **`handle_subscription_created` INSERT path** — keep writing `grant_micros` (rare out-of-band creation case; better some value than NULL/error since column is NOT NULL).

2. **`handle_subscription_created` UPDATE path** — DROP `grant_micros` from the `set_=` clause of the upsert. The subscribe endpoint already wrote the row's `grant_micros` at create-time per its flag-gated logic; the webhook should not overwrite.

3. **`handle_subscription_updated`** — DROP `grant_micros` writes entirely. Subscribe endpoint owns the initial set; renewal handler owns updates (per #4); this handler's job is state sync (`status`, `cancel_at_period_end`) only — same shape as the period-fields fix from 0024.5.

4. **`handle_invoice_paid` (subscription_cycle branch)** — write `grant_micros` from `invoice.lines.data[0].price.metadata.grant_micros` ONLY when `BILLING_SUBSCRIPTION_GRANTS_TO_LEDGER=True`. Rationale: when grants are active, this handler is already calling `grant()`/`subscription_reset` for the new period — it's natural for it to also keep the row's `grant_micros` field current (handles tier changes mid-subscription). When grants are disabled, the row's value is irrelevant; don't touch it.

5. **No subscribe endpoint changes.** Existing flag-gated logic (writes 0 when flag=false; metadata value when flag=true) is correct and stays.

6. **No frontend changes.** SubscriptionDisplay reads from API response; chassis backend writes correct values; UI just renders.

7. **No new env vars, no new chassis-contract entries, no migration, no new dependencies.**

8. **No backfill needed.** The 9 existing rows from earlier testing have whatever `grant_micros` value the webhook last wrote. After the fix, those rows are stable (no future overwrites). Adopters can manually correct via SQL if desired; chassis doesn't auto-backfill.

### Why this design over alternatives

**Alternative A** — change subscribe endpoint to always read and store `metadata.grant_micros` regardless of flag. This would make the row always informational/audit (mirror Stripe metadata). Rejected: changes 0024.2's chassis-flag-gate semantics; more invasive; doesn't match "row matches the flag's state" mental model.

**Alternative B** — keep current code, document that `grant_micros` is unreliable when flag=false. Rejected: chassis-discipline says fix the bug, don't paper over with docs.

**Path B (this ticket)** — keep subscribe endpoint as-is; make webhook handlers respect the source-of-truth model. Smallest change; mirrors 0024.5's architectural pattern; consistent semantics.

## Out of scope

- **Backfill of the existing 9 rows** with stale `grant_micros` from earlier testing — adopters can `UPDATE subscriptions SET grant_micros=0 WHERE ...` manually if they want clean state. Chassis won't auto-fix.
- **Subscribe endpoint refactor** to always-read-metadata.
- **Frontend changes.**
- **Schema changes.**
- **Multi-item subscription handling** (single-item v1).

## Chassis implications

`grant_micros` now has the same single-source-of-truth model as period fields:
- **Initial set:** subscribe endpoint (flag-gated: 0 when off, metadata when on)
- **Renewal updates:** `invoice.paid` cycle handler when flag=on (handles mid-subscription tier changes)
- **Other webhook handlers:** preserve the existing value

This eliminates the entire "webhook overwrites endpoint with metadata value regardless of flag" failure class for `grant_micros`, matching the architectural model 0024.5 established for periods.

No new chassis-contract entries (architectural pattern alignment, not new invariant).
