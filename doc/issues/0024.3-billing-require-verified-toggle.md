## Context

The chassis currently gates three billing endpoints behind `Depends(require_verified)`:

- `POST /billing/topup`
- `POST /billing/subscribe`
- `POST /billing/setup-intent`

This was a conservative choice when 0023 + 0024 landed. The intent was to prevent abuse (someone signs up with a fake email, makes purchases, no canonical contact for follow-up). But the trade-off — friction at the moment of payment — runs against the SaaS-industry default and against carddroper's adopter ethos ("don't make it difficult for people to pay").

This ticket adds a chassis flag — `BILLING_REQUIRE_VERIFIED: bool = False` — that gates verified-checking on the three billing endpoints. Default OFF means **anyone authed can pay**; adopters with stricter compliance/regulatory posture flip True to restore the verified-gate.

The flag follows the same shape as `BILLING_SUBSCRIPTION_GRANTS_TO_LEDGER` from 0024.2 (recently landed) — chassis-tunable behavior toggle, default-permissive, no `chassis-contract.md` entry (not a startup invariant).

### What's NOT in scope

- **GET endpoints** (`/billing/balance`, `/billing/subscription`, `/billing/tiers`) are already `get_current_user`-only (not verified-gated). No change.
- **Webhook endpoint** (`POST /billing/webhook`) is signed by Stripe, not user-authed. No change.
- **Other verified-gated endpoints** outside billing (e.g., none currently — change-email is already permissive per 0017's design that lets unverified users fix typos). No change.
- **0020 legal acceptance** — orthogonal concern; ToS gating is a different mechanism (acceptance flag on user, not email verification). Out of scope.

### Why default-OFF (permissive) is the right chassis default

| Argument | Detail |
|---|---|
| **Industry standard** | Stripe, Shopify, most B2C SaaS allow same-session signup-and-purchase. Email verification is parallel/post-purchase. |
| **Reduced abandonment** | Forcing email verification before purchase adds 30-60s friction at the most critical moment (when intent is highest). Conversion drops measurably. |
| **Receipt deliverability is acceptable risk** | Receipts go to the email captured at Stripe checkout (which IS captured at the point-of-sale and tied to the Stripe Customer). Even if `user.verified_at` is None, the address is the one the user typed at signup; bounces show in SendGrid Activity Feed; admin follow-up is possible. |
| **Chargeback handling** | Stripe handles disputes through Stripe's own contact; chassis's `verified_at` status doesn't affect chargeback workflow. |
| **Adopter flexibility** | Regulated industries / enterprise SaaS adopters who genuinely need verified-only payment flip the flag True. Chassis default doesn't constrain them. |

## Design decisions (pre-committed)

1. **New Settings field `BILLING_REQUIRE_VERIFIED: bool = False`** in `backend/app/config.py`. Inline comment per the 0018 audit pattern (un-validated chassis-tunable behavior toggle).

2. **New dependency `require_billing_user`** in `backend/app/dependencies.py` (or wherever existing deps live). Internally checks the flag:
   ```python
   async def require_billing_user(
       user: User = Depends(get_current_user),
   ) -> User:
       if settings.BILLING_REQUIRE_VERIFIED and user.verified_at is None:
           raise HTTPException(
               status_code=403,
               detail={"code": "EMAIL_NOT_VERIFIED", "message": "..."},
           )
       return user
   ```
   This is the chassis-correct pattern. FastAPI deps are statically declared at route-decoration time, so the conditionality has to live INSIDE the dep, not around it.

3. **Three billing endpoints swap their dep** from `Depends(require_verified)` to `Depends(require_billing_user)`:
   - `POST /billing/topup`
   - `POST /billing/subscribe`
   - `POST /billing/setup-intent`

4. **Error response when flag=True and user is unverified** matches the existing `require_verified` dep's error contract (so existing frontend error-mapping doesn't break). Verify the existing dep's error code (likely `EMAIL_NOT_VERIFIED` or similar) and mirror exactly.

5. **No change to existing `require_verified` dep.** Other code paths that use it (if any) keep current behavior. The new `require_billing_user` is a billing-specific dep that opts into the chassis-tunable posture.

6. **Test isolation per Kind-1** (per 0023.1): existing tests that asserted "unverified → 403" patch `BILLING_REQUIRE_VERIFIED=True` per test scope. New tests cover the OFF state explicitly.

7. **`cloudbuild.yaml` staging deploy:** add `BILLING_REQUIRE_VERIFIED=false` to `--set-env-vars` for explicit clarity (matches default; explicit is clearer for adopters reading the deploy config).

8. **No `chassis-contract.md` entry.** Behavior toggle, not startup invariant. Same shape as 0024.2.

9. **`payments.md` updates** — three additions:
   - §Flows item 2 (Topup): note the chassis posture is permissive by default; verified-only restored via flag.
   - §Flows item 4 (Subscribe): same note.
   - New brief §Verified-gate posture subsection (or amend §User states): document the flag and its effect on the three billing endpoints.

10. **`auth.md` updates** (small): if there's a paragraph describing "paid actions require verified email," update to note this is now chassis-tunable; default permissive per 0024.3.

11. **No frontend changes.** SubscribeForm + TopupForm error mapping already handles 403 EMAIL_NOT_VERIFIED gracefully (per 0023's TopupForm error states). When flag is True, the same UX appears as before; when False, the 403 path simply never fires.

12. **No `backend/.env.example` change strictly required** (default already False; absence equals False). Optional: add the field for adopter clarity.

## Out of scope

- **Other verified-gated capabilities outside billing** — none currently exist that this ticket should change.
- **Per-endpoint flag granularity** (e.g., topup permissive but subscribe strict) — too project-specific. The chassis flag is account-wide. Adopters wanting finer control wrap the chassis with their own deps.
- **0020 legal acceptance** — orthogonal; ToS gating is a separate flag-gated dep concern; not addressed here.
- **Receipt-email deliverability hardening** — chassis email subsystem already handles bounces via SendGrid; no change needed.
- **Project-layer UX messaging** about the implication of paying-while-unverified ("verify to enable [feature]") — chassis stays neutral; project decides.
- **Backfill of receipts to verified addresses post-verification** — too speculative; not in v1.

```python
async def require_billing_user(
    user: User = Depends(get_current_user),
) -> User:
    """Conditional verified-gate for billing endpoints.

    - When BILLING_REQUIRE_VERIFIED=False (chassis default): any authed user
      can call billing mutation endpoints. Permissive posture matches
      industry-standard SaaS UX (let people pay without friction).
    - When BILLING_REQUIRE_VERIFIED=True: unverified users get 403
      EMAIL_NOT_VERIFIED — restores the legacy chassis behavior for adopters
      with stricter posture needs.
    """
    if settings.BILLING_REQUIRE_VERIFIED and user.verified_at is None:
        raise HTTPException(
            status_code=403,
            detail={
                "code": "EMAIL_NOT_VERIFIED",
                "message": "Please verify your email address before continuing.",
            },
        )
    return user
```

The error response shape MUST match what the existing `require_verified` dep returns (so frontend error-mapping in TopupForm + SubscribeForm continues to work without change). Verify the existing dep first; if the format differs, mirror it exactly.
