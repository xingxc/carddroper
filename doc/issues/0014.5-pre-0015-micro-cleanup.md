## Context

After 0014 (cleanup batch) closed, the orchestrator re-triaged the audit items that were parked as "tracked deferrals" in 0015. Four of the seven items are cheap to fix now and genuinely improve reliability of 0015 delivery:

- **0009 F-2 viewport meta** — missing in `app/layout.tsx`; without it, the four user-facing pages shipped under 0015 render desktop-width on mobile, giving false confidence on mobile smoke tests.
- **0009 F-6 tsconfig tightening** — current config only sets `strict: true`. 0015 writes ~5 new form/api/context files; stricter rules applied now are cheaper than retrofitting.
- **Backend F-6 smoke sleep fragility** — flagged in `doc/audits/2026-04-21-backend-pre-0014-audit.md`. If `smoke_verify_email.py` copies the pattern in 0015 Phase 2, the fragility propagates.
- **Dockerfile-COPY postmortem** — the 0014 Phase 1 Cloud Build failure (`COPY failed: stat app/public: file does not exist` after deleting `next.svg` as the last file in `public/`) deserves a bullet in `doc/operations/testing.md` §Per-ticket checklist so no future agent repeats it.

The remaining three deferrals stay deferred with explicit triggers (documented in 0015's Tracked deferrals section): Backend F-3 `updated_at` until a bulk UPDATE route is written; verify-email 401 edge until 0015 Phase 1 exposes the UX; Frontend F-9 `.dockerignore *.md` until the file is touched for another reason.

## Acceptance

### Phase 0 — frontend hygiene (frontend-builder)

1. **Viewport meta.** In `frontend/app/layout.tsx`, add a Next.js 16 Metadata API `viewport` export:
   ```typescript
   import type { Metadata, Viewport } from "next";

   export const viewport: Viewport = {
     width: "device-width",
     initialScale: 1,
   };
   ```
   Do NOT set `viewport` inside the existing `metadata` export — Next.js 16 warns on that path.

2. **tsconfig tightening.** In `frontend/tsconfig.json`, add these flags under `compilerOptions`:
   - `"noUncheckedIndexedAccess": true`
   - `"noImplicitReturns": true`
   - `"noFallthroughCasesInSwitch": true`
   - `"forceConsistentCasingInFileNames": true`

   **Skip** `exactOptionalPropertyTypes` — clashes with library types, high churn for low payoff.

3. Fix any type errors these flags surface across existing files (`lib/api.ts`, `app/providers.tsx`, `app/layout.tsx`, `app/page.tsx`, `middleware.ts` if present). Do NOT silence errors with `any` or `// @ts-ignore`; solve them properly. If a fix is non-obvious, stop and report.

### Phase 1 — cloudbuild migrate-step hardening (backend-builder)

4. **Replace `sleep 3` in `cloudbuild.yaml:37`.** Per `doc/audits/2026-04-21-backend-pre-0014-audit.md` §F-6, after launching the Cloud SQL auth proxy as a background process the migrate step does a fixed `sleep 3` before running `alembic upgrade head`. Three seconds is a guess; on a cold start the proxy can take longer, and on a warm start it's wasted time.

   Replace with a bounded `until pg_isready` poll (or equivalent Postgres-readiness probe using tools already available in the migrate-step container) that:
   - Polls at 0.5s–1s intervals.
   - Exits the loop the moment the probe succeeds.
   - Has a hard timeout ≤15s; on timeout, fail loudly with a clear message so Cloud Build aborts rather than silently running alembic against an un-ready proxy.
   - Does not require adding new apt packages if at all avoidable — the existing Cloud SDK + Alembic image already has `psql` / `pg_isready` available in most cases. If not, prefer `python -c` with psycopg2 (already a backend dep) over installing new packages.

   **This file is at the repo root, not under `backend/`.** The "backend only" constraint is relaxed for this specific file — `cloudbuild.yaml` is the backend deployment pipeline and is in backend-builder's legitimate scope.

### Phase 2 — orchestrator doc (no dispatch)

5. Append a bullet to `doc/operations/testing.md` §Per-ticket checklist:

   > **Dockerfile COPY dependencies.** Before deleting the last tracked file in a directory referenced by a Dockerfile `COPY` (e.g. `public/`, `alembic/`), confirm either (a) the directory still has other tracked content, or (b) a `.gitkeep` is added, or (c) the `COPY` line is removed. Git does not track empty directories; a `COPY` that was valid at author time can silently break at build time. See the 0014 Phase 1 postmortem (commit `a8f1915`).

## Resolution

Closed 2026-04-21. All four items delivered.

**Phase 0 — frontend (frontend-builder):**
- `frontend/app/layout.tsx` — added `export const viewport: Viewport = { width: "device-width", initialScale: 1 }` alongside the existing `metadata` export. Next.js 16 now emits `<meta name="viewport">` correctly without the deprecation warning.
- `frontend/tsconfig.json` — added `noUncheckedIndexedAccess`, `noImplicitReturns`, `noFallthroughCasesInSwitch`, `forceConsistentCasingInFileNames`. `exactOptionalPropertyTypes` deliberately omitted. **Zero new type errors surfaced** — the existing 4-file frontend was already clean under all four flags. `npm run lint`, `npm run typecheck`, `npm run build` all green.

**Phase 1 — cloudbuild migrate step (backend-builder):**
- `cloudbuild.yaml` — replaced `sleep 3` in the migrate step with a psycopg2-based poll loop, 1s interval, 15s hard timeout, fail-loud on timeout. Chose psycopg2 over `pg_isready` because `postgresql-client` is not in the migrate image and `psycopg2-binary` is already a backend dep — **no new apt packages, no new dependencies**. Local docker-postgres smoke confirmed both happy path (0.37s to ready) and timeout path (15.10s, exit 1, clear error).
- `pytest 38 passed`, `ruff check` + `ruff format --check` clean.

**Phase 2 — orchestrator doc:**
- `doc/operations/testing.md` §Per-ticket checklist — added item 6: "Dockerfile `COPY` dependencies" bullet capturing the 0014 Phase 1 `public/` postmortem lesson.

**Deviations:** Phase 1 brief initially misidentified the `sleep 3` location as a smoke script; first backend-builder dispatch stopped correctly and flagged the conflict. Ticket was corrected (`cloudbuild.yaml:37` per audit F-6 truth), second dispatch proceeded cleanly.

**Staging verification deferred to user** — the cloudbuild change can only be exercised by the next deploy. Three existing smoke scripts (`smoke_healthz`, `smoke_auth`, `smoke_email`) remain the post-deploy verification suite.
