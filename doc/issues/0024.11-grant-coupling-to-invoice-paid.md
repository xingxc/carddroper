## Audit (per `doc/operations/audit-template.md`)

### 1. User-visible action

User submits a 3DS card. SetupIntent 3DS modal succeeds (PM attached). Subscription created with `payment_behavior=default_incomplete`; first invoice + PI generated; PI in `requires_action` (3DS challenge for the actual subscription charge). User then **fails** the second 3DS challenge (or cancels the modal). Subscription remains `incomplete`; first invoice never paid.

### 2. Full request flow trace

```
1. POST /billing/setup-intent → SetupIntent (consumable)
2. frontend confirmSetup → 3DS modal #1 SUCCEEDS → PM attached
3. POST /billing/subscribe → creates Stripe Subscription (incomplete);
                               first invoice + PI immediately created;
                               PI status = requires_action;
                               backend's 0024.9 three-way branch:
                                 returns 200 + requires_action=true + client_secret;
                               backend ALSO upserts subscriptions row
4. ASYNC: customer.subscription.created webhook arrives
   → handle_subscription_created:
       (a) upserts subscriptions row (idempotent with step 3's upsert)
       (b) calls grant() → posts subscription_grant ledger entry  ← THE BUG
5. frontend confirmPayment(client_secret) → 3DS modal #2 → USER FAILS
6. Stripe transitions PI to requires_payment_method or canceled
7. ASYNC: invoice.payment_failed webhook arrives
   → updates row.status; does NOT touch ledger (correct)
8. invoice.paid NEVER FIRES (the invoice never paid)
```

The grant from 4(b) is permanent. The user never paid. With `flag=true`, this is real value leakage.

### 3. Endpoint properties

| Step | Handler | DB | Ledger | Stripe state at trigger |
|---|---|---|---|---|
| 3 | POST /billing/subscribe | upsert subscriptions | none | Subscription exists in `incomplete` |
| 4 | handle_subscription_created | upsert (re-entrant) | **+grant()** | PI not yet confirmed (3DS pending) |
| 7 | handle_invoice_payment_failed | update status | none | PI failed |
| 8 (never fires for 3DS-fail) | handle_invoice_paid (subscription_create) | (would advance period) | (currently no-op) | (would mean payment succeeded) |

### 4. Consumability check

All resources are correctly classified per `idempotency-policy.md`. **The bug is not in any idempotency or consumability decision.** The bug is in the **event-trigger architecture**: the grant is coupled to subscription existence (`customer.subscription.created`) rather than to payment success (`invoice.paid`).

### 5. Adversarial scenario

`customer.subscription.created` fires the moment Stripe creates the Subscription resource. With `payment_behavior=default_incomplete`, **the first invoice is not yet paid** at that moment. Any path where the first invoice never pays produces a phantom grant:

- 3DS challenge fails or is canceled mid-flow (the reported case)
- Hard decline that race-arrives after the webhook (handle_subscription_created fires before 0024.9's sync cleanup deletes the row)
- Network failure between subscribe POST and frontend confirmPayment
- User abandons mid-3DS, subscription auto-expires at 23h with phantom grant in place

### 6. Test coverage

Existing tests **codify the bug**:
- `test_handle_subscription_created_grants_initial_period` (line ~1025) asserts grant fires when `customer.subscription.created` arrives with flag=true. This test passes the bug; it doesn't catch it.
- `test_handle_invoice_paid_subscription_create_no_op` (line ~1330) asserts `invoice.paid` for `billing_reason=subscription_create` is a no-op for the ledger. This is exactly the wrong handler to be a no-op.

Required composition test (per `doc/operations/testing.md` §Composition tests): `test_incomplete_subscription_does_not_post_subscription_grant_to_ledger` — full flow simulating `customer.subscription.created` arrival when the subscription is in `incomplete` state with no subsequent `invoice.paid`. Asserts no `subscription_grant` ledger row exists for the user.

## Context

The chassis's current event-coupling for grants:

```
customer.subscription.created → grant() [BUG: fires regardless of payment]
invoice.paid (subscription_create) → no-op
invoice.paid (subscription_cycle) → grant subscription_reset [renewal]
```

The architecturally correct coupling:

```
customer.subscription.created → upsert row only, NO grant
invoice.paid (subscription_create) → grant subscription_grant [initial; flag-gated]
invoice.paid (subscription_cycle) → grant subscription_reset [renewal; flag-gated]
```

**Grants must be coupled to `invoice.paid` events, never to `customer.subscription.created`.** This is Stripe's canonical pattern: `invoice.paid` has guaranteed semantics — the invoice was paid, money moved. `customer.subscription.created` makes no such guarantee under `payment_behavior=default_incomplete`.

The reason-vocabulary in `payments.md` stays correct under this change:
- `subscription_grant` = initial period grant (now sourced from `invoice.paid` `subscription_create`)
- `subscription_reset` = renewal grant (`invoice.paid` `subscription_cycle`)

Only the trigger event changes; the reason vocabulary is unchanged.

## Why alternatives were rejected

- **Gate `customer.subscription.created` grant on `sub.status == "active"`** — race-prone; the wrong event-trigger doesn't become correct by gating it on additional conditions. A 3DS that succeeds AFTER the webhook arrives would still leave a window with a stale grant. Use the right event.
- **Defer the grant via a follow-up `invoice.paid` check** — two-step grant logic introduces "intent" state and a third invariant. Stripe already has the right event; use it directly.
- **Refund the grant on `invoice.payment_failed`** — requires reversal ledger entries (negative-`subscription_grant_reversal`), expands the reason vocabulary, leaves a window of over-credit. Better to never grant in the first place.

## Why this didn't surface in earlier 0024.x audits

1. **Most test cards succeed quickly.** `4242` doesn't trigger 3DS, so `invoice.paid` fires near-immediately after `customer.subscription.created`. The bug-grant from step 4 and the would-be-correct-grant from step 8 produce the same end state for happy-path users — the bug was masked.
2. **3DS testing focused on the success path.** Test 7B (3DS challenge succeeds) had been verified passing; the variant where the second 3DS *fails* wasn't part of the matrix.
3. **0024.7 audit looked at WHO writes `grant_micros` (Path B for the field), not WHEN the grant() ledger call fires.** Different bug class; the audit was correctly scoped to its own question.

The new audit-template Question #5 (adversarial scenario) immediately reveals this bug class. The discipline is paying off.

## Design decisions (pre-committed)

1. **Remove `grant()` call from `handle_subscription_created`.** The handler keeps:
   - Upserting the row (status tracking, idempotent)
   - Path B period write (per 0024.5)
   - Path B grant_micros INSERT-only write (per 0024.7)
   The handler stops calling `grant()` regardless of flag state.

2. **Add `grant()` call to `handle_invoice_paid` subscription_create branch.** When `event.data.object.billing_reason == "subscription_create"` AND `BILLING_SUBSCRIPTION_GRANTS_TO_LEDGER=True`:
   - Post a `subscription_grant` ledger entry with `amount_micros = grant_micros` from the row (or from the invoice metadata, matching the renewal handler's pattern)
   - Idempotency via `stripe_event_id` (existing pattern; per 0023.2's atomic dedup)

3. **`handle_invoice_paid` subscription_cycle branch unchanged.** Continues to grant `subscription_reset` for renewals (per 0024.5/7).

4. **Reason vocabulary unchanged.** `subscription_grant` and `subscription_reset` keep their semantic meaning; only the trigger event for `subscription_grant` changes.

5. **No frontend changes.** No schema changes. No migration. No new env vars. No chassis-contract entries.

6. **Backfill is manual, not chassis-managed.** Per established 0024.x posture (0024.7, 0024.8 also deferred backfill). The user can run:
   ```sql
   DELETE FROM balance_ledger
   WHERE id IN (
     SELECT bl.id FROM balance_ledger bl
     JOIN subscriptions s ON s.user_id = bl.user_id
     WHERE bl.reason = 'subscription_grant'
       AND s.status NOT IN ('active', 'trialing', 'past_due')
   );
   ```
   This zeroes phantom grants for subscriptions that never reached a paid state. Chassis won't auto-fix.

## Out of scope

- `subscription_reset` (renewal) trigger — already correctly coupled to `invoice.paid` subscription_cycle.
- `topup` ledger writes — already coupled to `payment_intent.succeeded` (correct).
- Reason vocabulary changes (e.g., adding `subscription_grant_reversal` for refunds) — separate concern.
- Customer Portal redirect — 0025.
- Any frontend changes — the bug is purely backend.

## Chassis implications

After 0024.11, the chassis ledger-write coupling is:

| Ledger entry | Source event | Idempotency |
|---|---|---|
| `topup` | `payment_intent.succeeded` (with `metadata.kind=topup`) | event.id |
| `subscription_grant` | `invoice.paid` (billing_reason=subscription_create) | event.id |
| `subscription_reset` | `invoice.paid` (billing_reason=subscription_cycle) | event.id |
| `debit` | project-layer `billing.debit()` synchronous call | n/a (no Stripe event) |

Every chassis-managed ledger write now traces to a Stripe event that **guarantees money moved**. No ledger writes are coupled to lifecycle-only events that can fire without payment. This eliminates the entire phantom-grant bug class.

The architectural principle is documented in `payments.md` §Reason vocabulary and reinforced by `audit-template.md` Question #5.

No new chassis-contract entries (correctness fix, not new invariant).
