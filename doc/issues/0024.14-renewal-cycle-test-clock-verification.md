## Audit (per `doc/operations/audit-template.md`)

### 1. User-visible action

A subscriber's billing anchor date arrives (30 days after subscribe). Stripe automatically generates a renewal invoice, charges the stored payment method, and fires `invoice.paid (billing_reason=subscription_cycle)`. The chassis must:

- Update `subscriptions.current_period_start/end` to the new period (Path B per 0024.5)
- When `BILLING_SUBSCRIPTION_GRANTS_TO_LEDGER=true`: post a `subscription_reset` ledger entry with `+grant_micros` (per 0024.7's cycle-handler design + 0024.11's grant-coupling)
- When `BILLING_SUBSCRIPTION_GRANTS_TO_LEDGER=false`: update period only, no ledger write

This is invisible to the user except through their balance and the "next billing date" UI.

### 2. Full request flow trace

```
1. Test clock advances 31 days (or wall-clock advances in production)
2. Stripe creates a renewal Invoice
3. Stripe charges the default PM → succeeds (test card configured to succeed)
4. Stripe fires invoice.paid (billing_reason=subscription_cycle, basil shape)
5. Chassis handle_invoice_paid runs:
   a. extract_invoice_subscription_id (basil canonical path)  ← 0024.12
   b. resolve subscriptions row by sub_id
   c. extract period from invoice.lines.data[0].period            ← 0024.5
   d. update row.current_period_start/end                          ← 0024.5
   e. flag=true: extract grant_micros from
      invoice.lines.data[0].price.metadata.grant_micros            ← 0024.7
      → update row.grant_micros (handles mid-sub tier changes)
      → grant() with reason=subscription_reset                     ← 0024.11
   f. flag=false: skip e entirely                                  ← 0024.7+0024.13 invariant
6. Stripe also fires customer.subscription.updated (period advanced)
7. Chassis handle_subscription_updated:
   - status sync only (Path B preserves periods + grant_micros)    ← 0024.5+0024.7
```

### 3. Endpoint properties

| Step | Handler | Idempotency | DB writes | Stripe state |
|---|---|---|---|---|
| 5 | handle_invoice_paid (cycle branch) | per event_id (stripe_events UNIQUE) | row.current_period_*, row.grant_micros (flag=true), balance_ledger | Invoice paid, subscription period advanced |
| 7 | handle_subscription_updated | per event_id | row.status, cancel_at_period_end | Subscription period changed Stripe-side |

### 3.5. Cross-writer assumption check

This ticket is test infrastructure only — NO chassis code is modified. No writers' assumptions change. The cycle branch's invariants documented in `payments.md` §Field ownership matrix are exactly what the script verifies.

### 4. Consumability check

- Invoice: consumable (paid once); chassis idempotency via `stripe_events.id` UNIQUE constraint handles duplicate delivery (0023.2 atomic INSERT).
- SetupIntent / PaymentIntent: not directly involved in renewal (uses stored default PM).

No idempotency-policy violations introduced.

### 5. Adversarial scenario

The cycle branch was refactored 5 times during the 0024.x arc. Most likely failure modes the script will catch:

1. **Period field extraction returns NULL** because basil moved `invoice.lines.data[0].period` (similar pattern to 0024.4 / 0024.12 — basil API drift)
2. **`grant_micros` cycle-update returns 0** when metadata is present (similar to 0024.13 INSERT-path hole, but for cycle)
3. **`subscription_reset` ledger entry not posted** when flag=true (similar regression class to 0024.11 grant-coupling)
4. **Period DOES advance but `subscription_reset` amount mismatches `row.grant_micros`** (handler computes amount from one source, writes row from another)
5. **Two `subscription_reset` entries posted** (idempotency dedup failed; should be impossible per 0023.2 atomic INSERT but worth verifying)

The script asserts each of these.

### 6. Test coverage

Existing unit tests cover the cycle branch with mocked basil-shape events:
- `test_handle_invoice_paid_subscription_cycle_grants` — basic happy path
- `test_handle_invoice_paid_subscription_cycle_updates_periods` — period advance
- `test_handle_invoice_paid_subscription_cycle_grants_when_flag_off` — no-ledger path

But none of these run against real Stripe webhook delivery. The script is the integration test that mocks-tests structurally cannot replicate.

## Context

`stripe-side-tests.md` §Tier B documents the test-clock setup intent. This ticket creates the actual reusable artifact: a Python script that runs the renewal matrix.

## Why a script, not a pytest integration test

Pytest integration tests against external Stripe APIs are brittle: requires test-clock setup/teardown per run, requires database isolation per run, requires careful ordering to not interfere with other tests. A standalone script is:

- Run on demand (Phase 1 manual smoke for any future chassis ticket touching `handle_invoice_paid`)
- Isolated from the unit test suite (no risk of contaminating CI)
- Reusable across multiple renewal cycles (user can run it 12 times to simulate a year)
- Simple: clear inputs (fixture file), clear assertions, clear output

If a future ticket wants to convert this to a CI-runnable integration test, it can — but that's separate scope.

## Design decisions (pre-committed)

### Backend (Phase 0a)

1. **Create `backend/scripts/test_renewal.py`** — Python script (Python is the chassis idiom for scripts; matches existing `scripts/smoke_*.py` pattern). Async via asyncpg/SQLAlchemy to query DB; sync Stripe SDK calls. Uses `argparse` for `--days` and `--dry-run` flags.

2. **Script behavior:**
   - Reads fixture from `backend/.test-clock-fixture.local` (gitignored). Required keys: `customer_id`, `clock_id`, `user_id`, `subscription_id`.
   - Captures pre-renewal DB state: subscriptions row + balance_ledger entries for that user.
   - Advances the test clock via `stripe.test_helpers.TestClock.advance()` (or equivalent SDK call).
   - Waits ~15 seconds for webhook delivery.
   - Captures post-renewal DB state.
   - Asserts:
     - `current_period_start` advanced (post > pre)
     - `current_period_end` advanced (post > pre)
     - When `BILLING_SUBSCRIPTION_GRANTS_TO_LEDGER=true`: exactly ONE new `subscription_reset` ledger entry exists for this user; `amount_micros` matches `row.grant_micros`
     - When flag=false: no new `subscription_reset` or `subscription_grant` ledger entries
   - Prints GREEN summary or RED with which assertion failed.
   - `--dry-run` prints what it would do without advancing the clock.

3. **Create fixture template `backend/.test-clock-fixture.local.example`** — JSON template with field documentation as comments-via-docstring (since JSON doesn't allow comments natively, embed as a separate `_README` key or use a `.local.example.md` companion file).

4. **Add `.test-clock-fixture.local` to root `.gitignore`** — security: leaking Stripe customer/clock IDs is low-risk in test mode but discipline-correct to gitignore.

5. **Document the setup process in `doc/operations/stripe-side-tests.md`** — expand the existing §Tier B "Setup (do this once)" section with explicit reference to the new script: "After completing setup, save the IDs in `backend/.test-clock-fixture.local` (template at `.test-clock-fixture.local.example`); run `python scripts/test_renewal.py` to verify the renewal cycle behavior."

6. **DO NOT modify any chassis code.** This ticket is purely test infrastructure. Verify by `git diff backend/app/` returning empty after the work.

### Out of scope

- **Test-clock teardown automation** — each script run leaves the clock at a new "now"; user can delete and recreate the clock when desired. Not chassis-managed.
- **Multi-cycle stress test** — running the script 12 times in a row is a valid follow-up exercise but doesn't need automation.
- **Mid-cycle tier-change verification (Tier B2)** — covered conceptually in stripe-side-tests.md §B2; deferred to future ticket.
- **CI integration** — pytest integration tests against real Stripe are brittle; explicitly out of scope. Future ticket if/when test-clock isolation is solved.
- **Production renewal verification** — test clocks only; no real billing involved.
- **Failure-path renewal testing** — using a card that fails on renewal is interesting but requires a different test fixture; out of scope.
