## Context

The frontend at `https://staging.carddroper.com` cannot call the backend at `https://api.staging.carddroper.com`. Browser blocks the preflight because the backend responds without `Access-Control-Allow-Origin` — the staging Cloud Run service had `CORS_ORIGINS` unset, so `backend/app/config.py`'s pydantic default (`"http://localhost:3000"`) applied.

Python smokes (`smoke_auth.py`, `smoke_verify_email.py`) passed because `httpx` doesn't send an `Origin` header and doesn't trigger browser CORS enforcement — they test backend correctness, not browser-origin policy. Local docker-compose worked by accident: local default origin matched the local frontend URL exactly.

This is the first time any browser code has called the API. Latent since the backend was first deployed; surfaced by 0015 Phase 1 being the first phase with live browser→API traffic.

The fix is not just one env var. This is a class of bug — "frontend-on-origin-A / backend-on-origin-B" is the default shape for every project reusing this chassis. The structural answer has four pillars, landed together in one commit so a new adopter can never step on the same rake:

1. **Boot-time guard.** Pydantic validator on `Settings` that refuses to construct if `FRONTEND_BASE_URL` is not contained in `CORS_ORIGINS` (also accept a regex match via future `CORS_ORIGIN_REGEX` if set — see §Out of scope). Turns silent misconfiguration into a loud deploy failure.
2. **Env-var correction.** Add `CORS_ORIGINS=https://staging.carddroper.com` to the backend Cloud Run deploy step in `cloudbuild.yaml`.
3. **Post-deploy smoke.** `backend/scripts/smoke_cors.py` — OPTIONS preflight with a real `Origin` header, asserts the backend returns the correct `Access-Control-Allow-Origin` and `Access-Control-Allow-Credentials: true`. Runs alongside the existing smokes in Phase 2 of any future frontend ticket.
4. **Chassis contract doc.** New `doc/operations/chassis-contract.md` containing exactly one invariant today (the CORS rule). Coupled to the enforcement layer: every future `Settings` validator added to the chassis lands its contract entry in the same commit. `CLAUDE.md` gets the coupling rule.

## Design decisions (pre-committed)

- **Fail-loud posture.** The validator raises at `Settings()` construction. Cloud Run health check fails → revision marked unhealthy → rollback. Acceptable because CORS misconfiguration with `allow_credentials=true` is security-relevant (CSRF risk with wrong origin) — startup crash is cheaper than a user-facing break that looks like a server outage.
- **Keep `CORS_ORIGINS` as CSV string.** Existing pattern; pydantic already splits via `cors_origins_list`. No new env var for the common case. Optional `CORS_ORIGIN_REGEX` support is stubbed in the validator for future multi-subdomain projects, but not tested against or documented until a project needs it.
- **Least privilege at deploy.** Staging backend accepts only `https://staging.carddroper.com`. Not `http://localhost:3000`. Local dev uses local backend; no reason for staging backend to honor localhost origins.
- **Smoke script is part of the chassis**, not product. Lives in `backend/scripts/` alongside existing smokes. Takes `--base-url` and `--origin` so it's reusable across projects and environments.
- **Contract doc is coupling-rule driven** (Option C in the pre-ticket discussion). Doc grows only as enforcement grows. No speculative or aspirational entries.
- **No frontend code change.** `lib/api.ts` reports `NETWORK_ERROR` correctly — the diagnostic is accurate; the fix is entirely server-side + infra + docs.

## Acceptance

### Phase 0 — backend guard + smoke + docs + cloudbuild (backend-builder, one dispatch)

1. **`backend/app/config.py`** — add a `@model_validator(mode="after")` on `Settings` (or equivalent pydantic v2 idiom) that asserts `FRONTEND_BASE_URL` is either (a) present in `cors_origins_list` OR (b) matches an optional `CORS_ORIGIN_REGEX` if that field is set. On failure, raise `ValueError` with:
   ```
   CORS misconfiguration: FRONTEND_BASE_URL={frontend_url} is not in CORS_ORIGINS={origins_list}
   and does not match CORS_ORIGIN_REGEX={regex or "(unset)"}.
   A browser served from the frontend URL cannot call this API.
   Set CORS_ORIGINS to include FRONTEND_BASE_URL (CSV) or CORS_ORIGIN_REGEX to match it.
   ```
   The `CORS_ORIGIN_REGEX` field may be added if trivial (`str | None = None`), but it does NOT have to be wired to `CORSMiddleware.allow_origin_regex` in this ticket — stub it if adding the plumbing is more than a couple lines. The validator just needs to honor it when set.

2. **`backend/tests/`** — add unit tests covering:
   - Happy path: `FRONTEND_BASE_URL` in `CORS_ORIGINS` → `Settings()` constructs.
   - Failing path: `FRONTEND_BASE_URL` not in `CORS_ORIGINS` (and no regex) → raises `ValidationError` with the expected message substring.
   - Happy path via regex (if `CORS_ORIGIN_REGEX` stub is wired): matching regex → constructs.
   Skip mocking env vars at the OS level; construct `Settings()` with explicit kwargs.

3. **`cloudbuild.yaml`** — in the backend Cloud Run deploy step (`gcloud run deploy carddroper-backend`), add `CORS_ORIGINS=https://staging.carddroper.com` to the `--set-env-vars` CSV. **This file is at the repo root; backend-builder's default scope is relaxed for this one file** (same precedent as 0014.5 Phase 1).

4. **`backend/scripts/smoke_cors.py`** — mirror `smoke_auth.py` conventions (httpx, argparse). Args: `--base-url` (backend URL, default `https://api.staging.carddroper.com`), `--origin` (frontend URL, default `https://staging.carddroper.com`).
   Assertions (print PASS/FAIL per assertion, final `SMOKE OK: cors` on full success):
   - `OPTIONS /auth/login` with headers `Origin: <origin>`, `Access-Control-Request-Method: POST`, `Access-Control-Request-Headers: content-type` returns 200 or 204.
   - Response header `Access-Control-Allow-Origin` equals `<origin>` (not wildcard; wildcard is incompatible with credentials).
   - Response header `Access-Control-Allow-Credentials` equals `true`.
   - Response `Access-Control-Allow-Methods` includes `POST`.
   - Repeat for `/auth/me` (a GET endpoint) as a second target, to confirm CORS applies uniformly.
   Local dry run must pass against docker-compose (`python backend/scripts/smoke_cors.py --base-url http://localhost:8000 --origin http://localhost:3000`).

5. **`doc/operations/chassis-contract.md`** — new file, single invariant entry today:
   - Frontmatter / intro: "This document lists the invariants the chassis enforces at startup. Each entry corresponds 1:1 to a validator or middleware check in chassis code. Adopters: violating any invariant causes the service to refuse to start. Authors: do not add entries without matching enforcement."
   - One section: `CORS_ORIGINS ⊇ {FRONTEND_BASE_URL}` — required, purpose, error message the chassis emits, enforcement location (`backend/app/config.py::Settings.validate_cors_origins` or whatever the final symbol is).

6. **`doc/README.md`** — add one line under "Operations" pointing to `operations/chassis-contract.md`.

Position this rule in the project-conventions portion of CLAUDE.md so reviewers see it.

## Resolution

Closed 2026-04-22. All four pillars landed in commit `4eb33e0`.

**Staging verified:**
- `smoke_cors.py --base-url https://api.staging.carddroper.com --origin https://staging.carddroper.com` returns `SMOKE OK: cors`. Both `OPTIONS /auth/login` and `OPTIONS /auth/me` respond with exact `Access-Control-Allow-Origin: https://staging.carddroper.com` (not wildcard) and `Access-Control-Allow-Credentials: true`.
- Browser requests from `staging.carddroper.com` to `api.staging.carddroper.com` no longer blocked.

**Pattern established for reuse:**
- `chassis-contract.md` is coupled 1:1 to the enforcement layer. Validator + entry land in the same commit; nothing speculative, nothing aspirational.
- `CLAUDE.md` §Chassis contract (coupling rule) documents the policy for all future chassis work. First re-use: 0015.6 added a second entry (cookie-domain invariant) under the same rule.
- `smoke_cors.py` env-var-parameterized so future chassis-based projects reuse it unchanged.
