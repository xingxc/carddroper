## Context

Two issues, addressed together:

**Bug:** `handle_subscription_created` (and likely `handle_subscription_updated`) attempts to extract the Price object from `sub.items` and fails, logging `"could not extract price from sub.items"` and returning without granting. Result: subscription state lands in DB but `balance_ledger` gets nothing. Root cause (almost certainly): Stripe's `subscription.items` is a paginated list object (`{data: [...], has_more, ...}`), not a plain iterable. The handler likely accesses `sub.items[0]` when it should access `sub.items.data[0]`. The chassis subscribe endpoint already does this correctly when reading the Price at subscribe-time (otherwise subscribe wouldn't have succeeded), so this is purely a webhook-handler-side bug.

**Design improvement:** the chassis is currently opinionated as credit-based (per `payments.md §Denomination — gift-card model`). Subscriptions ALWAYS write to `balance_ledger` via `subscription_grant` and `subscription_reset`. But the broader SaaS world has two equally-valid models:

- **Flat-fee SaaS** (Netflix, Slack, GitHub, Notion): subscription = access tier; no balance concept; user pays $X for capability, not credits.
- **Credit-based SaaS** (Twilio, SendGrid, OpenAI, Stripe Issuing, this chassis's existing topup design): subscription = balance grant + access; user pays $X for $Y of spendable balance.

A chassis aiming for adopter-friendliness across both should make grant-on-subscribe **opt-in**. Default-OFF means flat-fee adopters get sensible defaults; credit-based adopters (carddroper) explicitly opt in via env var.

This ticket adds the toggle AND fixes the bug — the bug fix is required so that toggle=ON correctly grants when carddroper enables it on staging.

## Design decisions (pre-committed)

1. **New Settings field `BILLING_SUBSCRIPTION_GRANTS_TO_LEDGER: bool = False`.** Default OFF. Opt-in for credit-based adopters.

2. **Gating:** when False, `handle_subscription_created` and `handle_invoice_paid` (subscription_cycle branch) skip the `grant()` call. Subscriptions row still upserts, status mirrors Stripe, but `balance_ledger` is untouched. When True, current chassis behavior — grants happen.

3. **No effect on `handle_subscription_updated`/`deleted`/`payment_failed`** — these handlers don't grant; they sync state only. No flag check needed.

4. **`metadata.grant_micros` becomes optional when flag is OFF.** When the flag is OFF, the chassis doesn't need `grant_micros` in Price metadata at all. Subscribe endpoint should NOT 422 on missing `grant_micros` when the flag is OFF. When the flag is ON, `grant_micros` becomes required (current behavior). `tier_name` remains required in both modes (always used for display).

5. **Bug fix in `handle_subscription_created` (and `handle_subscription_updated` if affected): extract Price from `sub.items.data[0].price`** (not `sub.items[0].price`). Verify the agent's implementation; fix wherever the wrong access pattern lives.

6. **Tests cover both states** (Kind-1 isolation per 0023.1):
   - Existing tests assume grant happens → patch `BILLING_SUBSCRIPTION_GRANTS_TO_LEDGER=True` per test or via test class setup.
   - New tests for OFF state: subscribe → status=active in subscriptions row, NO ledger entry; renewal `invoice.paid` → subscriptions period_end updated, NO ledger entry.
   - Bug-fix regression test: handle_subscription_created with a real-shaped event (with `items.data[].price`) → grant fires when flag=True. Without the fix this test would log the warning + skip; with the fix it grants.

7. **Subscribe endpoint behavior change** — when flag is OFF, drop the `grant_micros` requirement check so subscribe can succeed without that metadata. When flag is ON, keep the existing requirement.

8. **`cloudbuild.yaml` staging update:** add `BILLING_SUBSCRIPTION_GRANTS_TO_LEDGER=true` to `--set-env-vars` for the carddroper-staging deploy. Carddroper is credit-based per design.

9. **No new chassis-contract.md entries.** Not a startup-time invariant; just a behavior toggle. Inline-comment in `config.py` per 0018 audit pattern.

10. **`payments.md` updates** — three additions:
    - §Denomination: note that the gift-card model is the chassis's *credit-based mode*; flat-fee mode is supported via `BILLING_SUBSCRIPTION_GRANTS_TO_LEDGER=False`.
    - §Subscription tier contract: clarify `grant_micros` is required when flag is True; optional when False.
    - §Flows item 4 + 5: note the flag's effect on grant ledger entries.

## Out of scope

- **Backfill grants for already-existing subscriptions** when the flag flips ON — adopters running production who flip the flag from OFF to ON later can write a one-shot script themselves; chassis doesn't auto-backfill.
- **UX messaging** ("Subscribe and get $X/month credit") in the chassis SubscribeForm — project layer decides messaging based on whether they enable the flag. Chassis stays neutral.
- **Per-tier grant override** (some tiers grant balance, others don't) — too project-specific. The chassis flag is account-wide. Adopters wanting per-tier behavior can write a project-layer wrapper around the chassis handlers.
- **Defaulting `grant_micros` to `unit_amount * 10000`** — that was the previously-discussed 0024.2 idea; deferred. This 0024.2 takes the "make grants opt-in" angle instead, which is more general.
- **Chassis-contract entry** for the new flag — not a startup invariant.

**1. Bug fix in `handle_subscription_created` (and audit `handle_subscription_updated` for the same pattern):**

Inspect the current implementation. The code is logging `"could not extract price from sub.items"` for valid subscription events. Stripe's `subscription.items` is a `ListObject` (`{data: [...], has_more, ...}`), not a plain list. Whatever access path the handler uses (likely `sub.items[0]` or `sub.items[0].price`), correct it to traverse `.data` first:

```python
# Wrong (current):
item = sub.items[0]   # IndexError or attribute error on ListObject

# Right:
items = sub.items.data if hasattr(sub.items, "data") else sub.get("items", {}).get("data", [])
if not items:
    logger.warning("handle_subscription_created: empty items.data", extra={"sub_id": sub.id})
    return
price = items[0].price
```

Or use bracket-access throughout for resilience to dict-vs-object Stripe SDK:
```python
price = sub["items"]["data"][0]["price"]
```

Apply the same fix to `handle_subscription_updated` if it also reads price from items (likely yes since plan changes update tier_key/grant_micros from re-read Price metadata).

In `handle_subscription_created`:

In `handle_invoice_paid` (subscription_cycle branch):
- If `settings.BILLING_SUBSCRIPTION_GRANTS_TO_LEDGER is False`: log info, return WITHOUT calling `grant()`. (Still update `current_period_*` on the subscriptions row.)

## Chassis implications

After 0024.2:
- Chassis supports both flat-fee and credit-based subscription models via a single env var.
- Default-OFF means flat-fee adopters get the sensible default — chassis doesn't write balance state they don't want.
- Credit-based adopters (like carddroper) explicitly opt in.
- The `metadata.grant_micros` Stripe-side requirement becomes conditional — adopters who don't enable the flag don't need to set it on their Prices.
- The bug fix unblocks credit-based use end-to-end; without it, even adopters who enable the flag wouldn't get grants.
