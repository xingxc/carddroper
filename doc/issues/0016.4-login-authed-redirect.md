title: /login auto-redirects to /app when silent refresh re-auths a visitor — completes the proxy-symmetric redirect pair
status: open
priority: medium (UX polish; not a security or data-integrity issue, but produces a visible inconsistency after any access-token-expired navigation)
found_by: local boundary testing (2026-04-22) with `JWT_EXPIRATION_MINUTES=1` and `REFRESH_TOKEN_DAYS=0.01`. After access-token expiry but before refresh-token expiry, reloading `/app` correctly redirected to `/login` (proxy saw no access_token cookie), but then directly navigating to `/app` rendered the app as authed — because `/login`'s own `/auth/me` mount-fetch triggered the 401 interceptor's silent refresh, which succeeded, re-issuing the access_token cookie. The user remained on `/login` (no client-side effect to redirect them) despite being fully re-authed.

## Context

The proxy at `frontend/proxy.ts:18-25` has two complementary rules:
- `/app` + no access_token cookie → 307 to `/login`.
- `/login` or `/register` + access_token cookie → 307 to `/app`.

These handle the **initial-request** authed/unauthed redirect. They cannot re-fire on client-side auth-state changes, because middleware only runs on HTTP requests — not on React state transitions.

0016.2 closed half the gap: when `useAuth` resolves to `!isLoading && !isAuthenticated` on `(app)/layout`, the effect `router.replace("/login")`s. That handles the case where the client-side becomes unauthed while on an authed page (cross-device reset, access-token expiry past refresh window, etc.).

The **inverse gap remains**: when the client-side becomes *authed* while on a public auth page (`/login` specifically), no effect redirects the user away. They remain on `/login`, confused.

### Specific reproduction (local, with reduced TTLs)

With `JWT_EXPIRATION_MINUTES=1` and `REFRESH_TOKEN_DAYS=0.01`:

```
t=0:   Log in. access_token cookie Max-Age=60s, refresh_token Max-Age=864s.
       HAS_SESSION_KEY="1" in sessionStorage.
t=65s: Reload /app.
       → GET /app. Proxy: no access_token (browser deleted expired cookie).
         Rule fires: 307 to /login.
       → GET /login. Proxy: still no access_token. Allows through.
       → /login page mounts. AuthProvider's useQuery for /auth/me fires.
       → GET /auth/me → 401.
       → 401 interceptor: hasSession()=true → POST /auth/refresh.
       → POST /auth/refresh → 200 with fresh access_token Set-Cookie.
         Browser stores new access_token with fresh 60s Max-Age.
       → Retry GET /auth/me → 200 with user data.
       → useAuth transitions to isAuthenticated=true.
       → /login form stays rendered. No effect redirects the user away.
t=70s: User types /app in address bar.
       → GET /app. Proxy: access_token cookie is now present (fresh from t=65s refresh).
       → /app renders as authed.
```

Two navigations for essentially the same state — `refresh_token valid, access_token refreshable` — produce different observable outcomes: reload says "logged out" (via proxy redirect), address-bar navigation says "logged in" (after silent refresh ran). Confusing UX.

In production with 15-min access + 7-day refresh, the window where this manifests is smaller but real: any page reload or cross-origin navigation happening in the first few seconds after an access-token expiry would land on `/login`, then silent-refresh behind the scenes.

### Industry-standard pattern

This problem is a known artifact of the "edge-middleware auth gating + client-side silent refresh" architecture that modern Next.js/FastAPI SPAs use. Standard fix: mirror the unauth-redirect effect with an authed-redirect effect on the public auth pages.

- Clerk ships `<RedirectToSignIn />` / `<RedirectToUserProfile />` for this exact pattern.
- Auth0 Next.js SDK does the equivalent inside `withPageAuthRequired`.
- NextAuth.js docs recommend `useEffect(() => { if (session) router.replace('/app') }, [session])` in login pages.
- GitHub, Stripe, Linear all redirect away from `/login` when the session is valid.

## Design decisions (pre-committed)

- **Scope: `/login` only.** Not `/register`, not `/verify-email-sent`, not the reset flow, not verify-email.
  - `/register`: adding the effect creates a race with the register onSubmit (which invalidates `['auth','me']` then `router.push('/verify-email-sent')`). During the invalidate's await, `useAuth` may transition to authed and the effect would fire `router.replace('/app')`, conflicting with the intended push. The spurious-silent-refresh-on-register case is rare (requires stale access + valid refresh + explicit navigation to `/register`), and accepting this one corner-case is simpler than adding an `isSubmitting` guard.
  - `/verify-email-sent`: authed-but-unverified users legitimately belong there.
  - `/forgot-password`, `/reset-password`, `/verify-email`: token-driven flows; cookie state irrelevant for the operation.
- **Redirect target: `/app`.** Not verified-status-aware. An authed user belongs on `/app` regardless of `verified_at` — soft-cap allows unverified users there for 6 days per auth.md. Matches the proxy's own authed-redirect-from-/login rule.
- **`router.replace`, not `router.push`.** The `/login` URL shouldn't enter history when the user was only there as an artifact of the access-token expiry race. Hitting Back from `/app` should take them wherever they came from, not back to `/login`.
- **No guard on `isSubmitting`.** Not needed — `/login`'s own onSubmit also ends at `router.push('/app')`. Effect and submit both aim at `/app`; no race, no conflict. (Contrast with `/register` where the two would target different destinations — see scope decision above.)
- **No change to `LoginBody`'s existing `useSearchParams` logic** (the `?reset=success` banner). That still renders for unauth'd visitors; the new effect only fires when authed, which by definition is after silent refresh succeeded, which by definition means the banner's purpose (prompt re-login after reset) no longer applies.
- **No backend change, no proxy change, no chassis-contract entry.** Behavioral UX rule, not a startup invariant.

## Acceptance

### Phase 0 — frontend (frontend-builder, one dispatch)

1. **`frontend/app/(auth)/login/LoginBody.tsx`** — add a `useEffect` that redirects authed visitors to `/app`.

   Imports: add `useEffect` from `react` (currently only `useState` is imported from react; add to existing import statement). `useRouter` and `useAuth` are already imported.

   Destructure: extend the existing `const { markLoggedIn } = useAuth();` to `const { markLoggedIn, isLoading, isAuthenticated } = useAuth();`.

   Effect placement: right after the destructure, before the `useForm` call. Exact code:

   ```tsx
   useEffect(() => {
     if (!isLoading && isAuthenticated) {
       router.replace("/app");
     }
   }, [isLoading, isAuthenticated, router]);
   ```

   No other code changes. `useSearchParams` banner logic, form submission, error handling all stay as-is.

2. **No other files touched.** Not `register/page.tsx`, not `(auth)/layout.tsx`, not the proxy, not lib/api.ts, not context/auth.tsx.

## Chassis note

The authed-redirect-from-`/login` effect is **chassis**. Every future project built on this chassis has the same edge-middleware-plus-silent-refresh architecture and will hit the same UX inconsistency without this fix. 0016.2 (unauth from `/app`) + 0016.4 (authed from `/login`) together form a symmetric pair that should stay part of the chassis. Document in `STARTER.md` when the chassis is extracted (per `site-model.md` Reusability constraint 7).
