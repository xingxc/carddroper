## Context

0015 Phase 1 shipped the `/register` page without a Terms of Service checkbox, explicitly deferring the legal-acceptance gate to this ticket. The ticket is a hard prerequisite for **charging users money** — collecting payment without recorded consent to the ToS is a compliance risk we won't run in production, and it's the natural gate between the free-tier email layer (0015) and the paid Stripe layer (PLAN.md §10 item 6).

Current state:
- `doc/legal/terms-of-service.md` — draft, marked `DRAFT — starting point, requires attorney review before launch` in `doc/README.md`.
- `doc/legal/privacy-policy.md` — same.
- Frontend has no `/legal/*` routes.
- Register form has no consent checkbox; backend `POST /auth/register` doesn't record a `terms_accepted_at` column on users.
- Brand source: `frontend/config/brand.ts` — domain, support email, etc. live there.

This ticket adds the checkbox, the two public static pages, and the backend column + validation so consent is recorded with timestamp + ToS version.

## Design decisions (pre-committed)

- **Version the ToS.** Store the version the user accepted (e.g. `"2026-04-22"` or `"v1"`) on the user row alongside the accepted timestamp. Future revisions → new version → prompt existing users to re-accept on next login. Without versioning, "they accepted" is ambiguous.
- **Store on the user, not a separate audit log.** A single column per user (`terms_accepted_version: str`, `terms_accepted_at: datetime`) is enough for launch. A full audit trail (every acceptance with IP, UA, delta) is over-engineering for v0.1.0. Reconsider once legal actually requests it.
- **Backend enforces the checkbox, not only the frontend.** Register without `terms_accepted=true` returns 422 with a clear field-level error on `terms_accepted`. Defense-in-depth: front-end Zod catches it first, backend is the authoritative gate.
- **Static pages are MDX or plain markdown rendered as HTML.** No CMS, no dynamic content. Copy lives in the repo (`frontend/app/(marketing)/legal/terms/page.tsx` and `.../privacy/page.tsx`, or an MDX equivalent). Source copy mirrors `doc/legal/*.md` drafts; keep them in sync manually until a CMS use case emerges.
- **Public routes under `(marketing)/legal/`.** Consistent with the site-model `(marketing)/` route group — these pages are public and unauthenticated. Link from the auth-aware marketing header footer if one is added; at minimum, linked from the register-form checkbox.
- **Chassis note.** "Legal acceptance is a capability required to register" is a chassis concern — every future project built on this chassis needs this. The specific ToS content is product-specific (body), but the wiring (column, validator, page routes, checkbox) is chassis. Copy-pasteable.
- **Reject scope creep:** this is not the ticket for cookie banner / GDPR consent management / user-data-export tooling / right-to-be-forgotten workflows. Legal will want those eventually — separate tickets.

## Acceptance

### Phase 0 — backend (backend-builder)

1. **Migration + model**: add to `backend/app/models/user.py`:
   - `terms_accepted_version: Mapped[Optional[str]]` (nullable only so existing users pre-migration don't block — once all users have a value, tighten).
   - `terms_accepted_at: Mapped[Optional[datetime]]` (nullable, tz-naive UTC to match other datetime fields).
   - Alembic migration named `add_terms_acceptance_to_users.py` or similar, adding the two columns. Autogenerated is fine after model change.

2. **Register endpoint** (`backend/app/routes/auth.py::register`):
   - Add `terms_accepted: bool` and `terms_version: str` to `RegisterRequest` pydantic model.
   - If `terms_accepted is not True` → 422 with message `"You must accept the Terms of Service and Privacy Policy to create an account."` and field context pointing to `terms_accepted`.
   - If `terms_version` doesn't match a constant (e.g. `CURRENT_TERMS_VERSION = "2026-04-22"` in `app/config.py` or a `app/legal.py` module) → 422 with message `"Terms version is out of date, please refresh the page and try again."`
   - On success: store `user.terms_accepted_version = body.terms_version` and `user.terms_accepted_at = datetime.now(timezone.utc).replace(tzinfo=None)` before the `_set_auth_cookies` call.

3. **Settings**: add `CURRENT_TERMS_VERSION` to `backend/app/config.py` with a sensible default (e.g., the date the ToS draft was frozen). No validator needed (not an invariant — just a constant the register endpoint reads).

4. **Tests** (`backend/tests/test_auth_flow.py` or a new `test_terms_acceptance.py`):
   - Happy path: register with `terms_accepted=true` + current version → 200.
   - Missing checkbox: register without `terms_accepted` or with `false` → 422 with field context on `terms_accepted`.
   - Stale version: register with `terms_version="2020-01-01"` → 422 with the "out of date" message.
   - DB assertion: after successful register, `user.terms_accepted_version == CURRENT_TERMS_VERSION` and `user.terms_accepted_at` is a recent timestamp.

### Phase 1 — frontend (frontend-builder)

5. **`/legal/terms` and `/legal/privacy` static pages** under `frontend/app/(marketing)/legal/terms/page.tsx` and `.../privacy/page.tsx`:
   - Render the content of `doc/legal/terms-of-service.md` and `doc/legal/privacy-policy.md`. Simplest shape: copy the markdown into the page component as JSX, formatted with Tailwind typography utilities (`prose`-like classes) or hand-styled. If using MDX, add `@next/mdx` as a dep — only if the content warrants it; for v0.1.0 drafts, inline JSX is fine.
   - Routes must be publicly accessible (no auth required). `(marketing)/` route group handles this by default.
   - Include a "Last updated: YYYY-MM-DD" line matching `CURRENT_TERMS_VERSION`.
   - Add a footer link back to `/` for navigation.

6. **Register page checkbox** (`frontend/app/(auth)/register/page.tsx`):
   - Add a `terms_accepted: z.literal(true, { errorMap: () => ({ message: "You must accept the Terms of Service and Privacy Policy." }) })` field to the Zod schema.
   - Render as a checkbox with an inline label that reads: `I agree to the <Link to /legal/terms>Terms of Service</Link> and <Link to /legal/privacy>Privacy Policy</Link>.`
   - Links open in a new tab (`target="_blank" rel="noopener"`) so filling the form isn't lost.
   - On submit: include `terms_accepted: true` and `terms_version: <CURRENT_VERSION>` in the request body. `CURRENT_VERSION` constant lives in `frontend/config/brand.ts` or a new `frontend/config/legal.ts`. Must match the backend's value; drift → 422, caught at test time.
   - On 422 with field `terms_accepted`: render the error inline next to the checkbox, not as a form-level error.

7. **Brand config**: add `TERMS_VERSION` to `frontend/config/brand.ts` (or a new `frontend/config/legal.ts` if brand.ts feels wrong). Importable from register page and anywhere else that needs it.

### Phase 2 — legal content finalization (user-owned; can precede or follow Phase 0/1)

8. Attorney review of `doc/legal/terms-of-service.md` and `doc/legal/privacy-policy.md` drafts. Incorporate edits. Update `CURRENT_TERMS_VERSION` to reflect the attorney-reviewed version date.

### Phase 3 — staging + smoke (user)

9. Push, redeploy, run smoke_auth: register call must include `terms_accepted=true` and `terms_version` — update `smoke_auth.py` to include these fields in its register payload (small addition to the existing backend-builder Phase 0).
10. Manual browser check: `/register` shows the checkbox; submitting without checking shows inline error; `/legal/terms` and `/legal/privacy` render content; links from checkbox open correct pages.

## Out of scope

- Full GDPR consent management (cookie banner, right-to-access, right-to-delete). Separate future ticket when EU customers or a DPO becomes a concern.
- Versioned re-acceptance UX for existing users when ToS revisions land. Next-time concern; for v0.1.0 there are no "existing users" prior to launch.
- A CMS for the legal pages. In-repo JSX is fine for v0.1.0.
- Audit-trail tables (per-acceptance rows with IP/UA/delta). Over-engineering for pre-launch; single column per user is sufficient.
- Marketing emails consent opt-in. Different from transactional ToS consent; separate ticket if/when marketing emails ship.

## Chassis note

The wiring pattern (column + validator + checkbox + static pages) is a chassis concern — every future project will need this shape. Consider:

- Generalize `CURRENT_TERMS_VERSION` as a chassis setting.
- `terms_accepted_version` + `terms_accepted_at` columns on `User` are chassis.
- `/legal/*` route skeleton is chassis; the *content* is body (product-specific, drafted per attorney review per project).
- Register-form checkbox + Zod field pattern is chassis.

No `chassis-contract.md` entry needed — this is a behavioral/structural pattern with no startup invariant to validate. It's a convention captured in this ticket; document in STARTER.md if/when the chassis is extracted (per site-model.md Reusability constraint 7).
