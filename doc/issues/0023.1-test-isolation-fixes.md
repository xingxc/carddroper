## Context

The test suite has two structurally-different isolation bugs tied to `BILLING_ENABLED`. Originally surfaced as "2 failing tests"; expansion on 2026-04-23 clarified that a further 10 tests are affected by a different mechanism.

### The two kinds of isolation bug

| Kind | What the test wants | What goes wrong | Correct fix |
|---|---|---|---|
| **Kind 1 — runtime-path** | `BILLING_ENABLED=false` so the register handler takes its "no billing hook" branch | Test relies on `.env` default; user's `.env` has it `=true`; register's billing hook fires; mocked `stripe.Customer.create` returns a `MagicMock` whose `.id` is another `MagicMock`; SQLAlchemy commit path misbehaves as `MissingGreenlet` | `patch.object(settings, "BILLING_ENABLED", False)` inside the test scope — effective because the register handler re-reads settings per request |
| **Kind 2 — feature-gated** | `BILLING_ENABLED=true` so the billing router exists on the app | `main.py:108-111` makes the `app.include_router(billing_router)` decision at **module-import time**. Once the app is built, patching settings mid-test doesn't retroactively mount routes. The shared `client` fixture's app has either the routes or doesn't, fixed | `@pytest.mark.skipif(not settings.BILLING_ENABLED, ...)` — module-level `pytestmark` if the entire file is feature-gated, per-test decorator otherwise |

### The two specific Kind-1 cases (original 0023.1 scope — FIXED, commit `5f2b355`)

1. **`test_register_does_not_create_customer_when_billing_disabled`** (`test_billing_foundation.py:301`) — asserted `BILLING_ENABLED=false` without explicit patch; failed under user's `.env=true` with `MissingGreenlet` cascade. Fixed by adding `patch.object(settings, "BILLING_ENABLED", False)` to the `with` block.

2. **`test_webhook_not_mounted_when_billing_disabled`** (`test_billing_foundation.py:522`) — asserted that `/billing/webhook` returns 404 when disabled; shared `client` fixture couldn't un-mount the route. Fixed by extracting a local `_make_app_with_billing_disabled()` helper that builds a fresh FastAPI app with `BILLING_ENABLED=false` at init time, then asserting no `/billing/*` paths in the resulting app's routes list.

### The ten Kind-2 cases (expansion scope — pending)

Under `BILLING_ENABLED=false` in `.env`, `test_billing_topup.py` tests fail because the billing router isn't mounted on the shared `client` fixture's app. Affected tests exercise `POST /billing/topup`, `GET /billing/balance`, and the dispatch-registry behavior through the HTTP layer. The agent's Phase 0a verification reported 10 failures of this class under `.env=false`.

These cannot be fixed by patching settings mid-test (Kind-2 problem — routes are baked in at module-import time, not re-read per request). The chassis-honest fix is **feature-gating the tests**: if billing is off, these tests don't make sense to run, so they should skip. This matches how Django / LaunchDarkly / standard feature-flag testing works.

### Why `skipif` is the right shape (not a session-wide conftest override)

A tempting alternative would be to force `os.environ["BILLING_ENABLED"] = "true"` at the top of `conftest.py` so every test session has billing enabled. Rejected because:

- **It lies about the env state.** The test harness becomes a compensating mechanism rather than an honest exercise of the feature flag.
- **It forces real-Stripe-mocking at session scope** so auth tests don't create leftover Customers. More infrastructure; more surprise for future contributors.
- **Skipif is standard.** Feature-flag testing patterns (Django, LaunchDarkly, Flipper) use skip-if-disabled as the documented shape.
- **Lower blast radius.** Skipif is a declarative annotation per file/test; conftest override is a session-wide global patch that future tests must remember exists.

## Design decisions

1. **Two patterns for two kinds of bug.** Kind 1 uses explicit `patch.object(settings, "BILLING_ENABLED", ...)` because the runtime path (register handler) re-reads settings per request. Kind 2 uses `pytest.mark.skipif` because the routing decision is fixed at module-import time and can't be unwound.

2. **Kind 1 (DONE):** both tests fixed per original scope, commit `5f2b355`.

3. **Kind 2 — module-level `pytestmark` on `test_billing_topup.py`:**

   ```python
   # near the top of test_billing_topup.py, after imports
   pytestmark = pytest.mark.skipif(
       not settings.BILLING_ENABLED,
       reason="test_billing_topup requires BILLING_ENABLED=true — feature-gated at app-init time",
   )
   ```

   Applies the skipif to every test in the file. Clean one-line file-wide declaration. Preferred over per-test decorators because:
   - Every test in `test_billing_topup.py` exercises billing-specific code; no mixed file.
   - Future tests added to the file inherit the gate automatically.
   - Reader sees the gate once at the top, not 19 times scattered.

4. **No session-wide conftest override.** See §Context for why that pattern was rejected (lies about env state, requires global Stripe mock, larger blast radius).

5. **`@pytest.mark.skipif` per-test stays an option for mixed files** — future test files that combine billing + non-billing tests would use per-test decorators. The discipline is "whatever gate scope matches the test's dependence on the feature flag."

6. **No chassis code changes.** Only:
   - `backend/tests/test_billing_foundation.py` (Kind 1 — DONE)
   - `backend/tests/test_billing_topup.py` (Kind 2 — new)
   - `doc/issues/0018-chassis-hardening-audit.md` (discipline note — expand from Kind 1 only to cover both)

7. **No scope creep beyond the two test files and the 0018 doc.** Don't refactor fixtures, don't add conftest helpers, don't touch any non-billing test.

## Out of scope

- Any non-test code changes.
- Test infrastructure refactoring beyond the two test files.
- New tests — we're adding skipif to existing tests, not writing new ones.
- Scanning the broader suite for other env-dependent tests beyond the two billing test files — that's 0018 territory.

**2. Fix `test_webhook_not_mounted_when_billing_disabled` (line 522):**

The current test uses the shared `client` fixture, which is bound to an app whose routes are already fixed. That doesn't work for testing mounting — we need a fresh app instance.

Replace the test body with a fresh-app construction that forces `BILLING_ENABLED=false` at app-init time. The cleanest approach is to assert directly against the app's routes list, avoiding a live HTTP request:

```python
async def test_webhook_not_mounted_when_billing_disabled():
    """BILLING_ENABLED=false — billing router is not mounted on the FastAPI app."""
    import importlib
    from app import main as main_module

    with patch.object(settings, "BILLING_ENABLED", False):
        # Re-import the main module to trigger the conditional router mount with the
        # patched value. importlib.reload rebuilds the app + its route list against
        # the current settings.
        reloaded = importlib.reload(main_module)
        mounted_paths = {route.path for route in reloaded.app.routes}

    assert not any(path.startswith("/billing") for path in mounted_paths), (
        f"Billing routes should not be mounted when BILLING_ENABLED=false; "
        f"found paths starting with /billing in: {sorted(mounted_paths)}"
    )
```

**Caveat on `importlib.reload`:** this rebuilds the app module and may have side effects on other tests (e.g., the shared `client` fixture becomes stale). If this proves brittle, fall back to an alternative: construct an isolated FastAPI app using a helper similar to `_make_billing_test_app()` but with billing disabled, and assert the billing router isn't in that fresh app's routes. The backend-builder should pick whichever approach stays inside this file and doesn't break the other 107 tests.

**If `importlib.reload` breaks fixture state:** abandon it and build a local test app factory for this one test:

```python
def _make_app_with_billing_disabled():
    """Construct a FastAPI app with BILLING_ENABLED=false to verify route mounting."""
    from fastapi import FastAPI
    app = FastAPI()
    # Mirror the conditional mount logic from main.py.
    if settings.BILLING_ENABLED:  # patched to False in the calling test
        from app.routes.billing import router as billing_router
        app.include_router(billing_router)
    return app

async def test_webhook_not_mounted_when_billing_disabled():
    with patch.object(settings, "BILLING_ENABLED", False):
        app = _make_app_with_billing_disabled()
        mounted_paths = {route.path for route in app.routes}
    assert not any(path.startswith("/billing") for path in mounted_paths)
```

The second approach is safer (no module-level reload) but duplicates a small slice of `main.py`'s mount logic. Either fix is acceptable; the backend-builder picks based on which works cleanly against the existing fixture suite.

## Chassis implications

Test isolation from `.env` state is now a documented two-pattern discipline, not ad-hoc:

1. **Runtime-path tests:** explicit `patch.object(settings, ...)` inside the test.
2. **Feature-gated tests:** `pytest.mark.skipif` at module or test level.

Future tests that depend on any `Settings` field must pick the right pattern. Future test files for feature-gated subsystems (e.g., a future `test_billing_subscribe.py` for ticket 0024) should start with a module-level `pytestmark` skipif. 0018 chassis-hardening audit sweeps the existing suite to catch any pre-existing violators.

Not a startup invariant → no chassis-contract.md entry.
