## Context

Two tests in `backend/tests/test_billing_foundation.py` (from 0021) don't isolate themselves from the process's `.env` state:

1. **`test_register_does_not_create_customer_when_billing_disabled`** (line 301) — asserts that with `BILLING_ENABLED=false`, register doesn't call Stripe and leaves `user.stripe_customer_id = None`. The test does NOT patch `settings.BILLING_ENABLED`; it relies on the chassis default being `false`. When `.env` overrides to `true`, register's billing hook fires; the test's mocked `stripe.Customer.create()` returns a `MagicMock`, whose `.id` attribute is itself a `MagicMock`; that `MagicMock` gets assigned to `user.stripe_customer_id` (a `Mapped[str]` column); SQLAlchemy's commit path misbehaves downstream, surfacing as `MissingGreenlet`. The "pre-existing failure" is a cascade from the missing explicit patch.

2. **`test_webhook_not_mounted_when_billing_disabled`** (line 522) — asserts that `POST /billing/webhook` returns 404 when billing is disabled. The test uses the shared `client` fixture, which is bound to the running app; the app's routes were mounted at startup based on `settings.BILLING_ENABLED` at that moment. No mid-test patch can un-mount a route that's already in the router. When `.env` has `BILLING_ENABLED=true`, the webhook IS mounted, so `POST /billing/webhook` returns 400 (bad signature) instead of 404.

Both failures are **test-infrastructure bugs, not chassis bugs**. The chassis code behaves correctly in both scenarios. The tests themselves don't isolate from process env state the way the other tests in the same file do.

### The pattern is already established in the file

`test_register_creates_customer_when_billing_enabled` (line 315) uses `patch.object(settings, "BILLING_ENABLED", True)`. Same file, same author, established convention. These two tests just missed the pattern.

`test_webhook_rejects_invalid_signature` (line 533) uses `_make_billing_test_app()` — a helper in the file that constructs a fresh FastAPI app with billing enabled for isolated webhook testing. Same pattern applies for testing the DISABLED case.

## Design decisions

1. **Surgical fix, match existing patterns.** Both fixes use patterns already in the file. No new test infrastructure, no new helpers.

2. **Test 1:** wrap the existing body in `with patch.object(settings, "BILLING_ENABLED", False):`. This explicitly sets the value regardless of `.env`, so the test behaves identically whether `.env` has `true` or `false`.

3. **Test 2:** refactor to use `_make_billing_test_app()` pattern but with `BILLING_ENABLED=false`. Build a fresh app, verify the billing router was not mounted (either by checking the app's routes list directly, or by POSTing to the fresh app and getting 404).

4. **No chassis code changes.** Only `backend/tests/test_billing_foundation.py`.

5. **No scope creep.** Don't refactor other tests, don't add helpers, don't touch fixtures. Just these two tests.

6. **Chassis-hygiene lesson captured in 0018.** Add to `doc/issues/0018-chassis-hardening-audit.md` candidates: "Test suite must be fully isolated from dev `.env` state — every test that depends on a Settings field must explicitly `patch.object(settings, ...)` and never rely on `.env` defaults." The 0018 audit pass should scan for any remaining tests that fail this discipline.

## Out of scope

- Any non-test code changes.
- Test infrastructure refactoring beyond these two tests.
- New tests (the tests being fixed already cover the scenarios; we're making them robust, not adding new coverage).
- Scanning the broader suite for other env-dependent tests — that's 0018 territory.

**1. Fix `test_register_does_not_create_customer_when_billing_disabled` (line 301):**

Wrap the existing test body in `patch.object(settings, "BILLING_ENABLED", False)`. Requires `from app.config import settings` (may already be imported for other tests — verify). The fixed shape:

```python
async def test_register_does_not_create_customer_when_billing_disabled(client):
    """Default config (BILLING_ENABLED=false) — no Stripe calls, stripe_customer_id stays NULL."""
    with (
        patch.object(settings, "BILLING_ENABLED", False),
        patch("app.billing.primitives.stripe") as mock_stripe,
    ):
        resp = await client.post("/auth/register", json=_REGISTER_BODY)
    assert resp.status_code == 200
    mock_stripe.Customer.create.assert_not_called()

    async with AsyncSession(engine, expire_on_commit=False) as session:
        result = await session.execute(select(User).where(User.email == _REGISTER_BODY["email"]))
        user = result.scalar_one_or_none()
    assert user is not None
    assert user.stripe_customer_id is None
```

The change is the `patch.object(settings, "BILLING_ENABLED", False)` line added inside the `with`. Everything else stays identical.

**2. Fix `test_webhook_not_mounted_when_billing_disabled` (line 522):**

The current test uses the shared `client` fixture, which is bound to an app whose routes are already fixed. That doesn't work for testing mounting — we need a fresh app instance.

Replace the test body with a fresh-app construction that forces `BILLING_ENABLED=false` at app-init time. The cleanest approach is to assert directly against the app's routes list, avoiding a live HTTP request:

```python
async def test_webhook_not_mounted_when_billing_disabled():
    """BILLING_ENABLED=false — billing router is not mounted on the FastAPI app."""
    import importlib
    from app import main as main_module

    with patch.object(settings, "BILLING_ENABLED", False):
        # Re-import the main module to trigger the conditional router mount with the
        # patched value. importlib.reload rebuilds the app + its route list against
        # the current settings.
        reloaded = importlib.reload(main_module)
        mounted_paths = {route.path for route in reloaded.app.routes}

    assert not any(path.startswith("/billing") for path in mounted_paths), (
        f"Billing routes should not be mounted when BILLING_ENABLED=false; "
        f"found paths starting with /billing in: {sorted(mounted_paths)}"
    )
```

**Caveat on `importlib.reload`:** this rebuilds the app module and may have side effects on other tests (e.g., the shared `client` fixture becomes stale). If this proves brittle, fall back to an alternative: construct an isolated FastAPI app using a helper similar to `_make_billing_test_app()` but with billing disabled, and assert the billing router isn't in that fresh app's routes. The backend-builder should pick whichever approach stays inside this file and doesn't break the other 107 tests.

**If `importlib.reload` breaks fixture state:** abandon it and build a local test app factory for this one test:

```python
def _make_app_with_billing_disabled():
    """Construct a FastAPI app with BILLING_ENABLED=false to verify route mounting."""
    from fastapi import FastAPI
    app = FastAPI()
    # Mirror the conditional mount logic from main.py.
    if settings.BILLING_ENABLED:  # patched to False in the calling test
        from app.routes.billing import router as billing_router
        app.include_router(billing_router)
    return app

async def test_webhook_not_mounted_when_billing_disabled():
    with patch.object(settings, "BILLING_ENABLED", False):
        app = _make_app_with_billing_disabled()
        mounted_paths = {route.path for route in app.routes}
    assert not any(path.startswith("/billing") for path in mounted_paths)
```

The second approach is safer (no module-level reload) but duplicates a small slice of `main.py`'s mount logic. Either fix is acceptable; the backend-builder picks based on which works cleanly against the existing fixture suite.
