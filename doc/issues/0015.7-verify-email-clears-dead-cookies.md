## Context

0015 spec (§Design decisions and §Implications) calls for verification to invalidate the user's session so they must log in again: *"success → markLoggedOut() (token_version bumped, access token invalidated) + user clicks the inline 'Log in' button to go to /login."*

The implementation went halfway:

- **Server-side** (`backend/app/routes/auth.py:492-494`): verify-email sets `verified_at`, bumps `user.token_version`, and calls `revoke_all_user_tokens`. Correct.
- **Client-side context** (`frontend/context/auth.tsx` + `frontend/lib/api.ts`): `markLoggedOut()` clears the `HAS_SESSION_KEY` sessionStorage flag. Correct.
- **What's missing**: the backend response to `/auth/verify-email` does not carry `Set-Cookie` headers that delete the now-dead `access_token` and `refresh_token` cookies. The frontend cannot clear HttpOnly cookies from JavaScript — only the server can.

Result: after verification, the browser still holds `access_token` and `refresh_token` cookies. They are invalid server-side (tv mismatch, refresh revoked) but present client-side. When the user clicks the "Log in" button (`<Link href="/login">` in `frontend/app/(auth)/verify-email/VerifyEmailBody.tsx:122`), the Next.js proxy at `frontend/proxy.ts:18-25` reads `request.cookies.has("access_token") === true` and 307-redirects to `/app`. The user never sees `/login`.

On `/app`, `useAuth()` fires `GET /auth/me`, the backend rejects the stale token (tv mismatch checked in `backend/app/dependencies.py:74`), the 401 interceptor in `lib/api.ts` correctly declines to refresh (`HAS_SESSION_KEY` is already cleared), and `user` resolves to `undefined`. The `/app` stub (`frontend/app/(app)/app/page.tsx:13`) renders `{user?.email}` as an empty span, so the page reads "You're logged in as ." — the user appears to be on the app shell despite having no live session. Depending on timing (React Query `staleTime: 30_000` may return briefly-stale cached user data from the pre-verify register call), the user may see their actual email for a frame before it clears. Either way, the intended UX — "land on /login, re-enter credentials" — is bypassed.

### Scope of the bug

Only the verify-email endpoint has this gap. Audit of all `token_version += 1` sites:

- `backend/app/routes/auth.py:381-388` (change-password) — bumps tv, revokes refresh tokens, then **re-issues fresh cookies** via `_set_auth_cookies`. Equivalent UX (user stays logged in under a fresh session). Correct.
- `backend/app/routes/auth.py:462-463` (reset-password) — bumps tv, revokes refresh tokens. This flow is pre-auth (user submits a reset token, not a session cookie). Nothing to clear. Correct.
- `backend/app/routes/auth.py:492-494` (verify-email) — bumps tv, revokes refresh tokens, returns a bare dict with no cookie handling. **This ticket's fix.**

### Why `decode_verify_token` doesn't catch stale clicks

`_decode_purpose_token` in `backend/app/services/auth_service.py:64-82` validates signature, expiry, audience, issuer, purpose, and the presence of `sub` and `tv`. It does **not** compare the token's `tv` claim against the current `user.token_version`. The `verify_email` endpoint's own guard is `if user.verified_at is not None: return {"message": "Email already verified."}` — so a stale token (tv=0, user.tv=1) after first verify is routed to the idempotent "already verified" response, not a 401. This is intentional and acceptable.

### Why clearing cookies on the idempotent ("already verified") branch is unsafe

Because the endpoint doesn't verify `token.tv == user.token_version`, the "already verified" branch can be reached by anyone holding any non-expired verify token for that user. Clearing cookies on that response would clobber whichever browser happens to be clicking the link — possibly an unrelated session, or one the user shares with someone else. Cookies must be cleared only on the newly-verified branch (where we actually mutated state).

## Design decisions (pre-committed)

- **Clear cookies only on the newly-verified branch.** The idempotent ("already verified") branch returns a bare dict as today. Matches the cookie behavior of other endpoints: logout clears, refresh re-sets, endpoints with no session-mutating work don't touch cookies.
- **Use the existing `_clear_auth_cookies` helper**, called after `revoke_all_user_tokens(...)`. Helper already passes `domain=settings.COOKIE_DOMAIN` + `secure=settings.COOKIE_SECURE`, so staging (with COOKIE_DOMAIN=.staging.carddroper.com) correctly clears the parent-domain-scoped cookies set by register.
- **Response shape change on the newly-verified branch:** construct `response = JSONResponse(content={"message": "Email verified."})`, call `_clear_auth_cookies(response)`, return `response`. Mirrors the logout endpoint pattern (`backend/app/routes/auth.py:315-318`). FastAPI honors the JSONResponse exactly — no implicit re-wrap.
- **Not a chassis-contract invariant.** This is a behavioral rule, not a startup check; it cannot be validated at `Settings()` construction. Note the broader pattern ("endpoints that bump token_version on an authenticated session MUST clear auth cookies") in the `0018` chassis-hardening-audit ticket so future token-invalidating endpoints (e.g. change-email in ticket 0017) are checked against it.
- **No frontend change.** `VerifyEmailBody` and `proxy.ts` are correct on the assumption that the server clears cookies. Keep them untouched.
- **No smoke-script change.** `smoke_verify_email.py` cannot mint a verify token (no public endpoint does so, and adding one would be a security hole — explicitly out of scope per 0015 Phase 2 design). Coverage for the fix is a pytest test against the FastAPI TestClient, where minting a verify token via the internal helper is legitimate.

   ```python
   if user.verified_at is not None:
       return {"message": "Email already verified."}

   user.verified_at = datetime.now(timezone.utc).replace(tzinfo=None)
   # Bump token_version to invalidate any session minted before verification.
   user.token_version += 1
   ```
