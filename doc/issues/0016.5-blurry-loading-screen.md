## Context

0016.4 resolved the functional issue — authed users are now redirected away from `/login` automatically after silent refresh re-authenticates them. But the *visual* experience still flashes the login form during the silent-refresh window, which feels off.

Similar flashes exist in several other places in the chassis:

1. `(app)/layout.tsx` — during `useAuth`'s initial `/auth/me` fetch (`isLoading=true`), the layout renders `<AppHeader>` + child stub with `user=null`, resulting in the "You're logged in as " empty-email state until the query resolves.
2. `LoginBody.tsx` — during `useAuth`'s initial fetch, renders the login form. If silent refresh succeeds, the 0016.4 effect fires and redirects, but the form flashed.
3. `/verify-email/page.tsx`, `/reset-password/page.tsx`, `/login/page.tsx` — Suspense fallbacks are inline text like "Loading…" or a minimal spinner. Inconsistent with the rest of the chassis's visual language.

The user chose a unified approach: all pre-decision page loading uses a Canva-style blurry backdrop. In-operation loading (form submit spinners, per-operation mutation states) stays inline.

## Design decisions (pre-committed)

- **Single chassis component `<LoadingScreen />`** at `frontend/components/loading/LoadingScreen.tsx`. Reusable across auth pages, authed pages, Suspense fallbacks, and any future chassis or body component that needs pre-decision loading.
- **Visual shape (final, refined 2026-04-23 after local review):** empty full-viewport fixed overlay (`fixed inset-0 z-50`) with translucent white backdrop (`bg-white/70`) + `backdrop-blur-md`. **No spinner, no text, no brand mark.** The blurred backdrop itself is the "loading" signal. Two reasons for the pure-blur shape:
  1. Any inner element would overlap visually with the inline operation spinners that render just underneath (`VerifyEmailBody` pending, `ResetPasswordBody` validating, `SubmitButton` pending) — two spinners stacked reads as "two things loading" which is inaccurate.
  2. The backdrop's crisp→blurred + 30%-whitened transition is itself a strong state signal; industry pattern (Canva, Notion, Linear) is to trust it alone for warm transitions <500ms.
- **Accessible:** `role="status"` + `aria-label="Loading"` (empty element needs programmatic label, not `aria-live` which applies to text content).
- **Server-component by default:** no hooks, no browser APIs, no `"use client"` directive. Importable from server or client layouts.
- **Tailwind-only**, no new CSS file, no new dependencies.
- **Pre-decision scope only:**
  - `(app)/layout.tsx` when `isLoading` (also during the brief `!isAuthenticated` window before 0016.2 effect navigates).
  - `LoginBody.tsx` when `isLoading` (also during `isAuthenticated`-but-not-yet-redirected — 0016.4's effect transition window).
  - Suspense fallbacks in `/login/page.tsx`, `/reset-password/page.tsx`, `/verify-email/page.tsx`.
- **In-operation loading stays inline:**
  - `VerifyEmailBody` spinner during the verify mutation — keep.
  - `ResetPasswordBody` spinner during validate-reset-token — keep.
  - `SubmitButton` pending state — keep.
  - `ForgotPasswordBody` submit → inline success panel — keep.
  - Rationale: operation-scoped spinners tell the user "we're acting on what you just did" with surrounding UI still relevant. Full-screen overlay would be jarring.
- **No banner / toast replacement.** We're not building a notification system; just loading shells.
- **No chassis-contract entry.** Visual pattern, not a startup invariant.
- **Mobile note:** the `LoadingScreen` name + role is a chassis convention. When mobile apps land (PLAN.md §10 item 8, React Native assumed), a corresponding RN `LoadingScreen` should implement the same visual concept natively (`BlurView` on iOS, `blurRadius` on Android image layers). This ticket is web-only but the convention stays.
- **Z-index:** `z-50` sits above page content but below any future modals we might add. Safe for all current chassis content.

## Acceptance

### Phase 0 — frontend (frontend-builder, one dispatch)

1. **Create `frontend/components/loading/LoadingScreen.tsx`** (final form, post local-review refinements on 2026-04-23):

   ```tsx
   /**
    * Chassis loading screen — full-viewport translucent blur.
    * See §Design decisions for rationale.
    */
   export function LoadingScreen() {
     return (
       <div
         className="fixed inset-0 z-50 bg-white/70 backdrop-blur-md"
         role="status"
         aria-label="Loading"
       />
     );
   }
   ```

2. **`frontend/app/(app)/layout.tsx`** — add early-return for pre-decision loading:

   ```tsx
   // ... existing imports + useEffect ...
   import { LoadingScreen } from "@/components/loading/LoadingScreen";

   export default function AppLayout({ children }: { children: ReactNode }) {
     const router = useRouter();
     const { isLoading, isAuthenticated } = useAuth();

     useEffect(() => {
       if (!isLoading && !isAuthenticated) router.replace("/login");
     }, [isLoading, isAuthenticated, router]);

     // Pre-decision: show blurry screen until auth resolves or redirect fires.
     if (isLoading || !isAuthenticated) return <LoadingScreen />;

     return (
       <>
         <AppHeader />
         <main>{children}</main>
       </>
     );
   }
   ```

   The `!isAuthenticated` branch covers the brief window between the query resolving as 401 and the effect firing `router.replace("/login")`. During that window we show LoadingScreen instead of the empty-email app stub.

3. **`frontend/app/(auth)/login/LoginBody.tsx`** — add early-return for the same reason (pre-decision during `useAuth` initial fetch + the 0016.4 redirect-firing window):

   ```tsx
   // ... existing imports ...
   import { LoadingScreen } from "@/components/loading/LoadingScreen";

   export function LoginBody() {
     // ... existing hooks including the 0016.4 useEffect ...

     if (isLoading || isAuthenticated) return <LoadingScreen />;

     // ... existing JSX (banner + form) ...
   }
   ```

   The `isAuthenticated` branch handles the moment between silent-refresh success and 0016.4's `router.replace("/app")` — form is hidden, blurry screen shown instead.

4. **`frontend/app/(auth)/login/page.tsx`** — Suspense fallback becomes `<LoadingScreen />`:

   ```tsx
   <Suspense fallback={<LoadingScreen />}>
     <LoginBody />
   </Suspense>
   ```

5. **`frontend/app/(auth)/verify-email/page.tsx`** — Suspense fallback becomes `<LoadingScreen />`. (The `VerifyEmailBody`'s own `mutation.isPending` inline spinner stays as-is; that's in-operation.)

6. **`frontend/app/(auth)/reset-password/page.tsx`** — Suspense fallback becomes `<LoadingScreen />`. (The `ResetPasswordBody`'s validate-reset-token spinner stays inline.)

7. **Do NOT change** `VerifyEmailBody` or `ResetPasswordBody` internal pending/spinner logic — those are in-operation loading and keep their current inline treatment per §Design decisions.

8. **Do NOT change** `ForgotPasswordBody`, `RegisterPage`, `SubmitButton`, any other existing spinner or pending UI that isn't explicitly listed above.

## Chassis note

`<LoadingScreen />` is **chassis**. Part of the reusable visual shell. Future projects built on this chassis inherit it by virtue of having `components/loading/LoadingScreen.tsx`. When `STARTER.md` lands (post-0016/0017 per site-model.md reusability constraint 7), the "pre-decision vs in-operation" distinction should be documented there so adopters know when to reach for `LoadingScreen` vs an inline spinner.

## Resolution

Landed on dev (2026-04-23, commit `3049fe7`); local validation passed. Final `LoadingScreen` shape after three iterative refinements with the user: `<div className="fixed inset-0 z-50 bg-white/70 backdrop-blur-md" role="status" aria-label="Loading" />` — spinnerless, textless, pure blur. Used in `(app)/layout` (pre-decision auth gate), `LoginBody` (during silent-refresh re-auth window), and the three auth-page Suspense fallbacks (`/login`, `/verify-email`, `/reset-password`). In-operation spinners (form submit, reset-token validate, verify-email mutation) explicitly retained inline per the pre-decision-vs-in-operation distinction documented in §Design decisions. Merged dev → main (2026-04-23). Cloud Build redeploy + Phase 2 staging sanity check next.
