## Context

This is the cross-device victim-device UX. It's an inherent limitation of cookie-based authentication — the server cannot remote-clear cookies on a browser that wasn't party to the reset request. Same-device reset (the Scenario 1 that 0016 Phase 0 hardens) works cleanly: cookies are cleared server-side in the response to the reset POST, the browser honors the Set-Cookie Max-Age=0 headers, and the user lands on /login with no residual state.

Cross-device is different. Device A has live-looking cookies that are dead server-side. The frontend needs to detect and handle this state itself.

### Current behavior trace on Device A (verified in browser)

1. User on /app, cookies present, `/auth/me` previously succeeded, useAuth.user populated.
2. On Device B, user completes password reset. Backend bumps `user.token_version` and calls `revoke_all_user_tokens`.
3. On Device A, user reloads /app (or any authenticated call fires via React Query's 30s staleTime).
4. `GET /auth/me` → 401 (token_version mismatch checked in `backend/app/dependencies.py:74`).
5. 401 interceptor in `frontend/lib/api.ts:147-194`: checks `isRefreshExempt(path)` (no, /auth/me is not exempt) and `hasSession()` (yes, HAS_SESSION_KEY was set at login and never cleared). Fires `POST /auth/refresh`.
6. `/auth/refresh` → 401 ("No refresh token" because the cookie was never cleared on A but the underlying refresh_token row in the DB was revoked, OR "Invalid or expired refresh token" — either way, 401).
7. Interceptor's refresh-fail branch (line 192) calls `markLoggedOut()` — clears HAS_SESSION_KEY. Throws the original 401.
8. `useQuery<User>` at `frontend/context/auth.tsx:53` receives the error. `data` stays undefined → `user = null` (default from `data: user = null`).
9. useAuth now returns `{ user: null, isLoading: false, isAuthenticated: false, ... }`.
10. `/app` page (`frontend/app/(app)/app/page.tsx:13`) renders `{user?.email}` → empty span. "You're logged in as [blank]."
11. No code path redirects anywhere. User is stuck in the ghost UI until they manually act (click Logout, type a new URL, or wait 15 minutes for the access_token to time out and be auto-deleted by the browser, at which point the proxy will start redirecting `/app` → `/login` naturally).

### Why this isn't a backend problem

- The backend's response to `/auth/reset-password` DOES correctly include Set-Cookie clear headers (verified by 0016's `test_reset_password_clears_cookies` unit test + Scenario 1 in-browser verification). Those headers only reach the browser that made the request. Device A is not in that conversation.
- No server push mechanism exists today (we don't run WebSockets or SSE). Even if we did, adding real-time session-revocation is heavyweight and over-engineered for v0.1.0.
- The cookie's `Max-Age` eventually handles the cleanup lazily (15 min for access, 7 days for refresh). That's the final backstop.

### Industry pattern choice

Scan of how other Pattern-A apps (ones that DO invalidate sessions on reset) handle the victim device:

- **GitHub, Stripe Dashboard, Gmail:** silent auto-redirect to /login on 401-after-refresh-fail. No banner, no modal.
- **Linear, Google (modern):** same + real-time push via WebSocket/SSE so the redirect happens within seconds of the reset, not on next action.
- **Some consumer apps:** "Session expired" modal before redirect.

For Carddroper's stage: **silent auto-redirect** matches GitHub/Stripe posture and is the minimum viable fix. Session-expired messaging and real-time push are deferred.

## Design decisions (pre-committed)

- **Auto-redirect via `useEffect` in `(app)/layout.tsx`.** The layout already wraps every authed route; adding the effect there covers the /app stub today AND every future authed page that lands under `(app)/` (card browser, send flow, account settings, etc. — none exist yet but the chassis will inherit the behavior for free).
- **Trigger: `!isLoading && !isAuthenticated`.** Both flags from `useAuth()`. The `!isLoading` guard prevents the effect from firing during the initial auth query (when `isLoading=true, user=null` briefly). After the query resolves (whether with a valid user or with an error), the effect evaluates and acts.
- **`router.replace`, not `router.push`.** The broken /app URL shouldn't enter browser history. Hitting Back from /login should take the user wherever they came from (e.g., marketing /), not loop back into the ghost state.
- **Redirect destination: `/login`.** Not `/forgot-password` or `/`. The user's intent was to be on /app; /login is the cleanest return path — they can re-authenticate and land back on /app. If they want to go elsewhere, the /login page has standard links.
- **No banner, no toast, no "session expired" message.** Matches the zero-toast posture we committed to for all of 0015. If user feedback post-launch reveals confusion ("why did I just get bumped to login?"), a banner can be added trivially — `/login?expired=true` + a green-banner style reuse from the reset-success banner.
- **No backend change. No proxy change.** Frontend-only.
- **No new chassis-contract entry.** Behavioral UX rule, not a startup invariant. Chassis-adopters inherit the `(app)/layout.tsx` pattern as part of the chassis shape.
- **No new dependencies. No new helpers.** `useEffect` + `useRouter` + `useAuth` are all already available.

## What this fix does NOT cover

- **Real-time revocation.** If Device B resets and Device A is actively on /app, A only notices on the next `/auth/me` refetch (up to 30s via `staleTime`) or user interaction that triggers an API call. For true instant kick-off, we'd need WebSockets/SSE. Deferred — not proportionate to the UX gap.
- **"Log out all other sessions" settings page.** User-initiated kill of all devices. Future feature; not needed for this fix. Tracked as a candidate for a later account-settings ticket, not 0016.2.
- **Banner messaging on the redirect landing.** Silent redirect is deliberate per the design decisions above. A future polish ticket can add `/login?expired=true` detection + banner copy if users ask.
- **Non-/app authed routes.** There are none today. The layout-scoped fix applies to anything under `(app)/` automatically.

## Acceptance

### Phase 0 — frontend (frontend-builder)

1. **`frontend/app/(app)/layout.tsx`** — add the auto-redirect effect. Imports: `useEffect` from react, `useRouter` from `next/navigation`. Call `useAuth()` at the `AppLayout` component level (currently it's only called inside the nested `AppHeader`; that stays; the new call is additional — React context returns the same value, no duplicate network requests).

   Exact target shape for `AppLayout`:

   ```tsx
   export default function AppLayout({ children }: { children: ReactNode }) {
     const router = useRouter();
     const { isLoading, isAuthenticated } = useAuth();

     useEffect(() => {
       if (!isLoading && !isAuthenticated) {
         router.replace("/login");
       }
     }, [isLoading, isAuthenticated, router]);

     return (
       <>
         <AppHeader />
         <main>{children}</main>
       </>
     );
   }
   ```

   `AppHeader` and everything else in the file stays unchanged. Don't refactor the `AppHeader`'s internal `useAuth` call — multiple calls to `useAuth` from the same provider are fine.

2. **No other files touched.** Not `(marketing)/layout.tsx` (no auth gate there), not `(auth)/layout.tsx` (public pages), not `proxy.ts` (its job is cookie-presence routing — this fix handles the case where the cookie is present but invalid, which the proxy can't see from the edge), not the context or the API client.

## Resolution

Landed on main (2026-04-22, commit `6d9ebbe`). `(app)/layout.tsx` gained a `useEffect` that calls `router.replace("/login")` when `!isLoading && !isAuthenticated`. Symmetric with the proxy's own unauth-redirect rule, covering the client-side transition to unauthed that the edge middleware cannot observe. Post-deploy staging verification then revealed the proxy-loop problem (stale `access_token` cookie on victim device kept the proxy bouncing `/login` → `/app` on RSC fetches) — resolved by 0016.3.
