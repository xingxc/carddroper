## Context

Post-0016.6, the logout flow in `LogoutButton` was:

1. `markLoggedOut()` — clear `HAS_SESSION_KEY` from sessionStorage.
2. `await queryClient.cancelQueries()`.
3. `await api.post("/auth/logout")` — backend clears cookies.
4. `markAuthLoggedOut()` — re-clear sessionStorage (redundant with step 1) + `queryClient.resetQueries(['auth','me'])` (triggers an in-tree refetch).
5. `window.location.href = "/"` — hard page reload.

Step 4's `resetQueries` forced a refetch of `/auth/me` by the still-mounted `(app)/layout`'s `useAuth` subscriber before the browser unmounted the tree in step 5. That refetch returned 401 (cookies just cleared), triggered the 0016.3 cleanup chain, fired another `POST /auth/logout`. Then step 5 reloaded, the fresh `AuthProvider` fired another `/auth/me` 401, triggering another cleanup `POST /auth/logout`.

Net: 3 POSTs + 2 401s per logout click.

## Design decision

Drop the step-4 state reset entirely. The hard reload (step 5) destroys the entire JS context + React Query cache atomically; any explicit client-state clearing is duplicative. Remove `markLoggedOut` from the `AuthState` context API — its only caller was LogoutButton, and exposing it invited adopters to soft-logout (via `router.push` instead of hard reload) and end up with partial state. `markLoggedIn` stays in the context because login is legitimately a soft-navigation flow that needs cache invalidation.

**Asymmetric by design, not oversight:** login is soft-navigation (needs invalidate), logout is hard-reload (doesn't need invalidate). The context API now narrows to the single correct pattern.

## Changes

- `frontend/components/auth/LogoutButton.tsx` — dropped the `useAuth` import and the `markAuthLoggedOut()` call; removed it from the useCallback deps.
- `frontend/context/auth.tsx` — removed `markLoggedOut` from the `AuthState` interface, removed the `handleMarkLoggedOut` useCallback, removed it from the context value, dropped the `markLoggedOut` name from the `lib/api` import list.

Lib-level `markLoggedOut` (sessionStorage-clear function in `lib/api.ts`) stays exported — still called directly from `LogoutButton` step 1 and from the 401 interceptor's refresh-failed branch.

## Verification

Frontend-builder report:
- `npm run lint`, `npx tsc --noEmit`, `npm run build` all clean (no other consumer of `useAuth().markLoggedOut` existed — tsc would have caught one).
- Runtime walkthrough confirmed across four scenarios: happy-path logout, logout-POST-fail, multi-tab logout, logout-during-in-flight-authed-request. No regressions.

## Out of scope

- Chassis-contract entry (behavioral refinement, not a startup invariant).
- `markLoggedOut` removal from `lib/api.ts` (still used internally and by LogoutButton step 1).
- Further logout-flow changes — 0016.8 follows up on the remaining redundancy (anonymous-visitor + post-reload cleanup POSTs) via backend error-code split.

## Chassis implications

- Logout is hard-reload-only in this chassis. Adopters that soft-logout (router.push + client-only state reset) accept responsibility for state-clearing completeness; the context does not expose a primitive for that.
- `useAuth()` API is asymmetric: `markLoggedIn` exposed (soft-nav login flow), `markLoggedOut` not exposed (logout uses hard reload). Asymmetry reflects the two flows' different correctness needs.

## Resolution

Landed on dev (2026-04-23, commit `82681b9`); local docker-log verification confirmed the logout flow now fires 1 intentional `POST /auth/logout` + 1 bootstrap `GET /auth/me 401` (down from 3 POSTs + 2 401s). Two fewer HTTP round-trips per logout.

Immediate follow-up **0016.8** (`52d1c25` + `a9057ac`) further eliminated the remaining bootstrap-cleanup POST by gating `attemptLogoutCleanup` on the backend error code, bringing the logout flow to 1 POST + 1 401 total and also removing the wasted cleanup POST on every anonymous page load.

Merged dev → main (2026-04-23). Cloud Build redeploy + Phase 2 staging sanity check next.
