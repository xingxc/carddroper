## Context

The current auth architecture produces an "expired access token, valid refresh token" window that surfaces as:
- Reload `/app` past access expiry → proxy redirect to `/login` → silent refresh on `/login` re-auths → 0016.4 effect redirects to `/app`. Net: brief blurry flash (0016.5 made it visual-friendly).
- Idle user returns after >15 min → same flash.

For **active users** (anyone interacting with the app), the flash is entirely preventable: schedule a `POST /auth/refresh` call *before* the access token expires. The scenario never materializes. For **inactive users** (backgrounded tab, closed laptop), the 0016.2 + 0016.3 + 0016.4 + 0016.5 chain continues to handle gracefully — proactive refresh is best-effort under OS-level timer throttling.

### How the industry handles TTL communication

Every mature auth system exposes the access-token TTL via the **response body**, not via client-side configuration:

| System | Response shape | Convention |
|---|---|---|
| **OAuth 2.0 (RFC 6749 §5.1)** | `{access_token, token_type, expires_in, refresh_token}` | Standard across every OAuth provider |
| **Auth0 SDK** | Same — OAuth-compliant | `expires_in` seconds |
| **Clerk** | `session.expireAt` | ISO timestamp |
| **Firebase Auth** | `IdTokenResult.expirationTime` | ISO timestamp |
| **Supabase Auth** | `{expires_in, expires_at, ...}` | Both |
| **NextAuth.js** | `session.expires` | ISO timestamp |
| **Supertokens** | JWT claims in response | `exp` |
| **Vercel Dashboard (in-house)** | `X-Access-Expires-In` header | Seconds |
| **Linear (in-house)** | WebSocket session-sync | Real-time TTL |

**No comparable system uses a client-side env var for TTL.** It's a shortcut that breaks when:
1. Backend tunes TTL → frontend env drifts → unnecessary refresh churn until caught.
2. Per-user TTLs arrive (MFA-verified longer, unverified shorter) → single frontend env var can't vary per user.
3. Multiple env coordination across adopters → STARTER.md checklist grows.
4. OAuth integration later → client must handle both conventions.

For chassis-grade quality, the response-body pattern is the single-source-of-truth approach. We adopt OAuth 2.0's `expires_in` field name for future compatibility.

## Design decisions (pre-committed)

### Response shape

**Field name: `expires_in: int`** (seconds, referring to the access token's remaining lifetime). Matches OAuth 2.0 RFC 6749 §5.1 exactly. Adopters familiar with OAuth recognize the shape immediately.

**Where it appears (siblings, not nested under user):**

```json
// POST /auth/register, POST /auth/login — AuthResponse (existing model, gains expires_in)
{
  "access_token": "eyJ...",
  "refresh_token": "...",
  "user": { "id": 1, "email": "...", ... },
  "expires_in": 900
}

// POST /auth/refresh — new RefreshResponse (replaces current MessageResponse usage on this endpoint)
{
  "message": "Token refreshed.",
  "access_token": "eyJ...",
  "expires_in": 900
}

// GET /auth/me — new MeResponse envelope (breaking change from current flat UserResponse)
{
  "user": { "id": 1, "email": "...", ... },
  "expires_in": 900
}
```

**Scope note — `PUT /auth/password` not in this ticket.** The password-change endpoint today returns `MessageResponse` (`{"message": "Password changed."}`), not `AuthResponse`. No frontend UI consumes it yet (no `/account` page). Adding a new `ChangePasswordResponse` is OAuth-completeness polish — right to do, but defer until the /account UI or mobile Bearer flow lands. The scheduler's correctness is unaffected: after a change-password call, a frontend caller will follow the existing `invalidate(['auth','me'])` pattern, and the scheduler picks up fresh `expires_in` from the next `/auth/me` response. Not a chassis regression.

**Breaking change:** `/auth/me` returns `{user, expires_in}` instead of flat User. Frontend's `useAuth` consumer is updated in the same commit. No external API consumers to coordinate — we're not shipped yet.

**Why envelope for `/auth/me`:** session state (TTL) is semantically distinct from user state (id/email/verified_at). Flat-on-user would conflate them. Envelope keeps the mental model clean and supports future session fields (last_activity_at, active_since, etc.) without further refactoring.

**First-cycle `expires_in` source (for chassis reader clarity):** `LoginBody` and `RegisterPage` currently call `queryClient.invalidateQueries(['auth','me'])` right after their respective `api.post` succeeds. That invalidation drives `AuthProvider`'s `useQuery<MeResponse>` to re-fetch `/auth/me`, which returns a fresh `expires_in`. The scheduler's first timer is scheduled against *that* value, not the login/register response body's value. Adding `expires_in` to `AuthResponse` (register/login bodies) is therefore OAuth-completeness — useful for future mobile Bearer flow that reads the body directly without the invalidate step, and visibly OAuth-consistent to any adopter inspecting the wire. The scheduler itself is driven entirely by `/auth/me` + `/auth/refresh` response bodies.

### Scheduler

- **Where:** inside `AuthProvider` (context/auth.tsx). It owns auth lifecycle; scheduler is an auth concern.
- **Timing:** at 80% of `expires_in`. E.g., 900s TTL → refresh at 720s.
  - Rationale: 80% is the Auth0/Clerk/industry default. Leaves 20% buffer for network latency, clock drift, and edge cases.
- **Self-rescheduling:** on success, the timer handler schedules the next one using the refresh response's new `expires_in` value. On failure, no reschedule — next user action falls through to 0016.3's cleanup path.
- **Cancellation:** useEffect cleanup cancels pending timer. A `cancelled` ref-capture guard prevents re-scheduling if cleanup fires while a refresh is in-flight.
- **Dedup:** shares `getRefreshPromise` with the existing 401 interceptor. If a user action 401s mid-proactive-refresh, both paths share the same in-flight fetch — no double-call.

### What proactive refresh does NOT fix

- **Tab backgrounding past ~5 min:** Chrome throttles/suspends inactive-tab timers. Timer may fire late or not at all. When user foregrounds, normal 0016.3 flow catches up. Not a regression.
- **Device sleep/hibernate:** same.
- **Network loss during scheduled refresh:** timer fires, fetch throws, scheduler stops. Next request falls through to 0016.3.
- **Cross-tab coordination:** each tab runs its own scheduler. `getRefreshPromise` is per-tab. Minor inefficiency (multiple tabs → multiple refresh calls) but not a correctness issue.

### Not adopted (considered and rejected)

- **Frontend env var `NEXT_PUBLIC_ACCESS_TOKEN_TTL_SECONDS`:** rejected per §Context. Env coupling, no per-user variation, OAuth divergence.
- **Response header `X-Access-Expires-In`:** elegant for broadcast on every authed response, but more plumbing (middleware to set the header on success; middleware to read it in interceptor). Not worth the trade-off vs body field.
- **ISO timestamp instead of seconds:** avoids clock-drift issue but adds parsing. Seconds is simpler and matches OAuth convention. 80% buffer absorbs small client/server clock drift.
- **Refresh-token rotation:** current design keeps refresh_token stable across refreshes. Rotation is a separate ticket if/when we need it.
- **Real-time session revocation (WebSocket/SSE):** out of scope. Deferred per 0016.2.
- **`NEXT_PUBLIC_*` env vars entirely:** not needed for this ticket; backend drives TTL.

## Acceptance

### Phase 0a — backend (backend-builder)

1. **Response models** (in `backend/app/routes/auth.py`):
   - `AuthResponse` (existing, used by `POST /auth/register` and `POST /auth/login`) gains `expires_in: int`.
   - New `RefreshResponse` model `{message: str, access_token: str, expires_in: int}` — replaces the current `MessageResponse` usage on `POST /auth/refresh`. Other endpoints that currently return `MessageResponse` (`/auth/logout`, `/auth/password`, `/auth/forgot-password`, `/auth/reset-password`, `/auth/verify-email`, `/auth/resend-verification`, `/auth/change-email`, `/auth/confirm-email-change`) stay as-is.
   - New `MeResponse` model `{user: UserResponse, expires_in: int}` — replaces the direct `UserResponse` return from `GET /auth/me`.

2. **Endpoint implementations:**
   - `POST /auth/register`, `POST /auth/login`: compute `expires_in = settings.JWT_EXPIRATION_MINUTES * 60` at the moment of token issuance, include in `AuthResponse`.
   - `POST /auth/refresh`: same — `expires_in` from settings, include in `RefreshResponse` along with the existing `message` and the `access_token` already returned in the body today.
   - `GET /auth/me`: compute from the decoded token's `exp` claim. The `get_current_user` dependency already decodes the token via `_authenticate`. Pass the `exp` claim through `request.state.access_token_exp` inside `_authenticate`, then the `/auth/me` endpoint reads it and computes `expires_in = max(0, int(exp) - int(time.time()))`. Return `MeResponse {user: _user_response(user), expires_in}`.
   - `PUT /auth/password`: **not modified** in this ticket — see §Design decisions scope note. Current `MessageResponse` return stays.

3. **Dependency injection change:**
   - `_authenticate(request)` at `backend/app/dependencies.py:44` — extend to set `request.state.access_token_exp = payload.get("exp")` immediately after successful `jwt.decode`, before returning `(user_id, payload)`. Does not break existing callers (`get_current_user`, `get_current_user_optional`). `require_verified` and `require_not_locked` inherit via their `Depends(get_current_user)` chain.

4. **Tests** in `backend/tests/test_auth_flow.py` and `backend/tests/test_jwt_claims.py`:
   - **New assertions:** `expires_in` present on register, login, refresh, `/auth/me` responses; value is a positive int; value ≈ `JWT_EXPIRATION_MINUTES * 60` on fresh-issued endpoints (register, login, refresh); `/auth/me` value decreases over time (compare two sequential calls — second ≤ first, within a small tolerance).
   - **Existing assertions to update** for the `/auth/me` envelope change — every test that reads `me.json()["email"]` or similar flat fields needs `me.json()["user"]["email"]`:
     - `test_auth_flow.py::test_register_login_me` (line 28: `me.json()["email"]`)
     - `test_auth_flow.py::test_verify_email_preserves_session` (line 398: `me.json()["verified_at"]`)
     - `test_auth_flow.py::test_locked_user_can_still_access_me` (line 355: `me.json()["email"]`)
     - `test_jwt_claims.py::test_access_token_accepted_by_me` (line 67: `r.json()["email"]`)
   - **Unaffected tests** (parse `body["user"]["..."]` already, via register response — not `/auth/me`): `test_verify_email_flow`, `test_password_reset_flow`, `test_reset_password_clears_cookies`, `test_email_change_flow`, `test_register_succeeds_when_email_send_raises`, `test_locked_user_blocked_from_change_password`. Leave untouched.

### Phase 0b — frontend (frontend-builder)

6. **`frontend/lib/api.ts`:**
   - Update `attemptRefresh` return type to `{ ok: boolean; expiresIn?: number }` (parse the `expires_in` field from the refresh response on success).
   - Update `logoutCleanupPromise` / `getRefreshPromise` to carry the richer result type through the dedup.
   - Update the 401 interceptor's `if (refreshed)` branch to match the new shape (`if (result.ok)`).

7. **`frontend/context/auth.tsx`:**
   - Add a TypeScript type `MeResponse { user: User; expires_in: number }` (mirrors the backend envelope).
   - Update `useQuery<User | null>` at line 53 to `useQuery<MeResponse | null>`; `queryFn: () => api.get<MeResponse>("/auth/me")`.
   - Derive `user` and `expiresIn` from the query data:
     ```tsx
     const { data, isLoading } = useQuery<MeResponse | null>({ ... });
     const user = data?.user ?? null;
     const expiresIn = data?.expires_in;
     ```
   - `AuthProvider` context value gains `expiresIn?: number`. Exposed so the scheduler useEffect can depend on it; other consumers typically ignore it.
   - Update the `AuthState` interface (line 23) to include `expiresIn?: number`.
   - New `useEffect` inside `AuthProvider`. Self-reschedules within the timer closure using the `expires_in` from the refresh response — no `invalidateQueries` call needed, since (a) the scheduler doesn't depend on React Query cache freshness for its own correctness, and (b) no other component reads `expiresIn` from `useAuth` today. If a future consumer needs fresh session state after a proactive refresh, add the invalidate then.
     ```tsx
     useEffect(() => {
       if (!isAuthenticated || !expiresIn) return;

       let cancelled = false;
       let timer: ReturnType<typeof setTimeout> | undefined;

       const scheduleRefresh = (delayMs: number) => {
         timer = setTimeout(async () => {
           if (cancelled) return;
           const result = await getRefreshPromise(); // shared dedup with 401 interceptor
           if (cancelled) return;
           if (result.ok && result.expiresIn) {
             scheduleRefresh(result.expiresIn * 1000 * REFRESH_BEFORE_EXPIRY_RATIO);
           }
           // On !result.ok: intentionally do not reschedule, do not markLoggedOut.
           // Next user action 401s → existing 0016.3 chain handles cleanup. This
           // avoids a false-logout on transient network errors (attemptRefresh
           // returns ok=false for both auth-fail and network-fail today).
         }, delayMs);
       };

       scheduleRefresh(expiresIn * 1000 * REFRESH_BEFORE_EXPIRY_RATIO);

       return () => {
         cancelled = true;
         if (timer) clearTimeout(timer);
       };
     }, [isAuthenticated, expiresIn]);
     ```
   - Export a module-level constant `const REFRESH_BEFORE_EXPIRY_RATIO = 0.8;` at the top of `context/auth.tsx`, with a comment pointing to the §Design decisions rationale. (Not env-configured; chassis default.)

8. **Consumers — verify untouched:**

## Out of scope

- Tab-backgrounding perfection (OS-level constraint).
- WebSocket/SSE real-time session revocation.
- Refresh-token rotation.
- Proactive refresh for mobile clients (no mobile client exists yet; when RN arrives, mirror the pattern).
- OAuth 2.0 `token_type` field (`"bearer"`) — we don't currently need it, can add later if we expose auth publicly.
- Per-user / per-tier TTLs — future feature. The response-body pattern doesn't preclude it; future tickets can vary `expires_in` per-user without frontend changes.
- Chassis-contract entry — behavioral/API-shape rule, not a startup invariant. Documented in auth.md instead.

## Chassis implications

This ticket is a **chassis API-shape change**. Future projects built on this chassis inherit:

- OAuth-2.0-compliant `expires_in` convention on auth responses.
- Envelope shape on `/auth/me` (`{user, expires_in}`) — future session fields go here.
- Client-side proactive refresh at 80% of TTL.
- Zero TTL-coordination burden on adopters (set backend TTL, frontend auto-adapts).

When `STARTER.md` lands (per site-model.md Reusability constraint 7), this pattern should be documented as a chassis convention.

No chassis-contract.md entry — it's a response-shape + scheduler rule, not a Settings invariant. `auth.md` is the right home.
