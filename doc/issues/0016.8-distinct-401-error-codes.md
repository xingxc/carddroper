Traced to the 0016.3 cleanup chain firing an `attemptLogoutCleanup()` on every `/auth/me` 401 that lacks a session signal — true for anonymous visitors, post-logout reloads, AND stale-cookie ghost sessions. ## Context

0016.3's 401 interceptor in `frontend/lib/api.ts:172-223` runs `attemptLogoutCleanup()` whenever a non-exempt 401 comes back with `hasSession()=false`. Rationale at the time: the backend rejected the cookies → ask backend to clear them client-side so the proxy doesn't bounce `/login` ↔ `/app` on stale cookie presence (the ghost-session scenario).

**The branch fires in three observably-distinct situations today:**

1. **Anonymous visitor** loads `/`. No cookies, no sessionStorage. `GET /auth/me` → 401. Interceptor sees `hasSession()=false` → `attemptLogoutCleanup()` → `POST /auth/logout`. Pointless — there are no cookies to clear.
2. **Post-logout hard reload** (after 0016.7). Cookies were cleared by the intentional logout POST. sessionStorage was cleared in step 1 of `handleLogout`. After reload, `GET /auth/me` → 401. Same pointless cleanup.
3. **Stale-cookie ghost session** (Case 5 of the 0016.7 analysis). User closed a tab; opened a new one. sessionStorage is empty but cookies from the previous session persist. `GET /auth/me` → 401. Cleanup here IS needed — without it, the proxy sees the stale `access_token` cookie and bounces `/login` ↔ `/app`.

Cases 1 and 2 are the common path. Case 3 is rare but real. The interceptor can't distinguish them because `HttpOnly` cookies aren't JS-readable.

**But the backend already knows the difference.** `backend/app/dependencies.py:_authenticate` emits different messages for these cases:

- No token sent → `"Authentication required."`
- Token sent but decode failed → `"Invalid or expired token."`
- Token decoded but `tv` mismatched → `"Session invalidated. Please log in again."`

All three currently come out as code `"UNAUTHORIZED"` (via the single `errors.unauthorized()` helper). Promoting the first case into its own error code — `"AUTHENTICATION_REQUIRED"` — and letting the interceptor gate cleanup on that code closes Cases 1 and 2 while keeping Case 3 correct.

### Measurable impact (from 2026-04-23 local docker logs)

- **Every anonymous visit to any page under `AuthProvider`** (which is every page — `AuthProvider` is at the root) fires one wasted `POST /auth/logout`.
- **Every logout click** fires two `POST /auth/logout` (intentional + post-reload cleanup). 0016.7 reduced it from three; 0016.8 reduces it to one.
- **Rate-limit pressure:** `LOGOUT_RATE_LIMIT = 10/minute` per IP. A marketing page with 10+ anonymous page loads in a minute from the same NAT can trip the limit.

## Design decisions (pre-committed)

### Error code split (minimal, Option B)

- **New code `AUTHENTICATION_REQUIRED`** — emitted only when no token was sent at all. Single helper `missing_auth(message)` added to `backend/app/errors.py`.
- **New code `INVALID_TOKEN`** — emitted when a token was sent but rejected (decode failure, `tv` mismatch, or "session invalidated" path). Single helper `invalid_token(message)` added to `backend/app/errors.py`.
- **Existing code `UNAUTHORIZED` stays** for the non-auth-chain 401s (wrong login credentials, wrong current password on change-password/change-email, invalid reset/verify/confirm tokens on exempt endpoints, refresh-endpoint failures). These represent domain-level auth failures that aren't about cookie state.

**Why not rename `UNAUTHORIZED` wholesale?** Scope creep. The interceptor's cleanup decision only needs to distinguish "no token sent" from everything else. Renaming the domain 401s (login, change-password) adds test churn without user benefit — the frontend LoginBody already gates on `err.status === 401`, not `err.code`.

### Interceptor gate

- Parse the 401 response body ONCE (via `response.clone().json()` so the downstream `ApiError` throw can still read it), extract `error.code`.
- Cleanup gate becomes:
  ```ts
  if (hasSession()) {
    // existing refresh flow; on refresh-fail, markLoggedOut + cleanup
  } else if (parsedError?.code !== "AUTHENTICATION_REQUIRED") {
    // No session signal + backend says token was present (or unknown code) → cleanup
    await attemptLogoutCleanup();
  }
  // else (no session + AUTHENTICATION_REQUIRED): nothing to clean, skip.
  ```
- Conservative fallback: if parsing fails (`parsedError` is null), treat as unknown → fire cleanup. Correctness preserved; worst case is the status quo.

### What 0016.8 explicitly does NOT change

- **`UNAUTHORIZED` error code on login / refresh / change-password / reset / verify / confirm flows** — unchanged. Their 401s are domain errors, not cookie-state diagnostics.
- **`attemptLogoutCleanup` internals** — unchanged. Still a dedup-shared POST to `/auth/logout`.
- **`hasSession()` contract** — unchanged.
- **Proxy middleware** — unchanged.
- **Chassis-contract.md** — no entry. This is an error-code convention + interceptor refinement, not a startup invariant.

## Case matrix (post-fix)

| # | Scenario | `hasSession()` | Cookies present | Initial 401 code | Cleanup fires? |
|---|---|---|---|---|---|
| 1 | Anonymous visitor | false | none | `AUTHENTICATION_REQUIRED` | **No** (saves round-trip) |
| 2 | Access expired, refresh valid | true | access gone, refresh present | `AUTHENTICATION_REQUIRED` | No — refresh succeeds, retry returns 200 |
| 3 | Refresh also expired | true | access gone, refresh gone | `AUTHENTICATION_REQUIRED` | No — gate skips cleanup; next user action still works via 0016.3 chain if cookies somehow linger |
| 4 | Cross-device revocation, sessionStorage alive | true | access+refresh present but stale | `INVALID_TOKEN` | Yes (refresh fails → cleanup) |
| 5 | New tab, stale cookies, fresh sessionStorage | false | access present, stale | `INVALID_TOKEN` | Yes (gate passes through) |
| 6 | Post-logout reload | false | none (intentional logout cleared them) | `AUTHENTICATION_REQUIRED` | **No** (saves round-trip) |

Cases 4 and 5 preserve the 0016.3 correctness guarantee (no proxy loop). Cases 1 and 6 save the wasted round-trip. Cases 2 and 3 are unaffected by the cleanup gate (handled inside the `hasSession()` branch).

### Phase 0b — frontend (frontend-builder)

7. **`frontend/lib/api.ts`** — restructure the 401 handling so body parsing happens once, up-front:
   - Before the 401 interceptor, attempt to parse the response body as the `ApiErrorBody` shape. Use `response.clone().json()` so the downstream `!response.ok` throw can still read from the original response. (Only clone when `!response.ok` and JSON parse makes sense; guard with try/catch.)
   - Extract `parsedError?.code` — may be undefined if parse fails.
   - In the 401 interceptor: after the existing `if (hasSession()) { ... refresh flow ... markLoggedOut() }` block, replace the unconditional `await attemptLogoutCleanup()` with a guarded version:
     ```ts
     if (hasSession()) {
       // ... existing refresh flow + markLoggedOut() on failure ...
       // After markLoggedOut, still need cleanup: cookies were definitely present.
       await attemptLogoutCleanup();
     } else if (parsedError?.code !== "AUTHENTICATION_REQUIRED") {
       // No session signal. If backend indicates a token was present (INVALID_TOKEN)
       // or code is unknown (defensive), clean up. Skip only on explicit no-token.
       await attemptLogoutCleanup();
     }
     ```
     Note: this changes the control flow slightly — `attemptLogoutCleanup()` is now called from two explicit branches rather than the fall-through. Both branches still throw the original 401 after.
   - The parsed `error.code` / `error.message` can be reused for the `ApiError` throw at the bottom of the function, avoiding the second parse. Optional cleanup but reduces duplication.

8. **Do NOT change** `attemptLogoutCleanup`, `logoutCleanupPromise`, `attemptRefresh`, `getRefreshPromise`, or any exports. Only the interceptor's internal body parsing and the cleanup branch-out changes.

9. **Do NOT change** any consumer (`LoginBody`, `LogoutButton`, `AuthProvider`, layouts, pages). Error-code shape on the throw side is unchanged for non-401 paths; on 401 paths, the codes are new but no UI consumer gates on `err.code === "UNAUTHORIZED"` today (they gate on `err.status === 401`).

## Chassis implications

This ticket formalizes the auth-chain's 401 error-code vocabulary:

- **`AUTHENTICATION_REQUIRED`** — no auth credentials were sent.
- **`INVALID_TOKEN`** — auth credentials were sent but rejected (decode fail, tv mismatch, session invalidated).
- **`UNAUTHORIZED`** — domain-level auth failure unrelated to cookie state (wrong login credentials, invalid reset token on exempt endpoints, etc.).

Future adopters of this chassis inherit the vocabulary plus the client-side cleanup gate that uses it. Document in `STARTER.md` (when extracted per `site-model.md` Reusability constraint 7). Until then, `doc/systems/auth.md` carries it.
