## Context

`backend/app/routes/billing.py` lines 95-113 implement webhook idempotency as **SELECT then INSERT**:

```python
async with db.begin():
    # SELECT
    existing = await db.execute(select(StripeEvent).where(StripeEvent.id == event.id))
    if existing.scalar_one_or_none() is not None:
        return Response(status_code=200)
    # dispatch handler (may write balance_ledger)
    if handler: await handler(event, db)
    # INSERT
    db.add(StripeEvent(id=event.id, event_type=event.type))
# commit
```

This is race-prone. Two concurrent requests for the same event id can both pass the SELECT check (because neither has committed yet), both call the handler, both attempt to insert. One commits successfully (200); the other commit-fails on the primary-key constraint of `stripe_events.id` or the partial unique index on `balance_ledger.stripe_event_id` → uncaught `IntegrityError` → 500.

Concrete observation: 2026-04-24 13:24:57 — same event id `evt_3TPqG3GPSfG3zV2f067yznhm` delivered to backend by two `stripe listen` instances within milliseconds of each other; one returned 200, other returned 500. Race outcome is nondeterministic.

### Why this matters in production

Stripe doesn't normally fan out the same event to multiple webhook endpoints in a single delivery, BUT:

1. **Stripe retries on 5xx** — if our backend returns 500 (from the race-loser), Stripe schedules a retry. The retry can race with whatever other event delivery is in flight, perpetuating 500s and creating a feedback loop.
2. **Multiple endpoint configs** — common Cloud Run misconfig where dev + staging webhooks both point at the same URL during cutover, or blue/green deploys briefly fan to both.
3. **Future horizontal scaling** — when backend scales to >1 Cloud Run instance, two instances can each receive a webhook delivery with the same event id (e.g., one from initial delivery, one from a retry). Same race.
4. **Future retry queue / DLQ** — any reliability mechanism that re-enqueues events introduces concurrent-replay potential.

The chassis should be robust to all four. SELECT-then-INSERT is not.

## Design decisions (pre-committed)

1. **Replace with atomic `INSERT … ON CONFLICT (id) DO NOTHING`.** Postgres handles concurrent INSERTs at the storage layer via row-level locking on the conflicting index — exactly the primitive needed.

2. **Move the handler dispatch INSIDE the "we own this event" branch.** After the INSERT succeeds with `rowcount=1`, we know we're the sole owner of this event. Handler runs. Both `stripe_events` row and any `balance_ledger` writes from the handler commit atomically.

3. **Race losers return 200 without re-invoking the handler.** When `INSERT … ON CONFLICT` returns `rowcount=0`, we know another concurrent request committed first. Return 200 immediately. Handler does NOT run again.

4. **Handler failure semantics preserved.** If the handler raises an uncaught exception, the entire transaction (including the `stripe_events` INSERT) rolls back. Stripe sees a 5xx, retries the event later. On retry, the INSERT succeeds again (`rowcount=1`), handler runs again. Existing "best-effort with auto-retry" semantics intact.

5. **Soft-failure handler semantics preserved.** Handlers that log a warning and return on known-bad inputs (missing `metadata.user_id`, invalid user, zero amount — see `app/billing/handlers/topup.py`) commit the `stripe_events` row, marking the event as processed. Stripe doesn't retry. Same as today.

6. **Use SQLAlchemy's `sqlalchemy.dialects.postgresql.insert`.** New import; supports `.on_conflict_do_nothing(index_elements=["id"])`. No new dependency — already part of installed sqlalchemy.

7. **No middleware, no distributed locks, no Redis.** ON CONFLICT solves it at the DB level. Adding any out-of-band locking is over-engineering for a problem Postgres already solves atomically.

8. **No new schema migration.** `StripeEvent.id` is already `primary_key=True` (verified 2026-04-24 in `app/models/stripe_event.py:21`), giving us the unique index ON CONFLICT needs.

## Out of scope

- Distributed lock infrastructure (Redis, Postgres advisory locks).
- Webhook retry queue / dead-letter for repeatedly-failing webhooks.
- Idempotency on non-webhook endpoints (different concern; topup has Stripe-side idempotency-key already from 0023).
- Webhook endpoint rate limiting (Stripe IPs known; signature gate is sufficient).
- Schema changes to `stripe_events` or `balance_ledger`.
- Refactoring `EVENT_HANDLERS` registry (works as-is).
- Updating any other webhook-related code path; only the dedup mechanism in `routes/billing.py` changes.

Replace the existing block (lines 93-113 currently, the whole `async with AsyncSessionLocal() as db: async with db.begin(): ... return Response(status_code=200)` body) with:

```python
async with AsyncSessionLocal() as db:
    async with db.begin():
        # Atomic idempotency: try to claim this event_id by inserting it.
        # If another concurrent request already inserted (race) or this is a
        # plain duplicate retry, ON CONFLICT DO NOTHING returns rowcount=0
        # and we short-circuit. Postgres serialises concurrent INSERT-with-
        # conflict at the row-lock level, so exactly one transaction "owns"
        # the event id and the handler runs exactly once.
        stmt = (
            pg_insert(StripeEvent)
            .values(id=event.id, event_type=event.type)
            .on_conflict_do_nothing(index_elements=["id"])
        )
        result = await db.execute(stmt)
        if result.rowcount == 0:
            logger.info(
                "stripe_webhook_duplicate_event",
                extra={"event_id": event.id, "event_type": event.type},
            )
            return Response(status_code=200)

        # We own this event id. Dispatch the registered handler, if any.
        # The handler runs inside the same transaction as the stripe_events
        # INSERT, so a handler exception rolls back BOTH the handler's
        # writes (e.g., balance_ledger row in topup handler) AND the
        # stripe_events row — Stripe will then retry the event cleanly.
        handler = EVENT_HANDLERS.get(event.type)
        if handler:
            await handler(event, db)
        else:
            logger.warning("Unhandled Stripe event type: %s", event.type)

return Response(status_code=200)
```

Note: the trailing `db.add(StripeEvent(id=event.id, event_type=event.type))` is REMOVED. The INSERT happens at the top via `pg_insert`; no second insert needed.

**3. New test — `test_webhook_concurrent_delivery_no_race`:**

Fires two simultaneous webhook requests for the same event id via `asyncio.gather`. Both must return 200; exactly one `stripe_events` row; exactly one `balance_ledger` row (since the test event is `payment_intent.succeeded` and the topup handler grants).

```python
import asyncio

async def test_webhook_concurrent_delivery_no_race(client, db_session):
    """Two concurrent POSTs with the same event.id → both return 200, handler runs once."""
    # Set up: register a verified user (handler needs user.id from metadata)
    user = await _make_verified_user(db_session, email="concurrent@example.com")

    event = _build_pi_succeeded_event(
        event_id="evt_concurrent_001",
        user_id=user.id,
        amount_cents=2000,
    )
    payload, sig_header = _sign_event(event)

    # Fire two concurrent POSTs
    responses = await asyncio.gather(
        client.post("/billing/webhook", content=payload, headers={"stripe-signature": sig_header}),
        client.post("/billing/webhook", content=payload, headers={"stripe-signature": sig_header}),
    )

    # Both must succeed
    assert all(r.status_code == 200 for r in responses), [r.status_code for r in responses]

    # Exactly one stripe_events row
    stripe_event_rows = (
        await db_session.execute(select(StripeEvent).where(StripeEvent.id == "evt_concurrent_001"))
    ).all()
    assert len(stripe_event_rows) == 1

    # Exactly one balance_ledger row tied to this stripe_event_id (handler ran once)
    ledger_rows = (
        await db_session.execute(
            select(BalanceLedger).where(BalanceLedger.stripe_event_id == "evt_concurrent_001")
        )
    ).all()
    assert len(ledger_rows) == 1
    assert ledger_rows[0][0].amount_micros == 20_000_000  # $2.00 cents → micros
```

The `asyncio.gather` may or may not actually trigger the race in a single-event-loop test (asyncio is cooperative; tasks yield at await points). The test is **deterministic about correctness** — the assertions hold regardless of whether the race fired. If the race fires (likely with real Postgres), Postgres's row-lock + ON CONFLICT handles it; if it doesn't fire (deterministic interleaving), the second request hits the dedup branch the same way a sequential duplicate would. Either way, both succeed.

**4. New test — `test_webhook_handler_failure_does_not_record_event`:**

If a handler raises an exception, the transaction rolls back including the `stripe_events` INSERT. Stripe will retry. This test guarantees that property.

```python
async def test_webhook_handler_failure_does_not_record_event(client, db_session, monkeypatch):
    """Handler exception → transaction rolls back → no stripe_events row → Stripe will retry."""
    # Monkey-patch the topup handler to raise an unexpected exception.
    async def _failing_handler(event, db):
        raise RuntimeError("simulated handler bug")

    from app.billing.handlers import EVENT_HANDLERS
    monkeypatch.setitem(EVENT_HANDLERS, "payment_intent.succeeded", _failing_handler)

    event = _build_pi_succeeded_event(event_id="evt_fail_001", user_id=1, amount_cents=1000)
    payload, sig_header = _sign_event(event)

    resp = await client.post("/billing/webhook", content=payload, headers={"stripe-signature": sig_header})
    assert resp.status_code >= 500  # handler raised → uncaught → 5xx

    # No stripe_events row — Stripe will retry the event later
    rows = (
        await db_session.execute(select(StripeEvent).where(StripeEvent.id == "evt_fail_001"))
    ).all()
    assert len(rows) == 0

    # No balance_ledger row either
    ledger_rows = (
        await db_session.execute(
            select(BalanceLedger).where(BalanceLedger.stripe_event_id == "evt_fail_001")
        )
    ).all()
    assert len(ledger_rows) == 0
```

This documents the rollback contract explicitly.
