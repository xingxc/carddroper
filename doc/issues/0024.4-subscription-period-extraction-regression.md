## Context

User Phase 1 verification of 0024 Test 6a (cancellation) revealed that 9 subscription rows have NULL `current_period_start` and `current_period_end`. Only id=1 (the very first subscribe before 0024.2 landed) has them populated.

```
id  | grant_micros | current_period_start | current_period_end  | created_at
----|--------------|---------------------|---------------------|------------
 1  | 19990000     | 2026-04-26 22:30:27 | 2026-05-26 22:30:27 | pre-0024.2
 2  | 0            | NULL                | NULL                | post-0024.2
 4  | 14990000     | NULL                | NULL                | post-0024.2
 6  | 0            | NULL                | NULL                | post-0024.2
 ...
18  | 14990000     | NULL                | NULL                | post-0024.2 (cancel_at_period_end=true)
```

Stripe API has the values:
```
top-level current_period_start: 1777264865  (Unix timestamp)
top-level current_period_end:   1779856865
```

So the chassis bug is purely in extract/persist logic, not in Stripe payload.

### Why this matters

- **Cancel-at-period-end UX broken.** When user 25 cancels at period end (Test 6a passed in flag terms), `cancel_at_period_end=true` flips correctly, but `current_period_end` is NULL — the chassis can't tell adopters or users when scheduled access actually ends. SubscriptionDisplay can't render "ends on <date>".
- **Renewal logic depends on period data.** `handle_invoice_paid` in subscription_cycle branch updates these from invoice — so renewals self-heal — but that's a month away and meanwhile the data is wrong.
- **Project-layer queries** ("active subs ending soon", "renewal projection") return nothing useful while NULL.
- **Past-due dunning UX** ("update card before <date>") can't render the date.

### Why this happened (hypothesis)

Two candidates given the timing (id=1 OK pre-0024.2; id=2+ NULL post-0024.2):

1. **Subscribe endpoint regression** — 0024.2 modified `POST /billing/subscribe` to make `grant_micros` conditional. The agent likely refactored the upsert path and accidentally dropped `current_period_start` / `current_period_end` from the row construction.

2. **Webhook handler overwrite** — pre-0024.2, the webhook handler returned early on the `sub.items` extraction bug, never overwriting the subscribe endpoint's correctly-populated row. Post-0024.2, the handler now successfully runs and upserts — and if it doesn't extract `current_period_*` correctly, it overwrites the subscribe endpoint's good values with NULLs.

The fix is the same regardless of which is the root cause: ensure **both** the subscribe endpoint AND `handle_subscription_created` / `handle_subscription_updated` correctly extract and persist period fields. Belt-and-suspenders.

## Design decisions (pre-committed)

1. **Read from `getattr(sub, 'current_period_start', None)` and `getattr(sub, 'current_period_end', None)`** — defensive against dict-method-name collisions (Stripe's StripeObject is dict-backed; bracket-access is also acceptable: `sub["current_period_start"]`). The chassis already has dual-access patterns from 0024.2's items fix; mirror that style for periods.

2. **Convert Unix timestamps to UTC `datetime`** — Stripe returns Unix epoch seconds; the chassis schema stores `TIMESTAMP`. Conversion: `datetime.fromtimestamp(int_value, tz=timezone.utc)` (or `timezone.utc`-aware naive depending on existing chassis pattern — match `created_at`'s style).

3. **Persist in BOTH subscribe endpoint AND webhook handlers** — defense in depth. Subscribe endpoint sets initial periods from the Stripe API response; `handle_subscription_created` and `handle_subscription_updated` keep them synced from event payloads.

4. **Backfill existing NULL rows from Stripe API** — one-shot script in `backend/scripts/backfill_subscription_periods.py`. Fetches each NULL-period subscription's Stripe state and updates DB. Optional but worth including — leaves the chassis in a clean state without waiting a month for renewals to self-heal.

5. **Add regression tests** — assert `current_period_start` and `current_period_end` are populated after subscribe + after webhook events. New tests in `test_billing_subscribe.py` for the OUR upsert paths (subscribe endpoint + handle_subscription_created + handle_subscription_updated).

6. **No new env vars, no new chassis-contract entries.** Pure bug fix.

7. **No new migration.** Schema already has the columns; just fix the population logic.

8. **No frontend changes.** SubscriptionDisplay already renders `current_period_end` as the next-billing date; once backend populates correctly, frontend works without change.

## Out of scope

- **Multi-item subscriptions with mixed billing intervals** — Stripe's newer API surfaces per-item `current_period_*` for this case. Chassis is single-item v1 (one Subscription per user, one item per Subscription); top-level fields are sufficient. Multi-item is a future chassis extension.
- **Migration to alembic data migration** rather than one-shot script — a script is simpler for one-time backfill and doesn't pollute the migration chain.
- **Adding indexes on `current_period_end`** for "ending soon" queries — premature; add when adopter use case requires.
- **Trial period semantics** (`trial_start`, `trial_end`) — chassis is interval-agnostic but doesn't support trials in v1.
- **Frontend changes.**

**4. Backfill script** — `backend/scripts/backfill_subscription_periods.py`:

```python
"""One-shot: backfill subscriptions.current_period_start/end from Stripe.

Usage: docker-compose exec backend .venv/bin/python scripts/backfill_subscription_periods.py
"""
import asyncio
from datetime import datetime, timezone
import stripe
from sqlalchemy import select
from app.config import settings
from app.database import AsyncSessionLocal
from app.models.subscription import Subscription


async def main():
    stripe.api_key = settings.STRIPE_SECRET_KEY.get_secret_value()
    async with AsyncSessionLocal() as db:
        rows = (await db.execute(
            select(Subscription).where(
                (Subscription.current_period_start.is_(None))
                | (Subscription.current_period_end.is_(None))
            )
        )).scalars().all()
        print(f"Found {len(rows)} subscriptions with NULL period fields.")
        for sub in rows:
            try:
                stripe_sub = await asyncio.to_thread(
                    stripe.Subscription.retrieve, sub.stripe_subscription_id
                )
                start_ts = getattr(stripe_sub, "current_period_start", None) or stripe_sub.get("current_period_start")
                end_ts = getattr(stripe_sub, "current_period_end", None) or stripe_sub.get("current_period_end")
                if start_ts:
                    sub.current_period_start = datetime.fromtimestamp(int(start_ts), tz=timezone.utc).replace(tzinfo=None)
                if end_ts:
                    sub.current_period_end = datetime.fromtimestamp(int(end_ts), tz=timezone.utc).replace(tzinfo=None)
                await db.commit()
                print(f"Backfilled sub {sub.stripe_subscription_id}")
            except Exception as exc:
                print(f"FAILED {sub.stripe_subscription_id}: {exc}")
                await db.rollback()


if __name__ == "__main__":
    asyncio.run(main())
```
