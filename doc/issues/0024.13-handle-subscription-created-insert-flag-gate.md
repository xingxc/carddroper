## Audit (per `doc/operations/audit-template.md`)

### 1. User-visible action

User submits a card that fails the first invoice (decline `4000 0000 0000 0341` is the canonical case; any first-invoice failure scenario hits the same path) with `BILLING_SUBSCRIPTION_GRANTS_TO_LEDGER=false`. Subscribe endpoint correctly cleans up the failed Stripe sub and returns 402 (per 0024.9). User sees decline message + form reset (working post-0024.9 + 0024.10). But Stripe asynchronously delivers `customer.subscription.created` for the briefly-existed Subscription. Webhook handler creates a `subscriptions` row with `status=incomplete` and `grant_micros=19990000` — violating the chassis invariant that `grant_micros=0` when flag=false.

### 2. Full request flow trace

```
1. POST /billing/subscribe → Stripe creates Subscription (incomplete)
2. Subscribe endpoint's three-way branch: terminal failure (PI=requires_payment_method)
3. Subscribe endpoint: _cancel_failed_subscription(sub.id), raise 402 (per 0024.9)
   ← Subscribe endpoint deliberately does NOT upsert
4. ASYNC: Stripe queues customer.subscription.created (Stripe doesn't yet know about the cancel)
5. ASYNC: Stripe queues customer.subscription.deleted (in response to the cancel)
6. Webhook customer.subscription.created arrives at chassis
7. handle_subscription_created runs:
   - Row doesn't exist (subscribe endpoint didn't upsert)
   - INSERT path of pg_insert(...).on_conflict_do_update(...) executes
   - values= clause writes grant_micros=grant_micros_insert
   - grant_micros_insert is computed UNCONDITIONALLY from metadata.grant_micros = 19990000
   ← THE BUG: flag is not checked
8. Row created: status=incomplete, grant_micros=19990000  ← phantom value
9. Webhook customer.subscription.deleted arrives
10. handle_subscription_deleted: status='cancelled' (or eventually incomplete_expired)
    grant_micros UNCHANGED (handler doesn't touch the field — correct per 0024.7 Path B)
11. Row final state: status=cancelled, grant_micros=19990000
```

### 3. Endpoint properties

| Step | Writer | Behavior | Flag-gate respected? |
|---|---|---|---|
| 3 | Subscribe endpoint | Skips upsert on terminal failure (0024.9 design) | n/a — doesn't write |
| 7 | handle_subscription_created INSERT path (`values=`) | Writes grant_micros from metadata UNCONDITIONALLY | **NO — the bug** |
| (any) | handle_subscription_created UPDATE path (`set_=`) | Omits grant_micros (0024.7 Path B) | n/a — doesn't write |
| (any) | handle_subscription_updated | Doesn't touch grant_micros (0024.7 Path B) | n/a |
| (any) | handle_invoice_paid (cycle) | Writes grant_micros only when flag=true (per 0024.7) | YES |

### 3.5. Cross-writer assumption audit (NEW question this ticket adds)

**0024.7 design assumed:** "INSERT path is rare; subscribe endpoint will have already upserted with the flag-gated value before this handler runs." The comment in `handle_subscription_created` says exactly this: *"When flag=OFF the subscribe endpoint will have already stored 0; the INSERT branch is only exercised for truly out-of-band subscriptions."*

**0024.9 invalidated this assumption** by introducing a path where the subscribe endpoint deliberately does NOT upsert (terminal-failure branch). After 0024.9, the INSERT path of handle_subscription_created is no longer "rare out-of-band only" — it runs for every decline.

This invalidation was not flagged during 0024.9's audit because no audit step explicitly required cross-writer assumption review. **0024.13 closes the bug AND adds the missing audit-template question** (see Phase 0a §4).

### 4. Consumability check

Not relevant to this bug (it's a flag-gate hole, not a Stripe-resource-state bug).

### 5. Adversarial scenario

`flag=false` + decline card → 0024.9 cleanup runs → subscribe endpoint returns 402 without upserting → ASYNC Stripe webhook arrives → handler INSERTs row with metadata-derived grant_micros=19990000 → chassis invariant "row.grant_micros respects flag uniformly" is violated. Same shape recurs for any first-invoice failure: 3DS-fail (handled by 0024.11), invoice-decline, network-failure-mid-confirm, etc.

### 6. Test coverage

**Existing tests don't catch this:**
- `test_handle_subscription_created_*` mocks scenarios where the row exists OR uses flag=true. None exercise the specific composition: flag=false + INSERT path (no pre-existing row) + truthy metadata.grant_micros.
- `test_subscribe_returns_402_when_first_invoice_declined` exercises the subscribe-endpoint side but doesn't fire the subsequent webhook in the same test.

**Required composition test (this ticket):** `test_handle_subscription_created_insert_path_respects_flag_off_grant_micros` — explicit flag=false + no-prior-row + metadata.grant_micros="19990000" → asserts row created with grant_micros=0.

## Context — why this is the THIRD compounding-assumption bug in the arc

The 0024.x arc has surfaced this pattern three times:

| Ticket | Composition that broke |
|---|---|
| 0024.5 | webhook period writes overlaid subscribe endpoint period writes |
| 0024.7 | webhook UPDATE-path overlaid subscribe endpoint INSERT writes (grant_micros) |
| 0024.13 (this) | webhook INSERT path doesn't respect a flag the subscribe endpoint respects |

Each was an honest design decision that became wrong when an adjacent ticket changed an assumption. The audit-template's Q3 (per-endpoint properties) and Q5 (adversarial scenario) catch SOME of these but miss the specific class where one writer's behavior change invalidates another writer's assumptions.

This ticket adds **Q3.5 (cross-writer assumption audit)** to the audit template, scoped to fire whenever a ticket changes when/whether/how any writer writes a column.

## Design decisions (pre-committed)

### Backend fix

1. **In `handle_subscription_created`, replace the unconditional `grant_micros_insert` computation (lines ~195–208) with a flag-gated branch:**

```python
# Flag-gated grant_micros for the INSERT path. The chassis discipline is uniform:
# subscriptions.grant_micros = 0 when flag=false, regardless of writer or path.
# Originally not flag-gated under the assumption (now false post-0024.9) that the
# subscribe endpoint always upserted with its own flag-gated value before this
# handler ran. Ticket 0024.13 closes the hole exposed by 0024.9's terminal-failure
# branch (which deliberately skips the upsert).
if settings.BILLING_SUBSCRIPTION_GRANTS_TO_LEDGER:
    raw_grant = (
        price_meta_raw.get("grant_micros") if hasattr(price_meta_raw, "get") else None
    ) or "0"
    try:
        grant_micros_insert = int(raw_grant)
    except (ValueError, TypeError):
        grant_micros_insert = 0
else:
    grant_micros_insert = 0
```

2. **Remove the misleading comment block at lines ~195–199** (the one that says "When flag=OFF the subscribe endpoint will have already stored 0"). It's wrong post-0024.9. Replace with the new comment at the top of the if/else (above).

3. **Do NOT touch the UPDATE-path (`set_=`) clause.** It already correctly omits grant_micros (per 0024.7 Path B).

4. **Do NOT touch `tier_name` extraction.** It's not flag-gated and shouldn't be.

### Composition test

5. **Add `test_handle_subscription_created_insert_path_respects_flag_off_grant_micros`** in `backend/tests/test_billing_subscribe.py`.

Required behavior:
- Pre-conditions: User exists in DB. NO subscriptions row for that user (simulates 0024.9 terminal-failure path where subscribe endpoint chose not to upsert).
- Build a `customer.subscription.created` event with `metadata.user_id` set, `items.data[0].price.metadata = {"grant_micros": "19990000", "tier_name": "Starter"}`. Use `spec=`-restricted MagicMock or dict fixtures (mandatory per 0024.12 discipline).
- Patch `BILLING_SUBSCRIPTION_GRANTS_TO_LEDGER=False`.
- Fire `handle_subscription_created(event, db)`.
- Assertions:
  - subscriptions row was created (INSERT path fired).
  - `row.grant_micros == 0` ← the regression guard. Without the fix, this is 19990000.
  - `row.status == "incomplete"` (status from event, not flag-related).
  - `row.tier_name == "Starter"` (tier_name preserved — confirms the fix doesn't accidentally zero non-grant fields).

### Backfill

6. **Manual SQL after fix lands:**

```sql
UPDATE subscriptions
SET grant_micros = 0
WHERE status IN ('incomplete', 'incomplete_expired', 'cancelled')
  AND grant_micros > 0;
```

Safe regardless of flag state at original creation time: incomplete/cancelled rows never reach a grantable state (per 0024.11 grants only fire on `invoice.paid` for active subs), so `row.grant_micros` is informational only and zero is the chassis-correct value for non-active subs.

### Discipline scaffolding update (Phase 0a docs — same commit as the code fix)

7. **`doc/operations/audit-template.md` — add Question 3.5 between Q3 and Q4:**

> ### 3.5. If this ticket changes WHEN or WHETHER any writer writes a column, audit every other writer to that column for assumptions that may now be invalidated
>
> When a ticket modifies a writer's behavior — e.g., reordering when an upsert happens, adding an early return that skips a write, changing a condition that gates a write — list every OTHER writer to the same column and check each writer's implicit or explicit assumptions about when it runs.
>
> Common assumptions to look for:
>
> - "I always run after writer X has populated this row, so my INSERT path is rare."
> - "Writer X always handles the flag-gate, so I can use the unflagged metadata value."
> - "Writer X always writes a non-NULL value before me, so I don't need a fallback."
>
> If the ticket invalidates any such assumption, the affected writer needs to be updated in the same ticket OR explicitly carved into a follow-up.
>
> Origin: ticket 0024.13. Tickets 0024.7 (webhook INSERT path assumed subscribe endpoint always upserted first) and 0024.9 (subscribe endpoint stopped upserting on terminal failure) composed into a chassis-correctness bug because no audit explicitly checked the cross-writer assumption.

8. **`doc/systems/payments.md` §Field ownership — append after the existing "Writer rules" subsection:**

> **Preserve-only writers' INSERT-path values** (the rare path where they write because no row exists yet) must respect the same flag-gates and chassis invariants as the authoritative writer. The INSERT path is not a "fallback that ignores the chassis flag" — it's a defensive write under the same semantic.
>
> Concrete: when a webhook handler is preserve-only on UPDATE (omits a column from `set_=`), its INSERT path (`values=`) MUST still apply any flag-gate the authoritative writer applies. The INSERT path is exercised when the authoritative writer didn't run (or ran but skipped the upsert, e.g., 0024.9's terminal-failure path).

## Out of scope

- Other columns that might have similar preserve-only INSERT-path holes — none identified during the system-level audit (only `grant_micros` has flag-gate semantics; period fields, status, tier_name, etc. don't).
- Refactoring `handle_subscription_created` to refuse INSERT entirely (treating subscribe endpoint as the only initial writer) — would break legitimate out-of-band subscriptions (Stripe Dashboard, Customer Portal); not done.
- Renaming or restructuring the helpers — pure correctness fix; no surface-area expansion.
- Frontend changes — none needed.
