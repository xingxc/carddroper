## Audit (per `doc/operations/audit-template.md`)

### 1. User-visible action

A subscriber's billing anchor arrives. Their stored payment method has degraded since subscribe (card expired, account closed, insufficient funds, fraud-flagged). Stripe attempts to charge → fails → fires `invoice.payment_failed`. The subscription transitions to `past_due` server-side. Already-granted balance remains fully spendable per chassis spec. User is expected to fix payment via Customer Portal (handled by 0025); this ticket only verifies the chassis enters past_due correctly so 0025's recovery flow has a valid starting state to recover FROM.

### 2. Full request flow trace

```
1. Save current sub.default_payment_method (e.g., pm_X — the working card)
2. Attach pm_card_chargeCustomerFail to the test customer
3. stripe.Subscription.modify(default_payment_method=pm_card_chargeCustomerFail)
   → fires customer.subscription.updated webhook
4. Wait ~5s for the .updated handler to settle
5. Capture pre-state DB (post-PM-swap, pre-clock-advance)
6. Advance test clock 31 days
7. Stripe creates renewal Invoice
8. Stripe charges pm_card_chargeCustomerFail → FAILS
9. Stripe fires invoice.payment_failed (basil shape)
10. Chassis handle_invoice_payment_failed:
    a. extract_invoice_subscription_id (basil canonical path)  ← 0024.12
    b. resolve subscriptions row by sub_id
    c. row.status = "past_due"
    d. (No other writes — Path B preservation by omission)
11. Stripe also fires customer.subscription.updated (status changed Stripe-side)
12. Chassis handle_subscription_updated:
    - status sync only (idempotent re-affirm of past_due)
    - period + grant_micros UNCHANGED (Path B per 0024.5+0024.7+0024.13)
13. Wait ~15s for webhook delivery, capture post-state DB
14. Run assertions
15. (finally) Restore original PM: stripe.Subscription.modify(default_payment_method=pm_X)
16. Optionally: stripe.Invoice.pay(failed_invoice_id) to bring sub back to active
    for clean fixture re-use
```

### 3. Endpoint properties

| Step | Handler | Idempotency | DB writes | Stripe state |
|---|---|---|---|---|
| 10 | handle_invoice_payment_failed | per event_id (stripe_events UNIQUE) | row.status='past_due' only | invoice failed; PI in canceled/requires_payment_method state |
| 12 | handle_subscription_updated | per event_id | row.status (idempotent re-affirm) | sub status flipped past_due Stripe-side |

### 3.5. Cross-writer assumption check

This ticket is test infrastructure only — NO chassis code is modified. No writers' assumptions change.

The PM swap (step 3) goes through Stripe API directly, not chassis code. The resulting `customer.subscription.updated` webhook does fire and the chassis handler runs — but the handler's behavior is well-defined: it syncs status / cancel_at_period_end / tier metadata and preserves period/grant_micros (Path B). The PM swap itself doesn't change the subscription's tier or period, so the .updated handler should be a near-no-op except for setting `default_payment_method` (which the chassis doesn't track).

### 4. Consumability check

- **Invoice**: consumable but NOT replayed (failed invoices stay in failed state until paid via Stripe Invoice.pay or canceled; chassis idempotency via stripe_events.id UNIQUE handles duplicate webhook delivery)
- **PaymentMethod**: not consumable; can be re-attached/detached/swapped freely
- **Subscription**: stays past_due until dunning resolves (chassis doesn't drive dunning; Stripe owns it)

No idempotency-policy violations introduced.

### 5. Adversarial scenario

Most likely failure modes the script will catch:

1. **`handle_invoice_payment_failed` doesn't update status** — chassis fails to transition to past_due; row stays active; recovery UX (0025) starts from wrong state
2. **Path B violation: handler accidentally writes period or grant_micros** — would lose the chassis invariant under failure path; test catches via "fields unchanged" assertions
3. **Phantom ledger write** — payment failed but somehow a ledger entry got posted (regression of 0024.11 grant-coupling); critical for `flag=true` adopters
4. **Status flips to wrong value** (e.g., `cancelled` instead of `past_due`) — would prevent dunning recovery; test catches by checking the exact value
5. **Idempotency dedup fails** — Stripe re-delivers the failed event; chassis flips status twice or somehow corrupts state (regression of 0023.2 atomic INSERT)
6. **PM-swap webhook side effect** — `customer.subscription.updated` fired by the PM swap accidentally writes a ledger entry or touches period/grant_micros (regression of 0024.5+0024.7 Path B for subscription.updated handler)

Each is plausible because the failure path was refactored less than the success path during the arc.

### 6. Test coverage

Existing unit tests cover `handle_invoice_payment_failed` with mocked basil-shape events:
- `test_handle_invoice_payment_failed_marks_past_due`

But no real-Stripe-event verification. The script extension is the integration test that mocked-tests structurally cannot replicate.

## Context

`stripe-side-tests.md` §Tier B documents test-clock testing. 0024.14 implemented the success path (B1). This ticket is the failure-path counterpart. Same fixture, different test path.

The mechanism — Stripe ships a set of pre-built test PaymentMethods that succeed at attach but fail on charge:

| Token | Behavior |
|---|---|
| `pm_card_chargeCustomerFail` | Attaches; **fails when charged** (default for renewal-failure simulation) |
| `pm_card_chargeDeclinedExpiredCard` | Attaches; fails with "expired card" |
| `pm_card_chargeDeclinedFraudulent` | Attaches; fails as fraudulent |
| `pm_card_chargeDeclinedInsufficientFunds` | Attaches; fails with insufficient funds |

Default to `pm_card_chargeCustomerFail` (most generic). The ticket's design is extensible to other failure types via a future `--decline-reason` flag, but that's out of scope for v1.

## Why a script extension, not a separate file

`test_renewal.py` already has the success-path infrastructure: fixture loading, DB state capture, clock advance, webhook wait, assertion reporting, ANSI color output. The failure path differs only in:
- Mid-test PM swap (and finally-restore)
- Different assertion set

Extending the existing file with a `--simulate-decline` flag reuses the infrastructure and keeps the test rig as ONE coherent tool. Future failure modes (e.g., PM expiry, fraud, dunning-then-recover) can extend the same file with additional flags.

## Why this should land before 0025

0025 is the Customer Portal redirect — the recovery UX for past_due subscriptions. Sequencing:

1. **0024.14** (done): renewal-success path verified end-to-end
2. **0024.15** (this): renewal-failure path verified — chassis correctly enters past_due
3. **0025**: Customer Portal redirect — user updates PM via Stripe-hosted UI; chassis observes recovery via subsequent webhooks

Without 0024.15 verifying the entry condition, 0025 builds recovery UX on top of an unverified foundation. Each ticket should land on a validated layer.

## Design decisions (pre-committed)

### Backend (Phase 0a)

1. **Extend `backend/scripts/test_renewal.py`** with a `--simulate-decline` flag. Mutually exclusive with `--dry-run` is NOT required — `--dry-run` can show the planned PM swap without executing. Default behavior (no flag) is unchanged: success-path verification (0024.14).

2. **Failure-path flow when `--simulate-decline` is set:**
   - Read current `sub.default_payment_method` from Stripe → save as `original_pm_id`
   - `stripe.PaymentMethod.attach('pm_card_chargeCustomerFail', customer=cus_X)`
   - `stripe.Subscription.modify(sub_X, default_payment_method='pm_card_chargeCustomerFail')`
   - Sleep 5s for the `customer.subscription.updated` webhook to settle
   - Capture pre-state DB
   - Advance test clock by `--days` (default 31)
   - Sleep `--wait` seconds (default 15)
   - Capture post-state DB
   - Run six assertions (below)
   - `finally`: restore `default_payment_method=original_pm_id` (always; even on failure)
   - Optional: attempt `stripe.Invoice.pay(failed_invoice_id)` to bring sub back to active for clean fixture re-use; document as a flag (`--restore-active` default true) so the user can opt out

3. **Six assertions for failure path (inverted from success path's six):**
   - `status` flipped from `active` (or whatever pre-state was) to `past_due`
   - `current_period_start` UNCHANGED (post == pre)
   - `current_period_end` UNCHANGED (post == pre)
   - `grant_micros` UNCHANGED (post == pre)
   - NO new `subscription_reset` ledger entry posted
   - NO new `subscription_grant` ledger entry posted

4. **Assertion structure mirrors the success path** — same `_assert_*` helper signatures, same RED/GREEN reporting, same final summary table format. This keeps the script's UX consistent across success/failure modes.

5. **Restore logic is idempotent and best-effort.** If the original PM is already detached (rare; user manually deleted it via Dashboard), log a warning and continue. The fixture's clock+customer survive; the user can reattach a working card via the chassis subscribe flow if needed.

6. **DO NOT modify any chassis code.** This ticket is test infrastructure only. Verify via `git diff backend/app/` returning empty before commit.

### Out of scope

- **Other failure-mode test cards** (`pm_card_chargeDeclinedExpiredCard`, etc.) — default to `pm_card_chargeCustomerFail`; future tickets can extend.
- **Dunning retry simulation** — Stripe's dunning schedule is configurable per project; the script doesn't simulate the multi-retry flow. Past_due → unpaid transitions are deferred.
- **Recovery-flow verification** — the script optionally restores the PM and re-pays the failed invoice for fixture re-use, but does NOT verify the recovery webhooks (sub goes past_due → active). That's 0025's territory.
- **Customer Portal integration** — 0025.
- **Multi-currency** — out of v1 scope per `payments.md`.
- **CI integration** — same posture as 0024.14; the script is a manual-run dev tool.
