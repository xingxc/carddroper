found_by: 0024.9 Phase 1 manual smoke 2026-04-28 — after 0024.9's soft-reset shipped, retrying 4000 0000 0000 0341 still produced "processing error" + IntegrationError ("elements should have a mounted Payment Element"). ## Audit (per `doc/operations/audit-template.md`)

### 1. User-visible action triggering this code path

User submits a card that fails the first invoice (e.g., test card `4000 0000 0000 0341`, or any production card declined for insufficient funds), then clicks Subscribe again **within ~60 seconds** of the original submit.

### 2. Full request flow trace

```
1. Page load: GET /app/subscribe                       → renders tier list
2. User clicks tier: POST /billing/setup-intent (#1)   → creates Stripe SetupIntent (consumable)
3. frontend stripe.confirmSetup()                       → consumes SetupIntent, attaches PM
4. POST /billing/subscribe                              → 402 INVOICE_PAYMENT_FAILED (per 0024.9)
                                                          backend cancels failed Stripe Subscription
5. frontend handleInvoiceDeclined fires (per 0024.9):
   a. setError(msg)
   b. setClientSecret(null) — Elements unmounts
   c. setPhase("fetching_secret")
   d. await POST /billing/setup-intent (#2)             ← the buggy call
   e. setClientSecret(newSecret)
   f. setPhase("card_entry")
6. Elements remounts with response from #2
7. User clicks Subscribe → handleSubmit → stripe.confirmSetup
                                          → IntegrationError (thrown, not returned)
                                          → form stuck in "submitting" forever
```

### 3. For each endpoint above

| Step | Endpoint | Idempotency key | DB state | Stripe state | Retry semantics |
|---|---|---|---|---|---|
| 2 | `POST /billing/setup-intent` | `f"setup:{user.id}:{int(time.time() // 60)}"` (time-window) | none | creates SetupIntent | replay if same minute |
| 3 | (Stripe-side) `confirmSetup` | n/a | none | SetupIntent → `succeeded` | terminal state |
| 4 | `POST /billing/subscribe` | `f"subscribe:{user.id}:{lookup_key}:{pm_id}"` (content) | upsert subscriptions | creates Subscription, may cancel | per-PM keyed (post-0024.6) |
| 5d | `POST /billing/setup-intent` (re-fetch) | **same key as step 2** | none | **replay returns step-2 SetupIntent** | replay if same minute → consumed SI returned |

### 4. Consumability check

| Resource | Consumable? | Endpoint | Idempotency pattern | Violation? |
|---|---|---|---|---|
| SetupIntent (step 2) | **YES** | `POST /billing/setup-intent` | time-window | **VIOLATION** — time-window key on consumable resource |
| PaymentMethod (step 3) | no | n/a | n/a | n/a |
| Subscription (step 4) | YES | `POST /billing/subscribe` | content-based (incl PM) | OK |

`SetupIntent` appears in the consumable-resource catalog as `yes`. Per `doc/operations/idempotency-policy.md`, time-window keys for consumable resources are forbidden. **The setup-intent endpoint violates the policy.**

### 5. Adversarial scenario

Soft-reset re-fetches setup-intent within the same wall-clock minute as the original fetch. Stripe's idempotency replay returns the cached response — i.e., the SetupIntent object as it was at original creation, with status `requires_confirmation` and the original `client_secret`. But the actual SetupIntent's current state on Stripe is `succeeded` (consumed by step 3's confirm). Frontend mounts `<Elements>` with the replayed `client_secret`. Stripe.js fetches the live state → `succeeded` → PaymentElement enters a non-mounted state. User clicks Subscribe → `confirmSetup` throws IntegrationError ("elements should have a mounted Payment Element"). Frontend's `handleSubmit` doesn't catch thrown exceptions → form stuck.

### 6. Test coverage

**No existing test exercises this scenario.** The setup-intent unit tests (`test_setup_intent_creates_setup_intent_on_stripe`, `test_setup_intent_lazy_creates_customer`) verify single-call behavior. The 0024.9 subscribe-decline tests verify the 402 response shape. **No test exercises the composition: setup-intent + subscribe-decline + re-fetch-setup-intent.** This is the audit-template's canonical missing-test scenario.

The test that should be added: `test_subscribe_decline_retry_uses_fresh_setup_intent` (a composition test per `doc/operations/testing.md` §Composition tests).

## Context — why this exists

0024.9 introduced a soft-reset path: on 402 INVOICE_PAYMENT_FAILED, the frontend re-fetches a fresh SetupIntent so the user can retry with a different card. The re-fetch is the new code path that exposes the bug.

The setup-intent endpoint's `f"setup:{user.id}:{int(time.time() // 60)}"` idempotency key was added in the original 0024 ticket as defensive deduplication against rapid double-clicks. The defense was speculative — there was no documented threat. Per the new `idempotency-policy.md`: speculative defense is a code smell. In this case the speculative defense created the bug it was supposed to prevent (broken UX on retry).

## Design decisions (pre-committed, derived from the new disciplines)

### 1. Drop the idempotency_key from setup-intent entirely (Pattern: per-request)

```python
# Before:
idempotency_key = f"setup:{user.id}:{int(time.time() // 60)}"
si = stripe.SetupIntent.create(..., idempotency_key=idempotency_key)

# After:
si = stripe.SetupIntent.create(...)  # no idempotency_key
```

Per `doc/operations/idempotency-policy.md`, the per-request pattern is correct for consumable resources. The frontend's `submitting` state already prevents user-driven double-clicks; backend rate limits are the systemic backstop; Stripe's account-level rate limits are the final guardrail. No threat justifies retaining the time-window key.

### 2. Defensive try/catch in `handleSubmit` for thrown Stripe.js errors

`stripe.confirmSetup(...)` returns `{error}` for most card errors (handled). But it **throws** for IntegrationError and similar structural failures. The current `handleSubmit` doesn't catch thrown errors → form stuck.

Wrap the entire confirmSetup + subscribe flow in try/catch. On thrown error: surface a generic "Something went wrong, please try again" via `onError`, reset phase to `card_entry` so the form is interactive again.

### 3. Keep subscribe endpoint and 0024.9 backend unchanged

0024.9's three-way branch on PI status is correct. The setup-intent fix is the missing piece.

### 4. Composition test that would have failed pre-0024.10

`test_subscribe_decline_retry_uses_fresh_setup_intent` — see Phase 0a §Tests. This test asserts that two consecutive calls to `POST /billing/setup-intent` (simulating step 2 and step 5d above) return **different** `client_secret` values. With the bug: same minute → same response → assertion fails. With the fix: fresh SetupIntent each call → assertion passes.

### 5. Time-mocked test for resilience

Use `freezegun` to control wall-clock time in the test so the minute-boundary behavior is deterministic. `freezegun` should be added to `backend/pyproject.toml` test deps.

### 6. No frontend tests added

Frontend has no test runner installed (Vitest deferred from 0024.9). Adding scaffolding is out of scope for 0024.10. Manual test plan covers the frontend behavior.

- Stripe webhook signature retry hardening — unrelated.

### Phase 0b — frontend (frontend-builder)

**1. Wrap `stripe.confirmSetup(...)` in `frontend/components/billing/SubscribeForm.tsx` with try/catch.**

The current `InnerSubscribeForm.handleSubmit` does:

```typescript
const setupResult = await stripe.confirmSetup({ elements, redirect: "if_required" });
if (setupResult.error) { ... handle ... }
```

Stripe.js's `confirmSetup` returns `{error}` for card errors (handled) but THROWS for IntegrationError and similar structural failures. The current code has no catch for the throw → form stuck.

Wrap the await in try/catch:

```typescript
let setupResult;
try {
  setupResult = await stripe.confirmSetup({ elements, redirect: "if_required" });
} catch (err) {
  // Structural Stripe.js error (IntegrationError, etc.). Surface as generic
  // error and let the user retry. Likely cause: stale Elements / consumed
  // SetupIntent — once 0024.10 backend lands, this should be unreachable in
  // normal flows, but defensive catch prevents future structural failures
  // from bricking the form.
  onError("Something went wrong, please try again.");
  return;
}
if (setupResult.error) { ... existing ... }
```
