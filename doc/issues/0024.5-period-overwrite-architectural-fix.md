## Context

Diagnostic data from user's Phase 1 testing (15 subscriptions across two days):

| id | When | stripe listen state | period populated? |
|---|---|---|---|
| 1 | yesterday pre-0024.2 | ON (webhook handler had pre-0024.2 sub.items bug → returned early → no overwrite) | ✅ |
| 2–18 | yesterday post-0024.2 | ON (handler ran end-to-end, wrote NULL because period extraction returned None) | ❌ |
| 20, 21 | today, after 0024.4 + before stripe listen restart | **OFF** (webhook never delivered → no overwrite) | ✅ |
| 22, 24, 26 | today, after stripe listen restart | **ON** (webhook delivered → handler ran → overwrote with NULL) | ❌ |

The pattern is unambiguous. **The bug is that the webhook handler treats itself as a source of truth for period fields, and its extraction is unreliable.** When `stripe listen` is off, no webhook arrives, no overwrite happens, and the subscribe endpoint's correctly-populated row survives. When `stripe listen` (or production Stripe direct delivery) is on, the webhook handler runs and writes NULL.

0024.4's defensive extraction fix worked for the SUBSCRIBE ENDPOINT (uses Stripe SDK API call, gets top-level `current_period_*`). It did NOT fix the WEBHOOK HANDLER reliably — likely because the webhook event payload uses a different Stripe API version (e.g., `2025-03-31.basil`) where `current_period_*` moved to subscription items, OR the StripeObject serialization in event payloads differs subtly from API-fetched objects.

This will hit production identically the moment we push, because Stripe's direct delivery is functionally equivalent to `stripe listen` — both deliver event payloads to the chassis webhook URL.

## Design decisions (pre-committed)

The chassis-correct architecture: **`customer.subscription.created` and `customer.subscription.updated` handlers should NOT be sources of truth for period fields.** They're for state synchronization (status, `cancel_at_period_end`, tier metadata on plan changes) — not period authoritative writes.

Period fields come from exactly two reliable sources:

1. **Subscribe endpoint** at creation — calls `stripe.Subscription.create()`, gets a Subscription object with `current_period_*` at top level (works in current Stripe SDK API version), extracts and persists. **Already working post-0024.4.**

2. **`invoice.paid` (subscription_cycle branch)** at renewal — when Stripe fires the renewal invoice, `invoice.lines.data[0].period.start/end` is the canonical new period. Stripe guarantees this structure for renewal invoices regardless of API version (it's invoice-payload semantics, not subscription-payload semantics). The chassis already handles this event for granting `subscription_reset`; extend to also update `current_period_*`.

### What this means concretely

| Handler | Period writes? | Rationale |
|---|---|---|
| Subscribe endpoint | ✅ writes on insert | Source of truth at creation |
| `handle_subscription_created` | ❌ DROP — only writes other state | Webhook payload extraction unreliable across API versions |
| `handle_subscription_updated` | ❌ DROP — only writes other state | Same |
| `handle_subscription_deleted` | ❌ no change (already doesn't touch periods) | |
| `handle_invoice_paid` (subscription_create branch) | ❌ no period writes; periods already set by subscribe endpoint | Initial period is endpoint's job |
| `handle_invoice_paid` (subscription_cycle branch) | ✅ NEW — writes from `invoice.lines.data[0].period.*` | Authoritative for renewal periods |
| `handle_invoice_payment_failed` | ❌ no change (doesn't touch periods) | |

Trade-off: if a user changes plans mid-cycle (e.g., upgrades from Starter to Pro), Stripe fires `customer.subscription.updated` with new period boundaries. The chassis won't capture them at that exact moment — but on the next `invoice.paid` (which Stripe fires for the new prorated invoice OR the next cycle), they get updated. Acceptable lag for v1; chassis doesn't surface in-app plan-change UI in the first place (Customer Portal in 0025 handles that, and the Portal-driven plan change ALSO fires `invoice.paid` for the new prorated invoice → period gets updated).

### Why this is more correct than Path A (defensive layering with items-fallback)

Path A would have added items-level period extraction to the webhook handlers (handle the API-version variance). It would WORK, but it would still treat the webhook handler as a period-source — embedding chassis-fragility in the most version-volatile part of the Stripe API surface (which fields move where between API versions). Path B (this ticket) puts period authority on stable code paths only:
- Subscribe endpoint: SDK-pinned API version; we control it.
- `invoice.paid` lines.period.start/end: invoice schema, much more stable across Stripe API versions.

Path B is the chassis-discipline answer.

## Out of scope

- **In-app plan-change UI** with mid-cycle period adjustments — not in chassis v1; Customer Portal (0025) handles it via Stripe-hosted UI; the chassis just consumes the resulting `invoice.paid` events.
- **Multi-item subscription period handling** — single-item v1 only; reading `invoice.lines.data[0]` is sufficient.
- **Backfill of currently-NULL rows beyond running the existing 0024.4 script** — the existing `scripts/backfill_subscription_periods.py` from 0024.4 already handles this via `stripe.Subscription.retrieve()`; reuse it.
- **Defensive items-level fallback in created/updated handlers** — explicitly NOT this ticket's approach (would be Path A; we're going Path B).
- **Frontend changes** — chassis SubscriptionDisplay reads from API response; no change.
- **Schema changes** — columns already exist.
- **Chassis-contract entry** — not a startup invariant; same as 0024.2/0024.3/0024.4.

**1. `handle_subscription_created` — drop period writes from UPDATE path; keep on INSERT path.**

The handler uses `pg_insert(...).on_conflict_do_update(...)` (or equivalent SQLAlchemy upsert). Currently the `set_=` clause likely includes `current_period_start` + `current_period_end`. Remove them from the UPDATE set. Keep them in `values=` for the INSERT case (rare but possible — e.g., admin-created subscription out-of-band where webhook is first to insert).

```python
# Approximate shape — adapt to actual code:
stmt = pg_insert(Subscription).values(
    user_id=user_id,
    stripe_subscription_id=sub.id,
    ...,
    current_period_start=start,    # Still in INSERT for fresh rows
    current_period_end=end,
)
stmt = stmt.on_conflict_do_update(
    index_elements=["user_id"],
    set_={
        # status, tier_key, tier_name, grant_micros, cancel_at_period_end, updated_at
        # NOTE: current_period_* deliberately omitted — subscribe endpoint and
        # invoice.paid handler are authoritative; this handler only syncs state
        # fields that change post-creation.
        ...
    },
)
```

**2. `handle_subscription_updated` — drop period writes entirely.**

This handler runs on plan changes, status transitions, cancel-at-period-end flag flips, etc. Period boundaries on plan changes will be captured by the subsequent `invoice.paid` event for the prorated invoice. Remove `current_period_start` and `current_period_end` from the UPDATE statement entirely.

**3. `handle_invoice_paid` (subscription_cycle branch) — extend with period writes.**

When `invoice.billing_reason == 'subscription_cycle'` (renewals), in addition to the existing `subscription_reset` grant logic (gated on `BILLING_SUBSCRIPTION_GRANTS_TO_LEDGER`), update the subscriptions row's period fields from the invoice payload:

```python
# Approximate shape:
new_start = None
new_end = None
lines_data = (
    invoice.lines.data if hasattr(invoice.lines, "data")
    else invoice.get("lines", {}).get("data", [])
)
if lines_data:
    line = lines_data[0]
    period = (getattr(line, "period", None) or line.get("period", {}))
    raw_start = getattr(period, "start", None) or period.get("start")
    raw_end = getattr(period, "end", None) or period.get("end")
    if raw_start is not None:
        new_start = datetime.fromtimestamp(int(raw_start), tz=timezone.utc).replace(tzinfo=None)
    if raw_end is not None:
        new_end = datetime.fromtimestamp(int(raw_end), tz=timezone.utc).replace(tzinfo=None)

# UPDATE subscriptions SET current_period_start=new_start, current_period_end=new_end
# WHERE user_id=user_id (only when both extracted successfully — defense-in-depth)
if new_start is not None and new_end is not None:
    await db.execute(
        update(Subscription)
        .where(Subscription.user_id == user_id)
        .values(
            current_period_start=new_start,
            current_period_end=new_end,
        )
    )
```

## Chassis implications

Period fields now have a clear single-source-of-truth model:
- **Initial set:** subscribe endpoint at create-time.
- **Renewal updates:** `invoice.paid` (cycle) handler from invoice payload.
- **Other webhook handlers:** explicitly do not touch periods. Eliminates the entire "webhook overwrites with extracted-None" failure class.

This is the chassis-discipline pattern: don't let multiple handlers race for source-of-truth on a field. Pick the handler with the most reliable signal and let others sync only fields they can reliably extract.

No new chassis-contract entries (architectural change, not invariant addition).
