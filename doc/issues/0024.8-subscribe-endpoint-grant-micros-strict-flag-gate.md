## Context

0024.2 introduced `BILLING_SUBSCRIPTION_GRANTS_TO_LEDGER` and the chassis-flag-gate semantic: when the flag is OFF, the subscribe endpoint stores `grant_micros=0` in `subscriptions` (column unused; value is a placeholder). 0024.7 then applied Path B to webhook handlers so they preserve the subscribe endpoint's value rather than overwriting with metadata.

But 0024.2's implementation in `app/routes/billing.py` (else-branch of the flag check, around lines 339–344):

```python
else:
    # Flag OFF: grant_micros is not needed; store 0 as placeholder in the row.
    try:
        grant_micros = int(raw_grant) if raw_grant else 0
    except (ValueError, TypeError):
        grant_micros = 0
```

The comment says "store 0", but `int(raw_grant) if raw_grant else 0` only writes `0` when `raw_grant` is falsy (None, empty string, `"0"`). With `raw_grant="19990000"` (truthy), `int(raw_grant)` returns `19990000`. The flag-gate is bypassed whenever metadata is present.

### Effect on production data

After 0024.7 deployed, fresh subscribes (user 37 / user 38) with flag=false produced `subscriptions.grant_micros=19990000` and **no** `balance_ledger` entry. The ledger half is correct (always was — `grant()` is properly flag-gated). The subscribe endpoint half is the lie: it claims to store 0 but stores the metadata value.

12+ historical rows are affected, mostly with `grant_micros=19990000` paired with empty ledger.

### Why earlier rows showed `grant_micros=0`

Rows 17, 19–23 used Stripe price `price_1TQbHzGPSfG3zV2fiMkKWkkX` ("start1") at a time when its metadata.grant_micros wasn't set — `raw_grant=None`, so `else 0` fired. Once metadata was populated on the Starter price (`price_1TOOku...`), every subsequent flag=false subscribe leaked the metadata value.

### Why 0024.7's audit missed it

The Explorer audit pass before 0024.7 dispatch said "lines 328–344 correctly write `grant_micros=0` when flag=false". The audit saw the `else 0` fallback and concluded the branch was correct, missing that the truthy-guard means `0` only fires when metadata is **absent**. 0024.7 was scoped to webhook handlers exclusively based on that audit; the subscribe-endpoint exclusion was unfounded.

### Why this is a 0024.2 bug, not a 0024.7 bug

0024.7 fixed the **webhook handlers**' Path B violations. The subscribe endpoint is an independent code path with an independent flag-gate implementation (introduced by 0024.2). Both bugs hit the same column (`subscriptions.grant_micros`); the root causes are distinct. The subscribe-endpoint bug pre-dated 0024.7.

### Why the bug went untested

The existing OFF-state test `test_subscribe_does_not_require_grant_micros_when_grants_disabled` (test_billing_subscribe.py:1772) uses a Price with **no** `grant_micros` in metadata (`price.metadata = {"tier_name": "Starter"}`). So `raw_grant=None`, the buggy line falls to `0`, and the assertion `assert row.grant_micros == 0` passes — but the bug isn't exercised. The bug surfaces only with truthy raw_grant + flag=false; that combination had no coverage.

## Design decisions (pre-committed)

1. **Strict flag-gate.** When `BILLING_SUBSCRIPTION_GRANTS_TO_LEDGER=False`, `grant_micros` is `0` regardless of Stripe Price metadata. Replace lines 339–344 with:

   ```python
   else:
       # Flag OFF: grant_micros is unused (no ledger writes); store 0 unconditionally
       # regardless of whether Stripe Price metadata.grant_micros is set.
       grant_micros = 0
   ```

   The `try/except` is dropped — nothing to parse anymore.

2. **Keep the flag=true branch unchanged.** When flag=true, the subscribe endpoint correctly reads, parses, validates `metadata.grant_micros` (and 422s if missing). No change.

3. **No webhook-handler changes.** 0024.7's Path B fixes are intact and correct.

4. **No backfill.** Same posture as 0024.7. Adopters can manually `UPDATE subscriptions SET grant_micros = 0 WHERE ...` for the historical ~12 stale rows if they want clean state. Chassis doesn't auto-fix.

5. **No frontend changes.** No schema, no migration, no env vars, no chassis-contract entry.

6. **Tests:** add one regression test asserting `grant_micros=0` when flag=false AND `metadata.grant_micros="19990000"` (truthy raw_grant). The existing OFF-state test (no-metadata path) stays as a separate sanity case.

7. **Documentation:** `doc/systems/payments.md` already describes the intended behavior (line 100 column comment, line 153 metadata-required-when-true note, line 241 subscribe-flow narrative). One tightening word — change "0 when flag=false" to "0 when flag=false (regardless of Price metadata)" to make the strictness explicit and prevent future re-introduction.

## Out of scope

- Webhook handler changes (0024.7).
- Backfill of historical rows.
- Frontend.

## Chassis implications

After 0024.8, `subscriptions.grant_micros` semantics are coherent end-to-end across both code paths:

| When | Writer | Behavior |
|---|---|---|
| Subscribe (initial) | `POST /billing/subscribe` | Strict flag-gate: `0` when off; `int(metadata.grant_micros)` when on (422 if missing). |
| Renewal | `invoice.paid` cycle handler | Strict flag-gate: untouched when off; `int(invoice.lines.data[0].price.metadata.grant_micros)` when on. |
| Other webhooks | `subscription.created` UPDATE / `subscription.updated` | Path B — preserve existing value (0024.5 + 0024.7). |

Adopters running with the flag off can trust both the row and the ledger.

No new chassis-contract entries.
